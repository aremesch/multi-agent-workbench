# Fix: bash agent terminal opens with two prompts and ~31 blank rows between them

## Context

When a user spawns a `shell` agent (the `cli-adapters/shell.jsonc` adapter that runs `bash -i`) and opens the terminal modal, xterm.js renders the bash prompt **twice**: once at row 0 of the pane, and once near the bottom where the cursor sits, with an entire empty pane (~30 blank rows) between them. The screenshot the user attached is the symptom.

This is a regression introduced by the v0.2 pane-snapshot protocol ([`docs/plans/v0.2-terminal-pane-snapshot.md`](v0.2-terminal-pane-snapshot.md)). It happens for `bash` more visibly than for the long-running CLI adapters because bash emits exactly **one** short, idempotent line at startup (the PS1 prompt, no leading `\r\n`), so any duplication is plainly visible. Real CLI adapters (`claude`, `codex`, `gemini`) emit thousands of bytes with cursor-positioning escapes that overwrite each other, so the same race tends to be cosmetically masked.

## Why it happens

The race lives in [`HubClient.handleSubscribe`](../../src/lib/server/ws/hub.ts#L152-L267). Today the order is:

1. **Sync** read `snapshotSeq = getLatestTerminalSeq(agentId)` — line 179
2. Register live `output` listener — line 214
3. **Await** `Tmux.capturePane(session, 0)` — line 236 (runs `tmux capture-pane -p -e -S 0`)
4. Send `pane_snapshot { ansi, seq: snapshotSeq }` — line 251
5. Catch-up: `listTerminalChunksSince(agentId, snapshotSeq)` — lines 253-261

Combined with the client handler at [`src/lib/client/ws.ts:259-270`](../../src/lib/client/ws.ts#L259) which does `term.reset()` then `term.write(ansi)` and **unconditionally** sets `maxSeenSeq = msg.seq` from the snapshot, this produces a deterministic double-paint in the following timing:

- t0: Client subscribes. Server reads `snapshotSeq = 0` (bash hasn't flushed its prompt to `terminal_log` yet — it's still loading its rcfiles).
- t1: Listener registered.
- t2: `capture-pane` is awaited.
- t3: Bash finishes loading and writes its prompt. Bytes flow `bash → pty → tmux pane buffer → tmux pipe-pane → FIFO → FifoStreamer → AgentRuntime.onChunk` ([src/lib/server/agents/AgentRuntime.ts:157-171](../../src/lib/server/agents/AgentRuntime.ts#L157-L171)) which **synchronously** `insertTerminalChunk` (seq=1) **then** `emit('output', ...)`. The live listener fires → server sends `output` (seq=1, prompt) to client.
- t4: `capture-pane` returns. The pane buffer has the prompt at row 0, ~31 empty rows below. Server CRLF-normalizes (line 247) and ships `pane_snapshot { ansi: "<prompt>\r\n\r\n…\r\n", seq: 0 }`.
- t5: Catch-up query: `seq > 0` returns the prompt chunk → server sends `output` (seq=1, prompt) **again**.

Client receives:

1. `output` (seq=1) → writes prompt at xterm (0,0), `maxSeenSeq = 1`.
2. `pane_snapshot` (seq=0) → `term.reset()`, writes ~32 rows of pane text, **SET** `maxSeenSeq = 0` (the v5 design intentionally sets-not-maxes to allow already-applied bytes to be re-applied after the reset).
3. catch-up `output` (seq=1) → passes `seq > maxSeenSeq(0)`, writes the prompt **at the current cursor position**, which after writing 32 rows is at the bottom of the pane.

End state: prompt at row 0 (from snapshot ansi), prompt at the bottom (from catch-up). That's the screenshot.

A secondary issue is purely cosmetic but still wrong: even if the race didn't trigger, writing the captured pane (`prompt\r\n\r\n…\r\n`) leaves xterm's cursor at the bottom-left of the pane, not where tmux's actual cursor is (right after the prompt's trailing space). Live keystrokes would echo at the bottom on subsequent typing.

## Recommended fix

Three minimal changes that together eliminate the double-paint, put the cursor in the right place, and remove the now-vestigial catch-up loop. The captured tmux bytes are **not** filtered or rewritten — we only **append** a cursor-position escape after the existing CRLF normalization, which honors the user's "no filtering" preference.

### Change 1 — Read `snapshotSeq` AFTER `capture-pane` resolves

Move the watermark read to immediately after the await. This makes `snapshotSeq` cover every chunk that landed in `terminal_log` by the time the pane was captured (including the bash prompt, which now sits at row 0 of the captured ansi).

The live listener stays registered **before** the await so bytes that arrive during the await are still forwarded live; they get wiped by `term.reset()` and re-painted by the snapshot ansi (since they were also captured into the pane buffer — `tmux pipe-pane` writes from the same screen-update event that updates the buffer, so any byte in the FIFO is also in the pane).

### Change 2 — Drop the catch-up loop

Once Change 1 lands, the catch-up loop is vestigial. Its sole purpose under v5 was to repaint live bytes that beat the snapshot to the wire — those bytes are now already in the captured pane buffer (and therefore in the snapshot ansi), so catch-up has nothing useful to add.

The only theoretical case where catch-up could still help is bytes that hit the FIFO **between** `capture-pane` returning and `getLatestTerminalSeq` running. The cursor-position query is the only `await` between those two points; for the bash bug we're fixing, bash is idle by then. For active TUIs that emit during that window, catch-up wouldn't help anyway — those bytes would have `seq <= snapshotSeq` (`getLatestTerminalSeq` runs after `insertTerminalChunk` in [`AgentRuntime.onChunk`](../../src/lib/server/agents/AgentRuntime.ts#L157-L171) is sync), so they wouldn't be in `listTerminalChunksSince(snapshotSeq)`. A TUI's next render frame self-corrects; no recovery loop required.

After the loop is removed, the client's existing live-output dedup (`output` paints only when `msg.seq > maxSeenSeq`, [src/lib/client/ws.ts:249-258](../../src/lib/client/ws.ts#L249)) keeps doing the right thing for live bytes that arrive after the snapshot, because the snapshot's `maxSeenSeq = snapshotSeq` SET correctly drops anything already represented in the captured pane.

### Change 3 — Append a cursor-position escape to the snapshot ansi

After CRLF-normalizing, query tmux for the live cursor position and append a CSI Cursor Position escape (`\x1b[<row+1>;<col+1>H`, 1-based) to `ansi`. xterm's cursor then lands where tmux's cursor is (e.g. column 39 of row 0, right after `…broker-comparison$ `), instead of at the bottom of the empty pane after writing 32 rows of pane text.

### Code touch points

- [src/lib/server/tmux/TmuxSession.ts:207-222](../../src/lib/server/tmux/TmuxSession.ts#L207) — add a sibling helper `static async getCursorPosition(session: string): Promise<{ x: number; y: number } | null>` that runs:
  ```
  tmux display-message -t <session> -p '#{cursor_x},#{cursor_y}'
  ```
  Parse `"x,y\n"`. Return `null` on failure (caller falls back to no-op so a transient tmux hiccup doesn't drop the snapshot).

- [src/lib/server/ws/hub.ts:152-267](../../src/lib/server/ws/hub.ts#L152-L267):
  1. Delete the synchronous `snapshotSeq` read at line 179.
  2. Reorder the async tail (lines 233-266):
     - `await Tmux.capturePane(session, 0)`
     - CRLF-normalize (existing line 247)
     - `await Tmux.getCursorPosition(session)` — append `\x1b[${y+1};${x+1}H` to `ansi` if non-null
     - `const snapshotSeq = getLatestTerminalSeq(agentId)` — moved here from line 179
     - existing `subs.get(agentId) !== sub` early-out and the `pane_snapshot` send
  3. **Delete** the `try { const rows = listTerminalChunksSince(...); for (const r of rows) ... }` block (lines 252-265).
  4. Drop the now-unused `listTerminalChunksSince` import if no other call site remains (grep confirms — it's only used in this file).
  5. Update the header comment at lines 173-178 to describe the new ordering rationale.

- No changes required to:
  - The wire protocol — `pane_snapshot { ansi, seq }` shape unchanged. The `seq` field still carries the post-capture watermark, which the client uses to dedup future live bytes.
  - The client (`src/lib/client/ws.ts:259-270` and `Terminal.svelte`'s `term.reset(); term.write(ansi)` flow) — its existing SET behavior on `pane_snapshot.seq` is correct dedup for the live-only model.
  - `terminal_log` persistence (kept intact for `/api/agents/[id]/log` archive replay).

## Verification

1. Build: `pnpm build` → `node build/server.js` and confirm no TypeScript / lint regressions.
2. Unit: `pnpm test` — new + updated cases above pass; existing 472 cases still green.
3. Manual: spawn a `shell` agent from the dashboard, open the modal. Expected: one prompt at row 0, cursor blinking right after `$ ` (column ~39 of row 0), the rest of the pane empty. Reload the page (re-subscribe) → same result, no doubling.
4. Manual regression: spawn a `claude-code` (or other long-running adapter) agent that has been running for a while; reattach via the modal. Expected: the captured pane renders intact, cursor lands where it would in `tmux attach`, no scroll-staircase or doubled glyphs.
5. Manual regression: kill the maw service mid-flight, restart it (`systemctl --user restart maw`), then reopen the modal. Verify the reattach path still streams live keystrokes (this exercises the same subscribe handler).

## Out of scope

- Switching `capture-pane` to `-S -` (full scrollback) — would change reattach semantics and emit large blobs. Leave as a follow-up if users ask for scrollback on first paint.
- Any change to the v5 wire protocol — the schema is unchanged.
