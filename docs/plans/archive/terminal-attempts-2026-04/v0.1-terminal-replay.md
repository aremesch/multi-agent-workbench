# Terminal replay: snapshot-on-reconnect instead of byte-log

> Rename to `v0.1-terminal-replay.md` on approval (per CLAUDE.md convention).

## Context

After a Vite dev-server restart тАФ the exact case MAW is designed to
survive gracefully тАФ reopening an agent's terminal modal paints a
badly corrupted view:

- The CLI welcome banner renders 3тАУ4 times stacked, separated by stray
  horizontal rules from progress spinner frames.
- Text wraps at the wrong column width; old `тФА` box-drawing fragments
  leak into the new layout.
- Once live output resumes after the replay, everything renders fine.

The frontend before the restart was working off a long-lived xterm.js
instance whose internal grid state matched tmux exactly. After the
restart, `AgentTerminalPanel.svelte` mounts a fresh `MawWsClient` with
`lastSeq: 0` (`src/lib/client/ws.ts:82`), and the hub replays the
**entire** `terminal_log` тАФ every byte the PTY ever emitted тАФ into a
virgin xterm via `listTerminalChunksSince(agentId, 0)`
(`src/lib/server/ws/hub.ts:115`). For a plain line-oriented CLI that's
harmless; for a TUI like Claude Code that uses the alt-screen buffer,
every intermediate repaint pass it has ever drawn gets faithfully
re-played back-to-back. Two things compound it:

1. **Width drift.** Tmux is spawned at `200├Ч50`
   (`src/lib/server/tmux/TmuxSession.ts:41`). All the logged bytes
   encode borders and cursor addressing at that width. The modal's
   xterm fits to a different column count, and the resize the client
   sends (`AgentTerminalPanel.svelte:107`) lands *after* scrollback
   has already been written into the grid.
2. **The correct fallback is dormant.** `hub.ts:122-134` already has a
   `Tmux.capturePane` fallback that would do the right thing, but it
   only runs when `listTerminalChunksSince` returns zero rows тАФ which
   never happens once an agent has produced any output.

Goal: reconnects paint the **current visible pane state** exactly once,
at the client's actual dimensions, before any live output arrives.
The byte log stays (it's a useful artifact for the LLM overseer) but
is no longer the reconnect path.

## Approach

Atomic "resize тЖТ capture тЖТ snapshot" in a single `subscribe_agent`
round-trip. The client includes its freshly-fitted xterm dimensions in
the subscribe message; the server resizes the tmux pane, waits for
tmux to redraw, captures the pane with ANSI preserved, and returns a
single `scrollback` frame carrying that snapshot. The byte-log replay
branch in `handleSubscribe` is deleted. Spawn defaults drop from
`200├Ч50` to `120├Ч32` so detached agents viewed before anyone attaches
still look reasonable.

## Concrete changes

### 1. Protocol тАФ `src/lib/shared/protocol.ts`

Extend `CS_SubscribeAgent` with optional `cols` / `rows`:

```ts
export interface CS_SubscribeAgent {
  type: 'subscribe_agent';
  agentId: string;
  /** Client's current xterm dimensions. When present, the server resizes the
   *  tmux pane to these dims before capturing the reconnect snapshot, so the
   *  snapshot's line widths match the viewer. */
  cols?: number;
  rows?: number;
  // lastSeq removed тАФ reconnect no longer does byte-log gap replay.
}
```

Drop `lastSeq` from the type. Bump `PROTOCOL_VERSION` to `2` so stale
clients are told to reload.

`SC_Scrollback` stays as-is (still a list of `{seq, b64}` chunks), but
on reconnect it will always carry exactly one chunk containing the
capture-pane snapshot. No schema change needed there.

### 2. Server тАФ `src/lib/server/ws/hub.ts`

Replace the body of `handleSubscribe` (lines 101тАУ135):

- Delete the `listTerminalChunksSince` call and the existing `else if
  (baseSeq === 0)` branch.
- If `msg.cols && msg.rows` are present and finite, call
  `runtime.resize(msg.cols, msg.rows)` first. Wrap in try/catch тАФ a
  failed resize must not block the snapshot (agent may be dying).
- After the resize resolves, `await Tmux.capturePane(runtime.tmuxSession)`
  and send a single `scrollback` frame with the result as one chunk
  (`seq: 0` is fine тАФ the client no longer tracks seqs for replay).
- Keep the existing `runtime.on('output' | 'event' | 'state')`
  attach block unchanged.
- Remove the `listTerminalChunksSince` import.

`handleResize` (lines 205тАУ212) stays тАФ it still handles live drag
resizes after the modal is open.

**Important ordering nit**: tmux's `resize-window` returns when the
command completes, but the pane's repaint is asynchronous. A
`capture-pane` called one microtask later will usually see the new
width, but not always. Use a single `await new Promise(r =>
setTimeout(r, 25))` between resize and capture as a pragmatic fence тАФ
the exact same latency a normal reconnect would incur anyway.
(Cleaner alternative if we hit issues: send `C-l` via
`Tmux.sendKey(session, 'C-l')` to force a repaint, then capture.
Start with the timeout; escalate if flaky.)

### 3. Client тАФ `src/lib/client/ws.ts`

- Drop `lastSeq` tracking entirely: `Sub` becomes `{ agentId: string }`,
  the `lastSeq: 0` seed in `subscribe` goes away, the `msg.seq > sub.lastSeq`
  bookkeeping in `dispatch` for `output` and `scrollback` goes away.
- `subscribe(agentId, cols?, rows?)` now accepts optional dimensions
  and forwards them in the `subscribe_agent` message.
- `onopen` reconnect loop re-issues subscribes with the latest known
  dimensions (cache them on the `Sub` record).

### 4. Client тАФ `src/lib/client/components/AgentTerminalPanel.svelte`

- In `onMount`, do **not** call `client.subscribe(agent.id)` immediately.
  Move the subscribe into the first `onTerminalResize` tick: the
  Terminal component unconditionally fires `onResize(term.cols, term.rows)`
  once after fit (see `Terminal.svelte:96`), so we get real dimensions
  synchronously on first render. Subscribe with those dims.
- Before writing the scrollback chunks in `onScrollback`, call
  `term?.clear()` (or `term?.reset()` тАФ see note below) so no residual
  state from a previous frame can bleed through. `Terminal.svelte`
  already exports `clear()` at line 45.
- `scheduleResize` stays as-is for subsequent live resizes.

**clear() vs reset()**: `term.clear()` only wipes the visible region
and keeps scrollback; `term.reset()` rebuilds the parser state from
scratch. Use `reset()` here тАФ we want the fresh snapshot to be
unambiguously the first frame, with no parser carry-over from any
previous dynamic import warm-up write.

### 5. Server тАФ `src/lib/server/tmux/TmuxSession.ts`

Lower spawn defaults:

```ts
const cols = opts.cols ?? 120;
const rows = opts.rows ?? 32;
```

No other change in this file. `capturePane`, `resizeWindow`, and the
`-e` ANSI-preserving flag are already correct.

### 6. Byte log stays

`AgentRuntime.onChunk` keeps calling `insertTerminalChunk` тАФ the log
is still written for the future LLM overseer path referenced in
`docs/plans/v0.1-foundation.md:483`. We're only removing the *read*
path in `hub.ts`. `listTerminalChunksSince` in
`src/lib/server/db/queries.ts:428` becomes an unused export; leave it,
since the overseer will need a variant of it (likely
`listRecentChunks(agentId, nKb)`).

## Non-goals

- No changes to `/agents/[id]` dedicated route wiring тАФ it uses the
  same `AgentTerminalPanel` and picks the fix up for free.
- No edits to `terminal_log` schema.
- No LLM-overseer plumbing (separate v0.2 track).
- No PWA/push-notification work.
- No `snapshot endpoint` reuse тАФ the dashboard thumbnail endpoint at
  `src/routes/api/agents/[id]/snapshot/+server.ts` already uses its own
  `capturePane(session, -50)` path and is unaffected.

## Verification

1. **Typecheck & build**
   - `pnpm check` тАФ 0/0 must stay clean; the protocol change propagates
     through the discriminated unions, so any dead `lastSeq` reference
     in hub.ts or ws.ts surfaces here.
   - `pnpm build` тАФ SvelteKit prod build must succeed (Terminal.svelte's
     dynamic xterm import is the known SSR hazard).

2. **Smoke adapter end-to-end**
   - Start dev server.
   - Spawn an agent with `cli-adapters/shell.jsonc`.
   - Open its modal, watch the shell banner render.
   - Type a few lines so the scrollback has real content.
   - Close the modal, stop the dev server, start it again.
   - Reopen the modal: the banner should appear exactly **once**, at the
     current modal width, with no stray `тФА` fragments or repeated
     blocks. Live typing afterwards should stay in sync.

3. **Claude Code / real TUI agent**
   - Repeat (2) with a claude-code adapter if configured. This is the
     canonical reproduction of the bug, per the screenshot in the user's
     report.

4. **Width sanity**
   - Resize the browser window while the modal is open. `scheduleResize`
     should debounce and the tmux pane should keep up тАФ this path is
     untouched by the plan but worth confirming as a regression check.

5. **Reconnect after transient WS drop**
   - In DevTools, take the network offline briefly so the ws client
     hits its backoff path. On reconnect, the snapshot-on-subscribe
     should paint the current pane state without duplication.
