# Terminal persistence across modal open/close

> **SUPERSEDED** by [`v0.1-terminal-scrollback-v2.md`](v0.1-terminal-scrollback-v2.md).
> The `TerminalRegistry` approach landed but the persisted xterm
> scrollback turned out to be unusable for TUI CLIs: Claude Code
> (which doesn't use the alt-screen buffer) emits a full banner+prompt
> redraw on every streaming tick, and xterm faithfully appended each
> copy into its 10k-line scrollback. Opening and closing the modal a
> few times stacked N superseded renderings on top of each other.
> A reattach-resize race (`fit()` measuring a `display: none` ancestor
> before `dialog.showModal()` had laid out, clamping to MINIMUM_COLS/ROWS
> and shipping a bogus 2├Ч1 or 7-col resize to tmux) made it worse.
>
> The follow-up plan reverts the registry and moves the backing
> scrollback to tmux: on every modal open, capture 500 lines and pipe
> through the existing `collapseRepeatingTailBlocks` dedup helper.
> Only the shared `MawWsClient` module-level singleton from this plan
> survived тАФ it's an independent win worth keeping.

## Context

The reconnect-snapshot work in `v0.1-terminal-replay.md` fixed the post-vite-restart corruption but left a product gap: closing the agent modal and reopening it loses the xterm's scrollback entirely. The new xterm instance is virgin, the server-side `capture-pane -S 0` only returns the currently visible pane, and everything that scrolled past while the modal was open is unrecoverable.

That's unacceptable for the "daily browser workbench" use case. A user who opens an agent modal, lets the CLI run for five minutes, closes the modal to go deal with another agent, and comes back expects to still be able to scroll up and see what happened. Today they can't.

We verified via `tmux capture-pane -p -S 0` against a live agent that tmux's visible pane itself is clean тАФ the pollution we saw earlier was from scrollback history, which is unrecoverable for CLIs that rerender on each streaming tick (the main buffer collects every intermediate rewrite). So "grab more from tmux" is a dead end. The real fix is to stop destroying the xterm instance on modal close: keep it alive, let its own 10k-line scrollback grow naturally from live output, and reparent the DOM node between a hidden pool and the active modal as the user opens and closes.

**Scope boundary:** this plan covers persistence across modal open/close within one page lifetime. Persistence across vite restart / page reload is explicitly out of scope тАФ it would require either `@xterm/addon-serialize` + localStorage, or a proper server-side headless-xterm emulator. Noted as follow-up below.

## Approach

Introduce a client-side `TerminalRegistry` singleton that owns xterm instances keyed by `agentId`. A hidden "pool" div in the root layout parents every xterm when no modal is showing it. On modal open the panel acquires its entry and reparents the host DOM node into the panel's mount slot; on close the panel reparents it back to the pool. The xterm, its scrollback, and its WebSocket subscription are untouched across the trip.

Consolidate `MawWsClient` into a module-level singleton at the same time тАФ necessary because a single shared client must fan `output`/`event`/`state` messages out to the right xterm instance, and worth doing on its own merits (one `/ws` connection instead of one per panel).

Key design decisions validated by the Plan agent:

- **xterm.js `term.open()` is one-shot** and not reparent-friendly тАФ the second call early-returns without rebinding. The fix is to own a wrapper `host` div per entry, call `open(host)` once at create time, and move `host` around the DOM. xterm's canvas sits as a child of `host` and moves with it.
- **The pool must have layout**, not `display: none`. Off-screen absolute positioning with fixed dims, `visibility: hidden`, `pointer-events: none`. `display: none` breaks xterm's initial fit.
- **One-parent-at-a-time semantics** тАФ no refcount. Within one window, at most one panel shows a given agent's xterm at a time. `attachTo` steals the host from wherever it was; `detachToPool` sends it home.
- **Defer the initial `subscribe_agent`** until the first `attachTo`, not `acquire`. The pool's dims are not the real panel's dims; we want the server to capture the snapshot at the viewer's actual width.
- **Dispose on agent death**, not modal close. The registry listens for `agent_state тИИ {exited, crashed, killed}` messages and tears down the entry. The existing `$effect` in `src/routes/+page.svelte:32-38` (auto-close on exit) still works тАФ panel unmount calls `detachToPool`, and the dispose runs in parallel because the registry heard the `agent_state` message directly.
- **LRU eviction cap of 8** as a memory backstop. Worst-case 10k-scrollback ├Ч 120 cols тЙИ 38 MB per terminal; 8 is a reasonable ceiling. Currently-attached entries are pinned. Treat as a soft limit and revisit after measuring.

## Files

### New: `src/lib/client/terminalRegistry.ts` (~280 LOC)

Module-level singleton. Public API:

```ts
interface TerminalHandle {
  readonly agentId: string;
  readonly host: HTMLDivElement;
  write(data: Uint8Array): void;
  sendKeys(text: string): void;
  sendInput(text: string, submit: boolean): void;
  answerPrompt(choice: string | number): void;
  onEvent(cb: (ev: SC_AgentEvent) => void): () => void;  // returns unsubscribe
  onState(cb: (status: string) => void): () => void;
  currentStatus(): string | undefined;
}

class TerminalRegistry {
  setPool(el: HTMLDivElement): void;
  acquire(agentId: string, initialStatus?: string): TerminalHandle;  // creates entry mounted in pool; does NOT subscribe yet
  attachTo(agentId: string, parent: HTMLElement): void;  // first call issues initial subscribe at real dims
  detachToPool(agentId: string): void;  // moves host back to pool; keeps entry alive
  dispose(agentId: string): void;  // full teardown; called on agent death or LRU eviction
  pin(agentId: string): void;  // opt out of LRU (for /agents/[id] route)
}

export const terminalRegistry: TerminalRegistry;  // module singleton
```

Entry internal state:

```ts
interface Entry {
  agentId: string;
  host: HTMLDivElement;
  term: XTerm;
  fit: FitAddon;
  resizeObserver: ResizeObserver;
  lastSentCols: number;  // debounce state moves here from AgentTerminalPanel
  lastSentRows: number;
  resizeTimer: ReturnType<typeof setTimeout> | null;
  subscribed: boolean;  // flipped on first attachTo
  status: string | undefined;
  eventListeners: Set<(ev: SC_AgentEvent) => void>;
  stateListeners: Set<(status: string) => void>;
  lastDetachedAt: number;  // for LRU
  pinned: boolean;
  disposed: boolean;
}
```

Key lifecycle:

- `acquire`: if entry exists return its handle; else dynamic-import xterm/addon-fit, create `term`, create `host` div (class `terminal-host`), `term.open(host)`, `pool.appendChild(host)`, create ResizeObserver observing `host` that runs `fit.fit()` and forwards `(cols, rows)` via the debounced resize, wire `term.onData` тЖТ `wsClient.sendKeys(agentId, тАж)`, wire `term.onResize` тЖТ resize forwarder. Does NOT call `wsClient.subscribe` yet.
- `attachTo`: remove `host` from its current parent (pool or other), append to new parent. If `!subscribed`, set `subscribed = true` and call `wsClient.subscribe(agentId, cols, rows)` with the first fit's dims. If already subscribed, just reparent тАФ fit's ResizeObserver will issue a debounced resize if dims changed.
- `detachToPool`: append `host` back to pool, record `lastDetachedAt = Date.now()`, run LRU eviction pass.
- `dispose`: `term.dispose()`, `resizeObserver.disconnect()`, `host.remove()`, `wsClient.unsubscribe(agentId)`, notify listeners, remove from `entries`.
- LRU: on any `detachToPool`, count non-pinned non-attached entries; if > 8, dispose the one with smallest `lastDetachedAt`.

Registers `wsClient` listener for `agent_state` globally; on `exited`/`crashed`/`killed` for a known agent, calls `dispose(agentId)`.

### Changed: `src/lib/client/ws.ts` (~183 тЖТ ~220 LOC)

- Export module-level singleton `getMawWsClient(): MawWsClient` (lazy-created on first call, auto-connects). Layout's `onMount` calls it to kick off the connection early.

Backwards-compatibility shim is not needed тАФ the only consumer is `AgentTerminalPanel` and it's being rewritten in the same PR.

## Subtleties worth spelling out

1. **xterm canvas inside `host`**. `term.open(host)` creates `.xterm` as a child of `host`. Reparenting `host` moves the whole thing. Do NOT try to reparent `.xterm` directly тАФ its internal wiring to xterm's core via the `host`'s ownerDocument matters.

2. **ResizeObserver lifetime**. The registry entry owns ONE ResizeObserver on `host`, created in `acquire`, disposed in `dispose`. Do not create/destroy observers per attach. RO fires reliably on both attach (new container size) and content-box size changes.

3. **Detach-before-remove**. Svelte's `{#if openAgent}` block unmounts the panel's `mountSlot` div synchronously on close. `onDestroy` runs *before* that removal for `{#if}` blocks, so the panel's `detachToPool` call in `onDestroy` runs in time: `host` is moved out of `mountSlot` before Svelte removes `mountSlot` from the DOM. If the order ever flipped, `host` would go into the void (still alive in JS, but detached) тАФ the registry's `detachToPool` handles that case by appending to pool unconditionally (it doesn't assume current parent).

4. **Subscribe ordering**. First `attachTo` issues the initial subscribe. If the ResizeObserver fires before the first `attachTo` (because the pool has its own fixed layout), we'll have dims ready by the time the subscribe goes out. If RO fires *after* `attachTo` (because the panel's mount slot is slow to lay out), we subscribe at pool dims and then issue a subsequent resize. That's one extra tmux-resize on the server тАФ no correctness issue, just a tiny inefficiency. Keep the behavior simple; don't try to optimize this.

5. **Agent death race**. If the hub sends `agent_state=exited` while the modal is open, both paths fire: the panel's `$effect` calls `closeModal()`, which unmounts the panel тЖТ `onDestroy` тЖТ `detachToPool`; and the registry's global state listener calls `dispose(agentId)`. `dispose` must be safe to call on an entry whose `host` is currently attached to a panel's DOM (it walks `host.parentElement?.removeChild(host)` first). Order-invariant.

6. **Shared MawWsClient timing**. Layout mounts before page, so by the time the dashboard's first modal open fires `acquire`, the ws client is already connecting or connected. If it's still `CONNECTING`, the first `subscribe_agent` message queues (the existing `send()` checks `readyState === OPEN`); we need to change that to a queue-until-open pattern тАФ about 15 LOC. Alternatively, `onopen` re-issues all subscriptions automatically (it already does this for the reconnect case), so as long as `subscribe` eagerly updates the internal map before sending, the reconnect path reissues it. Prefer the latter тАФ zero new code, uses existing machinery.

## Verification

1. **Typecheck & build**
   - `pnpm check` тАФ must stay 0/0. The registry touches enough surface area that type errors are likely on first pass; tidy them.
   - `pnpm build` тАФ SvelteKit prod build, checking that dynamic xterm import still works without SSR breakage (the registry imports xterm inside a browser-only code path, same pattern as today's Terminal.svelte).

2. **Modal open/close scroll-back**
   - Spawn an agent, open its modal, trigger enough output to fill xterm scrollback (e.g. `yes | head -500` in the shell adapter, or ask Claude for 200 lines).
   - Close the modal. Reopen. Scroll up in xterm. Previous output must be visible.
   - Type in the terminal after reopen тАФ live input should work; the pane should not flash or reset.

3. **Multiple agents alive in the pool**
   - Open agent A, let it output 100 lines, close.
   - Open agent B, let it output 100 lines, close.
   - Reopen A тАФ its 100 lines are still in scrollback, untouched by B's activity.
   - Reopen B тАФ same.

4. **Dedicated route**
   - Navigate to `/agents/<id>` directly. Terminal should attach from the registry (or create a fresh entry if this is the first time anyone has opened the agent). Scroll-back and input should work identically to the modal.

5. **Agent death cleanup**
   - Open an agent, close its modal, kill the agent externally (`tmux kill-session` or the supervisor reaper path).
   - Wait for the `agent_state` message to propagate. The registry entry should be disposed (verify via a temporary `console.log` in `dispose` or a DevTools heap snapshot).
   - Reopening the agent (if the dashboard still has a card for it) should create a fresh registry entry, not resurrect a stale one.

6. **LRU eviction smoke**
   - Temporarily drop the cap to 2 for testing, open 4 different agents in sequence closing each modal. After the 4th open, the first two should be disposed; verify by ws traffic (no more output messages for those agent IDs) and by the absence of their entries. Restore cap to 8 before committing.

7. **Resize behavior**
   - Open modal, drag the browser window to change size. xterm should refit smoothly; tmux should resize; live output should wrap at the new width.
   - Close modal, resize browser, reopen. xterm should refit to the new modal dims.

8. **Reconnect**
   - Take the network offline via DevTools while a modal is open. Wait for ws to hit backoff. Bring it back. The existing reconnect path should re-issue `subscribe_agent` for every registered agent, the server snapshot should land, xterm reset + apply. Live output resumes.

## Non-goals

- **Vite restart / page reload persistence.** Follow-up plan: layer `@xterm/addon-serialize` over the registry. On `detachToPool` dump xterm state to sessionStorage; on `acquire` for an existing agent after a page reload, restore state before the first subscribe. Needs careful handling of the initial snapshot (don't `reset()` if we have restored state).
- **Server-side headless xterm emulator** for "clean logical history" тАФ a v0.2 item that would also benefit the LLM overseer.
- **Dashboard thumbnail live-xterm mode** тАФ the thumbnails remain text snapshots polled from the snapshot endpoint. If we ever want live thumbs, the registry grows a second "mini" mount mode; not in scope here.
- **Multi-window / multi-tab sync** of terminal state. Each tab has its own registry.
