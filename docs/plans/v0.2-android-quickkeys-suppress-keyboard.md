# Suppress Android Soft Keyboard When Tapping Mobile Quick-Keys

## Context

The agent terminal modal renders a row of mobile quick-key buttons (cursor up/down, Shift+Tab, Esc, Ctrl+C, â€¦) under the xterm pane on touch devices, configured by `mobileQuickKeys` in each adapter's JSONC. Each button forwards raw bytes through the existing `send_keys` WebSocket path.

On Android, every tap on one of these buttons makes the on-screen virtual keyboard pop up and overlap the PWA. The user has to dismiss it after every tap. We want the keyboard to stay closed when tapping quick-keys; it should only open when the user explicitly taps the xterm terminal area.

### Root cause

`pressQuickKey` in [`src/lib/client/components/AgentTerminalPanel.svelte:179`](src/lib/client/components/AgentTerminalPanel.svelte:179) calls `term?.focus()` after sending the bytes:

```ts
function pressQuickKey(keys: string): void {
  getMawWsClient().sendKeys(agent.id, keys);
  term?.focus();
}
```

That call refocuses xterm.js's hidden `<textarea>`. On Android, programmatically focusing a text input **during a user gesture** (the click handler chain) is what triggers the on-screen keyboard. The original plan intent ([`v0.2-terminal-mobile-quickkeys.md:18-25`](docs/plans/v0.2-terminal-mobile-quickkeys.md)) explicitly says "refocuses xterm afterwards" â€” a desktop-friendly behaviour that doesn't translate to mobile.

A secondary contributor: the `<button>` itself steals focus on tap (default browser behaviour). Even without the explicit `term.focus()`, on Android any subsequent text-input focus inside the same gesture would still surface the keyboard.

## Approach

Two complementary changes on the quick-key buttons, both confined to [`AgentTerminalPanel.svelte`](src/lib/client/components/AgentTerminalPanel.svelte):

1. **Prevent the button from stealing focus** â€” add `onmousedown={(e) => e.preventDefault()}` to each quick-key `<button>`. This is the standard rich-text-editor pattern: `preventDefault` on `mousedown` cancels the default focus shift while leaving the synthesized `click` event intact. Mobile browsers synthesize `mousedown` from touch, so the same handler covers Android.

2. **Stop calling `term?.focus()` from `pressQuickKey`** â€” since focus never leaves xterm in the first place (change #1), the refocus call is dead weight on desktop and the actual keyboard trigger on Android. Remove it.

After the fix:

- Tap a quick-key â†’ focus stays on whatever it was â†’ bytes go through `send_keys` â†’ no soft keyboard.
- Tap the xterm `<div>` â†’ xterm's own pointer handler focuses the textarea via genuine user gesture on the terminal â†’ soft keyboard opens (correct, unchanged behaviour).
- Desktop physical keyboard usage is unaffected: focus never left xterm, so the next keystroke still routes to the terminal.

### Why not gate `term?.focus()` on `isTouch`?

`isTouch` (driven by `matchMedia('(pointer: coarse)')`, [`AgentTerminalPanel.svelte:110-134`](src/lib/client/components/AgentTerminalPanel.svelte:110)) is already in scope and could conditionally skip the refocus. Skipped â€” `mousedown.preventDefault()` makes the refocus unnecessary on every platform, so a touch gate is extra surface area for no gain. One mechanism, one code path.

### Why not `tabindex="-1"`?

That would make the buttons unreachable via keyboard navigation. `mousedown.preventDefault()` keeps Tab-focus working while suppressing pointer-driven focus shifts.

## Follow-up â€” Pixel 9 / Android 16 still surfaced the keyboard

Real-device test (Pixel 9 Pro XL, Android 16) showed the soft keyboard still
appears on every quick-key tap even after `mousedown.preventDefault()` +
removed `term.focus()`. Root cause: xterm's hidden `<textarea>` is auto-focused
on modal open (`Terminal.svelte`'s onMount calls `term.focus()`), and Android
Chrome treats *any* user gesture-while-an-editable-is-focused as "show the
soft keyboard" â€” focus doesn't have to move during the gesture.

Two complementary layers on top of the original fix:

1. **Skip xterm autofocus on touch devices** in
   [`Terminal.svelte`](src/lib/client/components/Terminal.svelte) â€” gate the
   onMount `term.focus()` on `!matchMedia('(pointer: coarse)').matches`. Phone
   PWA opens with the textarea unfocused; quick-key taps see no focused
   editable (no keyboard); xterm's own pointer handler still focuses the
   textarea on a real touch on the terminal â€” that's the legitimate trigger
   for opening the keyboard. Desktop unchanged.

2. **`interactive-widget=resizes-content` on viewport meta** in
   [`src/app.html`](src/app.html) â€” when the keyboard *does* legitimately
   appear (user tapped the terminal to type a prompt response), Chromium
   shrinks the layout viewport instead of overlaying. The agent modal (sized
   in `dvh`) just gets shorter and the terminal output stays visible above
   the keyboard. No JS, no `visualViewport` listener, no transform hack.
   Browser support: Android Chrome 108+. Whole-app meta tag is fine â€” pages
   without inputs aren't affected.

Why both: (1) is the *don't surface the keyboard if I didn't ask for it*
layer and fixes the user's immediate complaint. (2) is the *if it does
surface â€” because I tapped the terminal to type â€” make it not cover the
output* layer; without it the user is worse off the moment they genuinely
want to type.

No new unit tests for the autofocus gate â€” it's a one-line `matchMedia` guard
inside an async dynamic-import block that fails on `HTMLCanvasElement.getContext`
in jsdom anyway, so the focus call never executes in tests. The viewport meta
is a static HTML attribute with no test surface.
