# Suppress Android Soft Keyboard When Tapping Mobile Quick-Keys

## Context

The agent terminal modal renders a row of mobile quick-key buttons (cursor up/down, Shift+Tab, Esc, Ctrl+C, …) under the xterm pane on touch devices, configured by `mobileQuickKeys` in each adapter's JSONC. Each button forwards raw bytes through the existing `send_keys` WebSocket path.

On Android, every tap on one of these buttons makes the on-screen virtual keyboard pop up and overlap the PWA. The user has to dismiss it after every tap. We want the keyboard to stay closed when tapping quick-keys; it should only open when the user explicitly taps the xterm terminal area.

### Root cause

`pressQuickKey` in [`src/lib/client/components/AgentTerminalPanel.svelte:179`](src/lib/client/components/AgentTerminalPanel.svelte:179) calls `term?.focus()` after sending the bytes:

```ts
function pressQuickKey(keys: string): void {
  getMawWsClient().sendKeys(agent.id, keys);
  term?.focus();
}
```

That call refocuses xterm.js's hidden `<textarea>`. On Android, programmatically focusing a text input **during a user gesture** (the click handler chain) is what triggers the on-screen keyboard. The original plan intent ([`v0.2-terminal-mobile-quickkeys.md:18-25`](docs/plans/v0.2-terminal-mobile-quickkeys.md)) explicitly says "refocuses xterm afterwards" — a desktop-friendly behaviour that doesn't translate to mobile.

A secondary contributor: the `<button>` itself steals focus on tap (default browser behaviour). Even without the explicit `term.focus()`, on Android any subsequent text-input focus inside the same gesture would still surface the keyboard.

## Approach

Two complementary changes on the quick-key buttons, both confined to [`AgentTerminalPanel.svelte`](src/lib/client/components/AgentTerminalPanel.svelte):

1. **Prevent the button from stealing focus** — add `onmousedown={(e) => e.preventDefault()}` to each quick-key `<button>`. This is the standard rich-text-editor pattern: `preventDefault` on `mousedown` cancels the default focus shift while leaving the synthesized `click` event intact. Mobile browsers synthesize `mousedown` from touch, so the same handler covers Android.

2. **Stop calling `term?.focus()` from `pressQuickKey`** — since focus never leaves xterm in the first place (change #1), the refocus call is dead weight on desktop and the actual keyboard trigger on Android. Remove it.

After the fix:

- Tap a quick-key → focus stays on whatever it was → bytes go through `send_keys` → no soft keyboard.
- Tap the xterm `<div>` → xterm's own pointer handler focuses the textarea via genuine user gesture on the terminal → soft keyboard opens (correct, unchanged behaviour).
- Desktop physical keyboard usage is unaffected: focus never left xterm, so the next keystroke still routes to the terminal.

### Why not gate `term?.focus()` on `isTouch`?

`isTouch` (driven by `matchMedia('(pointer: coarse)')`, [`AgentTerminalPanel.svelte:110-134`](src/lib/client/components/AgentTerminalPanel.svelte:110)) is already in scope and could conditionally skip the refocus. Skipped — `mousedown.preventDefault()` makes the refocus unnecessary on every platform, so a touch gate is extra surface area for no gain. One mechanism, one code path.

### Why not `tabindex="-1"`?

That would make the buttons unreachable via keyboard navigation. `mousedown.preventDefault()` keeps Tab-focus working while suppressing pointer-driven focus shifts.

## Tests

- **Unit test** in [`src/lib/client/components/AgentTerminalPanel.test.ts`](src/lib/client/components/AgentTerminalPanel.test.ts) (extend if it exists; otherwise lean on whichever existing component test already mounts the panel) — render the panel with `mobileQuickKeys` populated and `quickKeysMode: 'always'`, dispatch a `mousedown` MouseEvent on a quick-key button, and assert `event.defaultPrevented === true`. Then dispatch `click` and assert `MawWsClient.sendKeys` was called once with the configured key bytes. If no AgentTerminalPanel-specific test file exists today, add a thin one — the focus-theft prevention is exactly the kind of behaviour that silently regresses on a refactor.
- No e2e test. Playwright on chromium can't reliably reproduce Android's virtual-keyboard-on-focus behaviour, and headless touch emulation does not surface a real on-screen keyboard. Manual Android verification (below) is the truth.
