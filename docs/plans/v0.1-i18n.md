# i18n: en/de/fr/es locale support

## Context

The app has ~100 unique user-facing strings hardcoded in English across 37 files. We need to support 4 locales (en, de, fr, es) with a user-selectable preference persisted in the existing `user_settings` table and Accept-Language detection for first-time visitors.

## Approach: hand-rolled (no library)

~100 strings, no pluralization, no route-prefix locales. A library is overkill. We follow the exact pattern already used for the theme preference: server-side resolution in hooks, `<html>` attribute injection, layout data pass-through, client store + API endpoint.

## New files

```
src/lib/i18n/
  index.ts       -- Locale type, SUPPORTED_LOCALES, LOCALE_NAMES, detectLocale(), t()
  en.ts          -- English dictionary (source of truth, Record<string, string>)
  de.ts          -- German (Partial<typeof en>, missing keys fall back to en)
  fr.ts          -- French
  es.ts          -- Spanish
src/lib/client/stores/locale.ts   -- writable store, initLocale(), setLocale() (clone of theme.ts)
src/lib/client/i18n.svelte.ts     -- useT() helper using Svelte context
src/routes/api/user/locale/+server.ts  -- PUT endpoint (clone of /api/user/theme)
```

## Modified files

| File | Change |

## Implementation steps

### 1. i18n core (`src/lib/i18n/`)

Create `index.ts`:
- `type Locale = 'en' | 'de' | 'fr' | 'es'`
- `SUPPORTED_LOCALES`, `DEFAULT_LOCALE = 'en'`, `LOCALE_SETTING_KEY = 'ui.locale'`
- `LOCALE_NAMES: Record<Locale, string>` with native names (English, Deutsch, Francais, Espanol)
- `detectLocale(acceptLanguage: string | null): Locale` — parse Accept-Language, match best
- `t(locale: Locale, key: string, params?: Record<string, string | number>): string` — lookup + `{placeholder}` interpolation, English fallback

Create `en.ts` with all ~100 keys organized by namespace (nav.*, login.*, account.*, settings.*, spawn.*, agent.*, roles.*, projects.*, archive.*, common.*). Create `de.ts`, `fr.ts`, `es.ts` typed as `Partial<typeof en>` — populate with actual translations.

### 2. Server plumbing

**`src/app.d.ts`**: Add `locale: import('$lib/i18n').Locale` to `App.Locals`.

**`src/app.html`**: Change `lang="en"` to `lang="%maw.locale%"`.

**`src/hooks.server.ts`**: After the existing theme resolution block, add locale resolution:
```ts
const locale = user
  ? parseLocale(getUserSetting(user.id, LOCALE_SETTING_KEY))
  : detectLocale(event.request.headers.get('accept-language'));
event.locals.locale = locale;
```
In `transformPageChunk`, chain a second `.replace('%maw.locale%', locale)`.

**`src/routes/+layout.server.ts`**: Add `locale: locals.locale ?? DEFAULT_LOCALE` to both return paths.

### 3. Client plumbing

**`src/lib/client/stores/locale.ts`** (mirror `theme.ts`):
- `currentLocale` writable store
- `initLocale(locale)` — set store + `document.documentElement.lang`
- `setLocale(locale)` — set store + DOM + `PUT /api/user/locale`

**`src/lib/client/i18n.svelte.ts`**:
- `useT()` — reads locale from Svelte context, returns `(key, params?) => t(locale, key, params)`

**`src/routes/+layout.svelte`**: Import `initLocale`, call alongside `initTheme`. Provide locale via `setContext('locale', ...)` so `useT()` works in descendants.

### 4. API endpoint

**`src/routes/api/user/locale/+server.ts`**: PUT handler, validates locale is in `SUPPORTED_LOCALES`, calls `setUserSetting()`. Clone of the existing theme endpoint.

### 5. Settings UI

**`src/routes/settings/+page.svelte`**: Add a "Language" section below Appearance. Radio group with 4 options showing native name + flag/English name. Calls `setLocale()` on change.

### 6. String migration (file by file)

Replace hardcoded strings with `t()` calls. Server-side actions use `t(locals.locale, key)`. Components use `useT()`. Order by visibility:

1. `+layout.svelte` — nav strings
2. `login/` — login page + server errors
3. `account/` — password change page + server errors
4. `settings/` — settings labels
5. `SpawnAgentForm.svelte` — largest component (~50 strings)
6. `RepoTreeSidebar.svelte` — sidebar labels
7. `+page.svelte` (dashboard) — FAB, modal titles
8. `AgentCard.svelte`, `AgentTerminalPanel.svelte` — agent UI
9. `roles/`, `projects/`, archive pages
10. API route error messages
11. `dashboard.ts` theme descriptions — map to translation keys in settings page

### 7. Populate translations

Fill `de.ts`, `fr.ts`, `es.ts` with actual translations for all ~100 keys.

## Verification

1. `pnpm check` — no type errors
2. Switch locale in settings — UI updates immediately, `<html lang>` changes
3. Reload — locale persists (read from `user_settings`)
4. Log out, set browser to `de` — login page renders German
5. Submit forms with validation errors — error messages appear in active locale
6. Missing translation key — falls back to English gracefully
