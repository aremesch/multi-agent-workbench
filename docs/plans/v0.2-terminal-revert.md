# Terminal scrollback / screen distortion — clean-slate revert

> After approval this file will be renamed to the repo convention
> (e.g. `v0.2-terminal-revert.md`) per [CLAUDE.md](../../CLAUDE.md).

## Context

Across v0.1 and v0.2 we made six successive attempts to fix the
"terminal distortion on reconnect" problem for TUI CLIs (Claude Code
especially). Each fix looked promising in isolation; the accumulated
stack is now the problem.

Current observed symptom (reported 2026-04-22):

- On first spawn the terminal is mostly fine.
- **After close + reopen of the agent modal, it falls apart:**
  - Scrollback is mostly lost.
  - Using Claude Code's Ctrl-O / Ctrl-G (expand/collapse) causes further
    corruption: ghost redraws, smearing, wrong columns.
  - xterm cursor detaches from actual content (floats below prompt).
  - History shown is wrong or duplicated.
- The modal is close to unusable once it has been reopened once.

We have accumulated six partially-overlapping mechanisms that all fight
each other:

| # | Commit | Mechanism | Plan file |
|---|--------|-----------|-----------|
| 1 | `9000c73` | Snapshot-on-subscribe (replaces byte-log replay) + protocol v2 cols/rows | `v0.1-terminal-replay.md` |
| 2 | `9000c73` | Dropped `TerminalRegistry` DOM-reparenting; kept `MawWsClient` singleton | `v0.1-terminal-persistence.md` (superseded) |
| 3 | `9000c73` | `capture-pane -S -500` + `collapseRepeatingTailBlocks` dedup | `v0.1-terminal-scrollback-v2.md` |
| 4 | `95e85c6` | Per-adapter `scrollbackMode: 'visible' \| 'history'` | – |
| 5 | `7e7881f` | Out-of-band Claude JSONL history prepend + protocol v3 | `v0.1-jsonl-history.md` |
| 6 | `41af347` | Trailing-blank rstrip + tmux `cursor_x/y` CSI CUP append + optional `forceRedrawOnReconnect` SIGWINCH nudge | `v0.2-terminal-output-alignment.md` |

The goal of this branch is **not to design the new fix**. It is to
reset the reconnect path to a clean, minimal baseline so the next plan
can design a viable approach against a code surface that isn't already
three-layers-of-patch-on-patch.

**User decisions captured up front:**

1. Symptom is present on modal reopen — scrollback loss, cursor drift,
   ghost redraw, history corruption.
2. Rewind scope: all the way back to byte-log replay (pre-Phase-1). Keep
   only the `MawWsClient` singleton (it solves an orthogonal real
   problem — one `/ws` per tab).
3. Branch from current `development` tip.

## Scope

### In scope

- Wipe every code surface that exists because of Phases 1–6.
- Restore the pre-Phase-1 reconnect mechanism: **byte-log replay** from
  the already-persisted `terminal_log` table. `terminal_log` is still
  written today (the archive log viewer reads it), so the data is there
  — we just stopped replaying it on subscribe in Phase 1.
- Move the superseded plan documents into an archive folder so the
  next designer can read the trail of what didn't work without
  cluttering the active plan index.
- Leave the DB column `agents.cli_session_id` in place (harmless,
  dropping it is a destructive migration we don't need yet).

### Out of scope (on this branch)

- **Designing the new reconnect/scrollback solution.** That belongs in a
  follow-up plan cut from the clean baseline this branch produces.
- Changing `terminal_log` persistence itself.
- Touching anything in the archive log viewer path
  (`/repos/[id]/archive`, `/api/agents/[id]/log`) — that path is
  orthogonal and works.
- `MawWsClient` singleton, `send_keys` raw input, xterm auto-focus
  (`19282b8`), tmux resize sync. All stay.

### Acknowledged baseline bug

The pre-Phase-1 byte-log replay was itself buggy — for TUI CLIs with no
alt-screen buffer it replayed every intermediate repaint back-to-back,
producing 3–4 stacked banners. The user has explicitly accepted this
trade: a known, simple bug is a better starting point for a fresh design
than six overlapping partial fixes.

## What the next plan needs to solve

Written here so the follow-up planner has an unambiguous problem
statement, independent of this branch's scope:

- **Problem:** after a TUI CLI (Claude Code in particular) has been
  running in a tmux pane for a while, a fresh xterm attaching to that
  pane cannot be rehydrated purely from the pane's current byte state
  or from a byte-log replay.
  - The pane's current state is a rendered-grid snapshot — replaying
    it feeds TUI repaints back into a fresh terminal parser that
    doesn't have the prior state the repaints assumed.
  - The byte log contains every transient repaint since spawn, which
    stacks on reconnect.
  - tmux `capture-pane` gives only the visible region for alt-screen-
    less TUIs; Ctrl-O style toggles mutate the buffer in ways
    `capture-pane` can't round-trip.
- **What's been tried** (see archived plans): snapshot + dedup,
  adapter-driven capture depth, out-of-band JSONL history prepend,
  trailing-blank rstrip + cursor CSI, forced SIGWINCH redraw.
- **What to explore next** (non-exhaustive, for the next planner):
  - Driving the CLI through a tmux client that maintains its own
    virtual screen, and serving that screen to the browser — i.e.
    make xterm a thin client of a server-side tmux-client process
    rather than trying to rehydrate from bytes.
  - Using `ttyd`/`asciinema`-style virtual-terminal replay where the
    server owns the authoritative grid.
  - For Claude Code specifically: render from the JSONL as the primary
    view and only attach to live tmux for the *current* prompt.
