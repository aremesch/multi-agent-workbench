# Spawn Dialog Redesign + Agent Roles Management

## Context

The current "Spawn Agent" dialog tries to do too much: it picks a role, picks a
repo, lets you create a brand-new role inline (but only with name + cli_kind —
no system prompt, no defaults), shows a task body textarea, and then submits.
But:

- Roles can't be edited or deleted from anywhere in the UI.
- Repo defaults to `repos[0]`, ignoring whatever repo the sidebar currently
  shows.
- Git-enabled agents have no way to pick the source branch and no opt-out
  for the worktree.
- Claude Code's two most important runtime knobs — model and
  permission-mode — are not exposed.
- The "task body" textarea **looks** wired up but the content never lands
  in Claude Code's input (`{{task.body}}` is substituted into
  `spec.initialInput` and sent via `tmux send-keys -l` 1.5s after spawn —
  but at that point Claude is still painting its welcome screen, and any
  raw `\n` in the body becomes a literal Enter that submits the partial
  prompt).

Goals:

1. Spawn dialog stays a **select-only** form: pick role, pick repo, optionally
   pick branch + worktree flag + CLI-specific knobs, write task title + body,
   spawn. No role CRUD here.
2. New **Agentenrollen** page (linked from top-right user dropdown) for role
   CRUD with a table view.
3. Claude Code spawns surface model + permission-mode pickers, with sensible
   defaults stored on the role.
4. The task body field actually reaches the running CLI, configurable per
   adapter.
5. Whatever runtime config the user picks (model, mode, branch, body) is
   persisted on the agent row so the archive page can show it.

## Decisions (already agreed)

- **Role defaults + per-spawn override:** roles store their preferred model
  and permission-mode; spawn dialog pre-fills from those, user can override
  for this one spawn.
- **Branch picker:** dropdown of existing local branches only, defaulting
  to the repo's current branch.
- **Initial-prompt delivery:** per-adapter, declared in JSONC. Only two
  modes: `cli-arg` (the body is baked into the CLI's argv at launch) or
  `none` (the adapter doesn't accept an initial prompt). **No `tmux send-keys`
  / paste-buffer fallback** — if a CLI's TUI doesn't accept a real CLI-level
  initial-prompt argument, its adapter is configured `none` and the spawn
  dialog **hides the task-body field** for that adapter.
- **i18n:** new menu item label via `nav.roles` (de: "Agentenrollen", en:
  "Agent roles").
- **Worktree naming:** branch = task slug, dir = task slug. Conflict →
  append `-2`, `-3`, … Worktree checkbox is per-spawn; default on for
  git-enabled adapters.

## Adapter schema + Claude Code capabilities

Extend [adapter.config.schema.ts](src/lib/server/agents/adapters/adapter.config.schema.ts):

```ts
initialInput?:
  | { delivery: 'none' }
  | {
      delivery: 'cli-arg';
      template: string;                      // e.g. "{{task.body}}"
      placement: 'positional-last' | { flag: string };
      // optional: skip the arg entirely when the substituted template is empty,
      // so we don't ship a phantom empty positional. Default: true.
      omitWhenEmpty?: boolean;
    };

capabilities?: {
  model?:           { label: string; values: { id: string; label: string }[]; default?: string; arg: string };
  permissionMode?:  { label: string; values: { id: string; label: string }[]; default?: string; arg: string };
};
```

No string-form backward compat — the existing JSONCs are updated as part of
this change. `cli-arg` is the **only** way to deliver a task body; there is
no post-launch `tmux send-keys` path.

Update [cli-adapters/claude-code.jsonc](cli-adapters/claude-code.jsonc):

- `capabilities.model.values`: `default`, `opus`, `sonnet`, `haiku`
  (omit `--model` flag when value === `default`).
- `capabilities.permissionMode.values`: `default`, `plan`, `acceptEdits`,
  `bypassPermissions` (default = `plan`, arg = `--permission-mode {{value}}`).
- `initialInput`: **verify during implementation** whether Claude Code
  v2.1.x accepts a positional initial prompt while remaining interactive.
  If yes → `{ delivery: 'cli-arg', template: '{{task.body}}', placement: 'positional-last' }`.
  If no → `{ delivery: 'none' }` and the spawn dialog hides the task-body
  textarea for this adapter; users put preamble into the role's
  system_prompt instead.
- The existing `--dangerously-skip-permissions` optionalArg becomes
  mutually exclusive with `permissionMode = bypassPermissions` — UI hides
  the toggle when mode is set.

Other adapters:

- `shell`, `browser`, `browser-stream` → `{ delivery: 'none' }`.
- `codex`, `gemini` → verify each CLI's `--help` for a real
  initial-prompt argument (positional or flag). If found, configure
  `cli-arg` with the right placement; otherwise `none`.

Extend `ConfigDrivenAdapter`'s template var resolver (the `vars` record at
[ConfigDrivenAdapter.ts:166](src/lib/server/agents/adapters/ConfigDrivenAdapter.ts)):
add `spawn.model`, `spawn.permissionMode` (substituted from per-spawn picks
falling back to role defaults). Update `AdapterRegistry.list()` to surface
`capabilities` and `initialInput.delivery` on the public listing so the
frontend knows which fields to render.

## Backend: /agents/new + supervisor

[src/routes/agents/new/+page.server.ts](src/routes/agents/new/+page.server.ts)
already reads `task_title` / `task_body` and slugifies. Changes:

- Read additional form fields: `branch`, `with_worktree`, `model`,
  `permission_mode`.
- Replace `const branch = 'maw/${agentId}'` with **slug-as-branch** plus
  conflict-avoidance suffix (`<slug>`, `<slug>-2`, `<slug>-3`, …). New
  helper in [WorktreeManager.ts](src/lib/server/agents/WorktreeManager.ts):
  `nextFreeBranchName(repoPath, slug)`.
- `startPoint` comes from form.`branch`, defaulting to current HEAD if
  missing.
- If `with_worktree=false` **and** adapter `createWorktree=true`: skip
  worktree creation, ensure the selected branch is checked out in
  `repo.path`, agent's cwd = repo path. (For adapters with
  `createWorktree=false` already, ignore the checkbox.)
- Persist `model`, `permission_mode`, `source_branch` on the agent row.
- Pass `model` + `permissionMode` to `supervisor.spawn` → through to
  `adapter.buildSpawnSpec` as `spawn: { model, permissionMode }`.

[AgentSupervisor.ts:633-639](src/lib/server/agents/AgentSupervisor.ts) —
**delete** the `setTimeout(1500, …) → runtime.enqueueInput(spec.initialInput)`
block entirely. `spec.initialInput` is no longer a thing; the task body
either is part of `spec.args` (cli-arg) or isn't sent at all (none).

The existing `runtime.enqueueInput(...)` method stays — it's still used by
later manual user input from the terminal/dashboard. We only drop the
auto-fire-after-spawn path.

`ConfigDrivenAdapter.buildSpawnSpec`: when `initialInput.delivery === 'cli-arg'`:

- Substitute the template via the existing `vars` resolver.
- If `omitWhenEmpty !== false` and the substituted result is empty/whitespace,
  emit no extra arg.
- Otherwise: append per `placement` — either as the final positional in
  `spec.args`, or as `[flag, value]` pair inserted after the existing args.

No `Tmux.pasteText` is added. No bracketed-paste / ready-event machinery
is introduced.

## Frontend: spawn dialog

[SpawnAgentForm.svelte](src/lib/client/components/SpawnAgentForm.svelte) —
trimmed and extended:

- **Remove** the inline role-creation block (`showNewRole`, `createRole`,
  the new-role form panel, related state). Replace the "+ New role" button
  with a small link "Manage roles →" that navigates to `/roles`.
- **Keep** inline repo creation (per user instruction).
- New optional prop `defaultRepoId?: string`. Initialise `selectedRepoId`
  to `defaultRepoId ?? repos[0]?.id ?? ''`.
- When the selected role's CLI kind has `createWorktree=true` (i.e.
  git-enabled), show two new fields below the repo:
  - `<select name="branch">` populated from
    `GET /api/repos/:id/branches`, defaulting to `current`. Re-fetches on
    `selectedRepoId` change.
  - `<label><input type="checkbox" name="with_worktree" checked> …</label>`
- **The task-body `<textarea>` only renders when the selected adapter's
  `initialInput.delivery === 'cli-arg'`.** For `none` (e.g. shell, browser,
  any TUI without a real initial-prompt arg), the field is hidden entirely
  and a small hint suggests using the role's system prompt for stable
  preamble. The browser-specific URL field continues to take its place
  for browser kinds.
- When the selected role's adapter advertises `capabilities.model`, show:
  - `<select name="model">` populated from capability values; default to
    `role.default_model ?? capability.default`.
- When `capabilities.permissionMode` is advertised, show:
  - `<select name="permission_mode">` populated from capability values;
    default to `role.default_permission_mode ?? capability.default`
    (so claude-code agents default to `plan` mode out of the box).

The page that hosts the dialog must compute `defaultRepoId`. For the
dashboard (`/`) and `/repos/:id` pages, derive it in their
`+page.server.ts` from the URL path: `/repos/:id` → that id, else `null`.
[+page.server.ts](src/routes/+page.server.ts) passes it as a prop to the
modal.

## Verification

E2E walkthrough on `pnpm dev`:

1. Log in, navigate to `/repos/<id>` of a real repo.
2. Open the spawn modal → repo is **pre-selected to that repo**, role
   defaults to first role, no "New role" form in sight, "Manage roles →"
   link visible.
3. Pick a claude-code role → confirm Branch dropdown lists the actual
   local branches with the current one selected, "Use worktree" checkbox
   is ticked, Model + Permission-mode dropdowns appear with role defaults
   (or adapter defaults if role.default_* is null), and Permission-mode
   defaults to **plan**. The task-body textarea is visible IFF the adapter
   advertises `initialInput.delivery === 'cli-arg'`.
4. Type a multi-line task body (3+ lines, include shell metacharacters
   like `'`, `"`, `$`, `` ` ``). Spawn.
5. Verify in the agent's tmux pane: Claude Code launched with the body
   baked into argv (visible in `ps aux | grep claude` or in the adapter's
   resolved-spec log). The CLI either honours the initial prompt or — if
   we determine during implementation that it doesn't — we flip its
   adapter to `delivery: 'none'` and re-run step 3 to confirm the
   task-body textarea is hidden.
6. Verify in DB: `tasks.body` matches the textarea, `agents.model`,
   `agents.permission_mode`, `agents.source_branch` match the form.
7. Verify the worktree dir is named after the slug and the new branch is
   the slug (not `maw/<ulid>`). Spawn a second agent with the same title
   → branch becomes `<slug>-2`.
8. Toggle worktree OFF → spawn → no worktree created, agent's cwd is the
   repo path, repo is on the selected branch.
9. Open the top-right menu → click "Agentenrollen" → table view loads.
   Create a new role with system_prompt + default_model = `opus` + default
   mode = `plan`. Spawn an agent → dropdowns pre-fill with Opus + plan.
10. Edit the role → change defaults → spawn → confirm reflected.
11. Delete the role with no agents using it → succeeds. Delete one with
    active agents → 409 + UI shows error.
12. Switch locale (de ↔ en) → menu label flips correctly.

## Notes / open questions for implementation

- **CLI-arg verification per adapter:** before merging, confirm whether
  each CLI (claude-code, codex, gemini) accepts an initial prompt as a
  CLI argument while remaining interactive. Configure each adapter's
  `initialInput` accordingly — `cli-arg` (with the right `placement`) if
  yes, `none` if no. Document the choice in a code comment on the JSONC.
- **Argv length / quoting:** even very long task bodies go through
  `tmux new-session sh -lc "… <cmd> …"` (see [TmuxSession.ts:101](src/lib/server/agents/TmuxSession.ts)).
  The body must be safely shell-quoted by the spec builder. ARG_MAX on
  Linux is ~2 MiB so practical task-body sizes are fine, but the quoting
  must handle `'`, `"`, `$`, `` ` ``, and newlines correctly. Add a unit
  test covering all of these.
- **Remote-branch listing:** not in v1. The dropdown could later show
  `origin/foo` entries that auto-fetch on selection; out of scope here.
- **`--dangerously-skip-permissions` ↔ `permission_mode`:** when the user
  picks any `permission_mode` other than the default, hide the skip-perms
  toggle. When `bypassPermissions` is picked, treat it as equivalent and
  drop the redundant flag.
- **Branch checkout for non-worktree git agents:** running `git checkout
  <branch>` in the live repo can fail on a dirty tree. Surface that error
  from the spawn action rather than letting the supervisor crash.
