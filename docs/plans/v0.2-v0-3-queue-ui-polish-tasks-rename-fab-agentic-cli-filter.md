# v0.3 Queue UI polish — Tasks rename, FAB, agentic-CLI filter

## Context

Stage 1 of the queue feature shipped with developer-shorthand labels ("Queue", "New entry"), a toolbar button instead of a FAB, and a role picker that surfaces every adapter — including non-automatable ones (browser, shell). The user wants four targeted polish edits:

1. Rename the user-visible label "Warteschlange" → "Aufgaben" (DE), "Queue" → "Tasks" (EN). **Labels only** — the route stays `/queue`, the DB / code identifiers stay `queue_entries` / `QueueScheduler` / etc. The existing `tasks` DB table (per-agent task records) keeps its meaning; only the human-facing label collides, which is fine — both mean "an agent's task" to the user.
2. Replace the toolbar **"+ New entry"** button on the `/queue` page with a floating action button matching the repo dashboard's `.fab` pattern (`/repos/[id]/+page.svelte` lines 178, 279–296).
3. Modal title becomes **"New task" / "Neue Aufgabe"** (was "New entry" / "Neuer Eintrag").
4. The queue's role dropdown should only list adapters that make sense to automate — claude-code, codex, gemini — and hide browser, browser-stream, shell. The /agents/new spawn flow keeps showing all adapters.

For (4), the user confirmed they want a new explicit adapter-level flag rather than a hardcoded allowlist or a proxy-via `initialInputDelivery`. The flag is `agenticCodingCli: boolean` (camelCase to match existing schema keys like `createWorktree`, `acceptsImageAttachment`, `initialInputDelivery`).

### 4. `agenticCodingCli` adapter flag

Adapters declare whether they are agentic coding CLIs (claude-code, codex, gemini) or interactive/manual tools (shell, browser, browser-stream). The queue role picker filters by this flag.

**Schema** — [src/lib/server/agents/adapters/adapter.config.schema.ts](src/lib/server/agents/adapters/adapter.config.schema.ts):
```ts
// inside adapterConfigSchema:
agenticCodingCli: z.boolean().default(false),
```
Defaults to `false` so an unmarked adapter is silently excluded from the queue (conservative — opt-in).

**Registry listing** — [src/lib/server/agents/adapters/AdapterRegistry.ts](src/lib/server/agents/adapters/AdapterRegistry.ts):

Add `agenticCodingCli: boolean` to `AdapterListing` and surface it in `list()`:
```ts
return {
  kind: e.config.kind,
  displayName: e.config.displayName,
  agenticCodingCli: e.config.agenticCodingCli,
  …
};
```

**JSONC files**:
- [cli-adapters/claude-code.jsonc](cli-adapters/claude-code.jsonc) → add `"agenticCodingCli": true`
- [cli-adapters/codex.jsonc](cli-adapters/codex.jsonc) → add `"agenticCodingCli": true`
- [cli-adapters/gemini.jsonc](cli-adapters/gemini.jsonc) → add `"agenticCodingCli": true`
- shell / browser / browser-stream JSONCs: **no edits** — the schema default `false` is correct, and an explicit `"agenticCodingCli": false` would add noise.

**Schema** — [schemas/adapter.schema.json](schemas/adapter.schema.json): mirror the new field so the JSONC IDE tooltips still work.

**Queue page filter** — [src/routes/queue/+page.server.ts](src/routes/queue/+page.server.ts):

```ts
const cliKinds = locals.supervisor.registry.list();
const agenticKinds = new Set(cliKinds.filter((k) => k.agenticCodingCli).map((k) => k.kind));
const allRoles = listRoles(locals.user.id);
const queueRoles = allRoles.filter((r) => agenticKinds.has(r.cli_kind));
return {
  …
  roles: queueRoles,
  cliKinds,  // still all kinds, so SpawnAgentForm's downstream lookups by cli_kind keep working
  …
};
```

The roles array fed to `SpawnAgentForm` is already filtered, so the form's `<select name="role_id">` only renders agentic options — no `mode='queue'`-conditional branch needed inside the form component.

`cliKinds` keeps the full list because the form's capability lookups (`selectedAdapter`, `showGitFields`, etc.) join role.cli_kind → cliKind by `kind`, and we don't want a NULL adapter listing when the user picked an agentic role.
