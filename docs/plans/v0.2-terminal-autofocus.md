# Auto-focus terminal when the agent modal opens

## Context

After spawning a new agent or reopening an existing agent from the sidebar,
the terminal modal appears but keyboard input is not routed to xterm until
the user manually clicks inside the terminal area. That extra click is
friction on what is otherwise a direct "open → type" workflow, and it is
the single biggest UX paper-cut in the daily dashboard loop.

**Why it happens today:**

- `src/lib/client/components/Modal.svelte` drives a native `<dialog>` via
  `dialog.showModal()` (line 46). The browser's modal-dialog focus trap
  then automatically focuses the first focusable element in the dialog,
  which is the `×` close `<button>` in `.maw-modal-head` (line 74).
- xterm's textarea proxy is focusable, but xterm itself is mounted lazily:
  `Terminal.svelte` dynamic-imports `@xterm/xterm` inside `onMount`
  (lines 66–87), so at the moment `showModal()` runs, xterm doesn't yet
  exist in the DOM and can't win the focus trap.
- Nothing in `Terminal.svelte` or `AgentTerminalPanel.svelte` ever calls
  `term.focus()`, so even after the dynamic import resolves, focus stays
  on whatever the dialog initially focused.

## Approach

Call `term.focus()` once, at the end of the xterm initialization inside
`Terminal.svelte`. Because `{#key openAgent.id}` in both
`src/routes/+page.svelte` (line 98) and `src/routes/repos/[id]/+page.svelte`
(line 126) unmounts/remounts `<AgentTerminalPanel>` (and therefore
`<Terminal>`) on every open and on every agent switch, a single focus call
at the tail of Terminal's `onMount` async flow covers both the spawn flow
and the reopen flow with no additional plumbing.

Ordering is favorable: `dialog.showModal()` runs synchronously from
Modal's `$effect`, which sets the native focus trap on the close button.
The dynamic `import('@xterm/xterm')` resolves a microtask later, at which
point we call `term.focus()` — that runs *after* the native focus trap
ran, so we win and focus lands on xterm's textarea.

## Change

**File:** `src/lib/client/components/Terminal.svelte`

Add a single `term.focus()` call immediately after the initial
`fit.fit()` try/catch block (around line 96), before the `onData`/
`onResize` listener wiring. xterm's `.focus()` is a no-op-safe method
that moves browser focus to xterm's internal textarea helper, which is
exactly what a manual click on the canvas would do.

No new props, no new events, no change to `AgentTerminalPanel.svelte`,
no change to `Modal.svelte`. The fix is one line inside the existing
`onMount` async IIFE.

## Why not an explicit `autofocus` prop or Modal-side focus?

- A Modal-level prop would have to reach into arbitrary children and
  know which element to focus — extra indirection for zero benefit,
  since the only consumer we care about is the terminal panel.
- An `AgentTerminalPanel`-level `term?.focus()` after mount would race
  the dynamic import inside `Terminal.svelte` and need its own
  "pending" queue, mirroring the existing `write()`/`clear()` pending
  buffer. Doing it inline where the Terminal instance actually becomes
  live is simpler and has no races.

## Scope note (out of scope)

- Not restoring focus after the in-panel `pendingPrompt` section
  disappears. The user's report is only about the open path.
- Not touching the dedicated `/agents/[id]` page; the same change
  benefits it for free (same `<Terminal>` component, no modal
  focus-steal to compete with), and it is not affected by the reported
  bug.

## Verification

1. `pnpm dev` and open the dashboard.
2. **Spawn path:** click the `+` FAB, fill the spawn form, submit. When
   the terminal modal opens, type a character without clicking — it
   should appear in the agent's prompt.
3. **Reopen path:** close the modal, click the agent's card (or its
   entry in the left sidebar tree). Modal reopens; type a character
   without clicking — it should land in xterm.
4. **Agent switch:** with one agent open, click a different agent in
   the sidebar. The `{#key}` re-mounts the panel; typing should go to
   the newly selected agent without a click.
5. **Esc still closes:** press Esc with focus on xterm — the native
   `<dialog>` close-on-Esc behavior should still fire (xterm's textarea
   does not swallow Esc at the dialog level in default config; verify).
6. **Smoke adapter:** repeat (2) using `cli-adapters/shell.jsonc` so
   the test doesn't depend on having claude/codex/gemini installed.
7. `pnpm check` stays at 0/0.
