# v0.3 Queue + Scheduler

## Context

Today MAW spawns one agent at a time through the redesigned [SpawnAgentForm.svelte](src/lib/client/components/SpawnAgentForm.svelte) (shipped in [v0.2-spawn-dialog-redesign-agent-roles-management.md](docs/plans/v0.2-spawn-dialog-redesign-agent-roles-management.md)). The user fills in role + repo + branch + worktree flag + model + permission-mode + title + (conditional) body + optional args, hits **Spawn**, the form action calls `supervisor.spawn()`, the agent runs to exit. There is no way to pre‑plan a batch of these, no priority, no dependencies, no "start at 3 AM", no "don't run while another agent touches this repo".

The user wants exactly that: a queue UI where many requests are defined up front (capturing **every** field the spawn form now captures, plus priority / deps / scheduled-for / exclusivity), and a scheduler that promotes them to running agents one (or N) at a time, reacting to agent exit immediately so the next one starts without manual action.

We build this **natively** in MAW rather than on top of an external issue tracker:

- The MAW data model already encodes every structured field a queue entry needs. After the spawn redesign, `roles` carries `default_model` / `default_permission_mode` / `system_prompt` / `cli_kind` / `default_args_json` / `tool_config_json`; `agents` freezes the chosen `model` / `permission_mode` / `source_branch`; `worktrees` and `tasks` are unchanged. An issue tracker would force this structured shape into YAML inside an issue body.
- The supervisor already fires precise, event‑driven exit signals (`finishAsExited` resolved by the `tmux wait-for` watcher in [AgentSupervisor.ts](src/lib/server/agents/AgentSupervisor.ts)). A native scheduler reacts in milliseconds; an external API forces polling and seconds of latency.
- A native queue is small surface (one new table, one scheduler module, one route, one extension to the existing spawn form).

Issue trackers (Gitea, GitHub Issues) are **opt‑in bridges** layered on top — the user can drive the queue from existing workflows without making the queue depend on either tracker.

The plan is split into two stages so they can land as separate reviewable PRs. Both are in v0.3 scope.

## Stage 1 — Native queue + scheduler

### Data model

New migration [`migrations/010_queue_entries.sql`](migrations/010_queue_entries.sql) (next free slot after `009_role_and_agent_runtime_config.sql`).

#### `queue_entries`

Columns mirror what the post‑redesign `SpawnAgentForm` collects, plus queue‑specific metadata. Capability‑driven fields (model, permission_mode, body, target_url) are nullable and only filled when the chosen adapter exposes them — exactly like agents/roles do today.

| column | type | notes |
|---|---|---|
| `id` | TEXT PK | ULID |
| `user_id` | TEXT FK users | owner-scoped, indexed |
| `role_id` | TEXT FK roles | required |
| `repo_id` | TEXT FK repos | required (locking unit) |
| `title` | TEXT | non-empty; promoted task title + slug source for the branch name |
| `body` | TEXT NULL | the prompt; only stored when adapter `initialInputDelivery === 'cli-arg'` |
| `target_url` | TEXT NULL | only for browser-kind adapters (`isAnyBrowserKind(cli_kind)`) |
| `model` | TEXT NULL | only when adapter declares `capabilities.model`; falls back to `role.default_model` at promote time if NULL |
| `permission_mode` | TEXT NULL | only when adapter declares `capabilities.permissionMode`; falls back to `role.default_permission_mode` at promote time if NULL |
| `source_branch` | TEXT NULL | branch to base the worktree on (or check out into repo root if `with_worktree=0`); resolved against `GET /api/repos/:id/branches` on save |
| `with_worktree` | INT (bool) | 1 = isolated worktree, 0 = run in repo root; default 1 for adapters with `createWorktree: true`, always 0 for adapters without |
| `optional_args_json` | TEXT | `{ [argId]: boolean }` mirroring spawn form's `optionalArgs[*]` toggles |
| `priority` | INT | higher first, default 0 |
| `depends_on_json` | TEXT | JSON array of `queue_entries.id` |
| `scheduled_for` | INT NULL | unix ms; NULL = "as soon as ready" |
| `exclusive` | INT (bool) | 1 = while running, no other agent (queued or human-spawned) runs on this repo; `with_worktree=0` implies `exclusive=1` regardless of this flag |
| `status` | TEXT | `pending` \| `blocked` \| `ready` \| `running` \| `done` \| `failed` \| `cancelled` |
| `agent_id` | TEXT FK agents NULL | set when promoted |
| `external_source_json` | TEXT NULL | populated by Stage 2 bridge |
| `last_error` | TEXT NULL | failure reason (e.g. `"dependency failed"`, `"role deleted"`, `"branch missing"`, agent exit reason) |
| `created_at` / `updated_at` / `started_at` / `completed_at` | INT | timestamps |

Indexes: `(user_id, status)`, `(user_id, scheduled_for)`, `(repo_id, status)`.

Status semantics:
- `pending` — newly created, not yet evaluated
- `blocked` — has unresolved deps OR `scheduled_for > now` OR pre‑spawn validation failed transiently
- `ready` — eligible but waiting on a slot
- `running` — agent spawned, linked via `agent_id`
- `done` — linked agent exited with code 0
- `failed` — linked agent crashed / non‑zero exit, or pre‑spawn validation hard‑failed (role/repo deleted, branch gone, adapter validation refused)
- `cancelled` — user cancelled (kills linked agent if running)

Existing `tasks` and `worktrees` tables are **unchanged**. On promotion the scheduler creates a fresh `tasks` row (status `'active'`) and a fresh `worktrees` row with branch name resolved via `WorktreeManager.nextFreeBranchName(repoPath, slug(title))` — same code path the spawn form already uses.

#### Concurrency settings

Stored on existing `user_settings` JSON (same pattern as `ui.sidebar.collapsed`):
- `queue.maxConcurrentGlobal` — int, default 1
- `queue.maxConcurrentPerRepo` — int, default 1
- `queue.perRepoOverrides` — `{ [repoId]: int }`, optional

Per‑repo overrides editable from [RepoEditDialog.svelte](src/lib/client/components/RepoEditDialog.svelte) (extra numeric field).

### Scheduler

New singleton [`src/lib/server/queue/Scheduler.ts`](src/lib/server/queue/Scheduler.ts), wired into the existing `AgentSupervisor`.

```
class QueueScheduler {
  start(supervisor, db): void
  stop(): void
  scheduleTick(): void           // debounced 50ms, serialized via async lock
  promoteEntry(entryId): void    // manual "run now"
  cancelEntry(entryId): void     // marks cancelled, kills agent if running
}
```

#### Tick logic (single async‑locked function)

1. **Refresh state.** For every `pending` / `blocked` entry: re‑evaluate deps (all referenced entries must be `done`; if any dep is `failed`/`cancelled`, entry stays `blocked` with `last_error="dependency failed"`). If deps OK and `scheduled_for ≤ now` → `ready`. Else → `blocked`.
2. **Slot accounting.** Count running entries by repo. Add manually‑spawned agents (rows in `agents` with status in `running|spawning|waiting_input|idle`) to the per‑repo count — but **not** the global count (the cap is on queue work, not human-driven work).
3. **Pick & promote.** Order `ready` entries by `priority DESC, created_at ASC`. For each:
   - Break if global queue running count ≥ `maxConcurrentGlobal`.
   - Treat the entry as exclusive if `exclusive=1` OR `with_worktree=0`.
   - Skip if any running agent on the repo is `exclusive` (from queue) or running in the repo root (no worktree), or if this entry is exclusive and any agent is running on the repo, or per‑repo cap hit.
   - **Pre‑spawn validation** (fail‑hard on permanent errors, soft‑block on transient): role exists and belongs to user; repo exists and path resolves; `source_branch` (if set) appears in `WorktreeManager.listBranches(repoPath)`; if `model`/`permission_mode` set, they're still valid for the adapter's current capabilities. Permanent failure → status=`failed`, `last_error` set, skip. Transient failure → status=`blocked`, will retry next tick.
   - Otherwise **promote**: build a `SpawnAgentArgs` from the entry (apply role defaults for any NULL capability fields), call `supervisor.spawn()`, update entry → `running` + `agent_id` + `started_at`. The branch name is resolved fresh inside `supervisor.spawn()` via the existing `WorktreeManager.nextFreeBranchName` call — re-running a queue entry after failure naturally gets a fresh branch.
4. **Re‑arm scheduled‑for timer.** Single `setTimeout` pointed at the earliest future `scheduled_for` across the user's entries; cleared and re‑armed at the end of every tick.

#### Tick triggers

- Agent transitions to a terminal state — hook into [`AgentSupervisor.finishAsExited`](src/lib/server/agents/AgentSupervisor.ts) (linked agent's queue entry transitions to `done`/`failed`, then a tick fires).
- Queue entry created / edited / cancelled (API routes call `scheduleTick()`).
- `scheduled_for` timer fires.
- Bootstrap finishes — start scheduler from [bootstrap.ts](src/lib/server/bootstrap.ts), reconcile any `running` entries whose linked agents are already in a terminal state (matches existing tmux reattach pattern).

### API routes

Under `src/routes/api/queue/`. All gated by `locals.user.id` ownership, JSON in/out. Validation **reuses the spawn form validators** from [`/agents/new/+page.server.ts`](src/routes/agents/new/+page.server.ts) (e.g. capability sanitization via [`capabilityValidation.ts`](src/lib/server/agents/adapters/capabilityValidation.ts), branch existence via `WorktreeManager.listBranches`) — extract a shared `validateSpawnFields(input)` helper if needed.

- `GET /api/queue` — list (filter by `status`, `repo_id`)
- `POST /api/queue` — create (all spawn fields + queue-specific)
- `GET /api/queue/[id]`
- `PUT /api/queue/[id]` — edit (rejected when `running`/`done`)
- `DELETE /api/queue/[id]` — cancel (kills linked agent if running, via existing supervisor stop)
- `POST /api/queue/[id]/promote` — manual "run now" (skips slot checks)
- `POST /api/queue/reorder` — bulk priority update from drag‑reorder

### UI

#### Reuse `SpawnAgentForm` rather than fork it

The redesigned spawn form already implements the entire capability‑driven conditional UI (branch dropdown async‑loaded, worktree toggle, model/permission‑mode pickers, body vs target_url switch, optional‑args toggles). Re‑implementing that in a parallel "queue entry form" would mean shipping ~300 lines of duplicate Svelte that drifts as adapters evolve.

Plan: add a `mode: 'spawn-now' | 'add-to-queue'` prop and an `onQueue(payload)` callback to `SpawnAgentForm.svelte`. When `mode='add-to-queue'`:
- Replace the **Spawn** primary button with **Add to queue**.
- Reveal an extra panel above the actions row with queue-specific fields: `priority` (number), `scheduled_for` (datetime-local, optional), `exclusive` (checkbox, auto‑forced + disabled when `with_worktree=false`), `depends_on` (multiselect of the user's open entries — fetched on dialog open from `GET /api/queue?status=pending,blocked,ready,running`).
- On submit POST to `/api/queue` instead of the form action.

On the dashboard, the existing "Spawn agent" CTA stays `mode='spawn-now'`. The new **Queue** page (below) gets a "+ New entry" button that opens the same component in `mode='add-to-queue'`.

#### `/queue` (global) — new route

`src/routes/queue/+page.{svelte,server.ts}`. Server load returns the user's entries grouped by status. Page layout:
- Toolbar: concurrency summary, **New entry**, **Reorder mode** toggle.
- A live "running now" panel listing in‑flight queue entries, each with a button that opens the existing terminal modal for the linked agent.
- A grouped table (`ready` → `blocked`/`scheduled` → terminal): title, repo, role, model badge, branch, priority, deps, scheduled-for, status badge, edit/cancel actions.
- Drag‑reorder rewrites priority sequence client‑side, debounced 500 ms (reuses the gridstack debounce pattern from [AgentGrid.svelte](src/lib/client/components/AgentGrid.svelte)), POSTs to `/api/queue/reorder`.

#### `/repos/[id]/queue` — new route

Same component, scoped to one repo. Linked from the per‑repo dashboard.

### WebSocket protocol

## Stage 2 — Gitea + GitHub Issues bridges

Lands as a separate PR after Stage 1 is in.

### Data model

`queue_entries.external_source_json` is already provided. New migration `migrations/011_tracker_configs.sql`:

#### `tracker_configs`

| column | type | notes |
|---|---|---|
| `id` | TEXT PK | ULID |
| `user_id` | TEXT FK | |
| `repo_id` | TEXT FK repos | one config per repo |
| `tracker_kind` | TEXT | `gitea` \| `github` |
| `base_url` | TEXT | e.g. `https://gitea.example.com`; empty for github.com |
| `tracker_repo` | TEXT | `owner/name` |
| `token_encrypted` | TEXT | API token encrypted with new `MAW_SECRET_KEY` env (introduced by this stage, documented in [.env.example](.env.example)) |
| `import_label` | TEXT | label that flags issues for import (default `maw:queue`) |
| `default_role_id` | TEXT FK roles | role assigned to imports lacking an override |
| `auto_close_on_done` | INT (bool) | post comment + close issue when entry done |
| `last_polled_at` | INT | watermark |

### Polling

New `src/lib/server/queue/bridges/`:
- `Gitea.ts` — Gitea REST API (`/api/v1/repos/{owner}/{repo}/issues?labels=…&since=…`); no SDK dep, plain `fetch`.
- `Github.ts` — `@octokit/rest` (justified: official, maintained, small, exact rate-limit handling).
- `BridgePoller.ts` — singleton; iterates `tracker_configs` every 60 s, dedups by `(tracker_kind, tracker_repo, issue_number)` against `external_source_json`, creates `queue_entries` for new issues with the import label.

Issue body parsed for an optional fenced block tagged `maw` carrying YAML overrides (`priority`, `scheduled_for`, `exclusive`, `depends_on`, `role`, `model`, `permission_mode`, `source_branch`, `with_worktree`, `body`). Without the block, the issue title becomes the queue entry title and the issue body becomes the prompt; capability fields fall back to role defaults (existing behaviour at promote time).

### Status sync (back to tracker)

When a queue entry transitions to `done` or `failed`, the bridge posts a comment on the source issue with agent ID, exit code, branch name, and commit SHA(s) from the existing `agent_commits` table. If `auto_close_on_done`, close the issue on `done`.

### UI

- New `/settings` section: list / create / edit / delete tracker configs, with an inline **Test connection** button.
- The repo edit dialog gets a **Tracker** tab.
- Queue rows display the source issue link.

### Webhooks (deferred)

Polling is the default. A webhook receiver (`/api/webhooks/[trackerKind]/[configId]`) is documented as a follow-up — it needs public reachability and HMAC verification.

### Tests

- Vitest for issue parsing + dedup logic.
- Vitest for `MAW_SECRET_KEY` encrypt/decrypt round-trip.
- Manual: import one issue from a real Gitea instance, confirm a queue entry appears at the expected scheduled time and that capability fields fall back to role defaults correctly.

## Verification

1. **Migration smoke** — wipe `MAW_DB_PATH`, run `pnpm dev`, confirm migration 010 (and 011 in Stage 2) applies cleanly on both fresh and upgraded DBs.
2. **Unit tests** — `pnpm test:unit src/lib/server/queue/`.
3. **Serial mode** — `maxConcurrentGlobal=1`, queue three `shell`‑adapter entries with priorities 10/5/1, confirm execution order 10 → 5 → 1, one at a time, no API key needed.
4. **Dependencies** — queue A and B with `B.depends_on=[A.id]`, confirm B stays `blocked` until A is `done`, then auto‑runs.
5. **Scheduled-for** — queue C with `scheduled_for = now + 60s`, confirm `blocked` until the timer fires, then auto‑runs.
6. **Exclusive** — mark D `exclusive`, spawn a manual agent on the same repo via the existing form, queue D, confirm D stays `blocked` until the manual agent exits.
7. **Worktree-off implies exclusive** — queue E with `with_worktree=0`, queue F (non‑exclusive) on the same repo, confirm F stays `blocked` until E is done.
8. **Per-repo cap** — `maxConcurrentPerRepo=2`, queue 3 non-exclusive entries on the same repo, confirm only 2 run at once.
9. **Pre-spawn hard-fail** — queue G referencing a role, delete the role via `/roles`, observe G transitions to `failed` with `last_error="role deleted"` on next tick.
10. **Branch validation** — queue H against a non-existent source branch, observe G transitions to `failed` with `last_error` mentioning the missing branch.
11. **Capability fallback** — queue I with `model=NULL` against a role with `default_model='sonnet'`, observe the spawned agent row has `model='sonnet'`.
12. **Live UI** — open `/queue` in two browsers, edit an entry in one, confirm the other updates over WS without refresh.
13. **e2e** — `pnpm test:e2e tests/e2e/queue-*.spec.ts`.
14. **Stage 2 bridge** — add a Gitea config pointing at a test repo, label an issue `maw:queue`, wait one poll cycle, confirm a `queue_entry` is created with the correct `external_source_json` and the issue link appears in the UI.

## Out of scope (future plans)

- Webhook receivers for trackers.
- Multi-step pipelines (a queue entry that spawns a chain of follow-ups).
- LLM "overseer" prioritization.
- Cron-style recurring entries — only one-shot `scheduled_for` in v0.3.
- Per-CLI-kind quota (e.g. "max 2 Claude Code spawns at once" regardless of repo) — punt until usage suggests it's needed.
