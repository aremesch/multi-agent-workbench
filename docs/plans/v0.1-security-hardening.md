# v0.1 Security Hardening

## Context

Pre-pentest hardening pass identified in
[`dazzling-popping-stallman.md`](dazzling-popping-stallman.md). Goal:
close the trivially-fixable gaps so the external pentest measures the
intended posture, not low-hanging fruit. Scope is intentionally narrow
— no PWA, no 2FA, no overseer. Everything here is a small, surgical
change in existing files.

**Priorities, in order:**

1. CSRF actually enforced on JSON routes.
2. WebSocket `Origin` check.
3. Password change flow + `must_change_password` gate for bootstrap.
4. Auth event log + fail2ban drop-ins.
5. Per-IP login rate limit.
6. Security response headers + `SameSite=strict` on session cookie.

## Step-by-step execution

Run from the worktree root. One commit per numbered step.

### 1. Migration — schema for hardening

Create `migrations/003_user_security.sql`:

```sql
ALTER TABLE users ADD COLUMN must_change_password INTEGER NOT NULL DEFAULT 0;
ALTER TABLE users ADD COLUMN password_updated_at INTEGER;

CREATE TABLE auth_events (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  ts         INTEGER NOT NULL,
  event      TEXT    NOT NULL,  -- login_ok | login_fail | pwchange_ok | pwchange_fail | session_revoked | rate_limited
  user_id    TEXT,
  username   TEXT,
  ip         TEXT,
  user_agent TEXT,
  detail     TEXT

CREATE INDEX idx_auth_events_ts ON auth_events(ts);
CREATE INDEX idx_auth_events_ip_ts ON auth_events(ip, ts);
```

Extend `UserRow` in `src/lib/server/db/types.ts` with the two new columns (both optional `number`/`number|null`). Bump the bootstrap user insert in whatever seeds the initial user to set `must_change_password=1` **only when the password came from `MAW_BOOTSTRAP_PASSWORD`**.

Verify: delete `data/maw.sqlite` (or the configured path), restart, log in with bootstrap creds, confirm row has `must_change_password=1`.

### 2. Auth log + client-IP helper

`src/lib/server/net/clientIp.ts`:

```ts
import type { RequestEvent } from '@sveltejs/kit';
import { getConfig } from '../config.js';

export function clientIp(event: RequestEvent | { request: Request; getClientAddress: () => string }): string {
  const cfg = getConfig();
  if (cfg.trustProxy) {
    const xff = event.request.headers.get('x-forwarded-for');
    if (xff) return xff.split(',')[0].trim();
  }
  return event.getClientAddress?.() ?? '-';
}
```

`src/lib/server/auth/authLog.ts`:

```ts
import { appendFileSync } from 'node:fs';
import { getConfig } from '../config.js';
import { insertAuthEvent } from '../db/queries.js';

const SAFE = /[\r\n\t]/g;
const clean = (s: string | null | undefined) => (s ?? '-').replace(SAFE, ' ').slice(0, 256);

export function logAuth(event: string, fields: {
  userId?: string | null;
  username?: string | null;
  ip: string;
  userAgent?: string | null;
  detail?: string | null;
}) {
  const ts = Math.floor(Date.now() / 1000);
  const line = `${new Date(ts * 1000).toISOString()} ${clean(event)} user=${clean(fields.username)} ip=${clean(fields.ip)} ua="${clean(fields.userAgent)}"\n`;
  try { appendFileSync(getConfig().authLogPath, line); } catch { /* never throw from logger */ }
  try { insertAuthEvent({ ts, event, user_id: fields.userId ?? null, username: fields.username ?? null, ip: fields.ip, user_agent: fields.userAgent ?? null, detail: fields.detail ?? null }); } catch {}
}
```

Config additions (in `src/lib/server/config.ts`): `trustProxy` (bool, default `false`), `authLogPath` (string, default `${dataDir}/auth.log`), `loginRateLimit` (parsed `"10/60"` → `{count:10, windowSeconds:60}`).

### 3. Rate limiter

`src/lib/server/auth/rateLimit.ts` — simple in-memory, enough for single-node:

```ts
const buckets = new Map<string, number[]>();

export function checkRate(key: string, count: number, windowSeconds: number): boolean {
  const now = Date.now();
  const cutoff = now - windowSeconds * 1000;
  const hits = (buckets.get(key) ?? []).filter(t => t > cutoff);
  hits.push(now);
  buckets.set(key, hits);
  if (buckets.size > 10_000) {
    // occasional GC
    for (const [k, v] of buckets) if (v[v.length - 1] < cutoff) buckets.delete(k);
  }
  return hits.length <= count;
}
```

### 4. Login hardening

Edit `src/routes/login/+page.server.ts`:

- At the top of the `default` action:
  - Resolve `ip = clientIp(event)`, `ua = request.headers.get('user-agent')`.
  - `if (!checkRate('login:'+ip, cfg.loginRateLimit.count, cfg.loginRateLimit.windowSeconds)) { logAuth('rate_limited', {...}); return fail(429, {...}) }`.
- On wrong creds: `logAuth('login_fail', { username, ip, userAgent: ua })` and preserve the constant-time `verifyPassword` path so timing doesn't leak enumeration.
- On success: `logAuth('login_ok', ...)`. If `user.must_change_password`, redirect to `/settings/security` instead of `/`.
- In `logout` action: `logAuth('session_revoked', ...)`.

### 5. Password change

`src/routes/api/user/password/+server.ts` (POST):

- Require auth (`locals.user`).
- `verifyCsrf(request, cookies)`.
- Body `{ currentPassword, newPassword }`. Minimum length 12.
- Re-verify `currentPassword` with `verifyPassword`.
- `hashPassword(newPassword)` → `updateUserPassword(user.id, hash)`, set `must_change_password=0`, `password_updated_at=now`.
- Delete **all other** sessions for this user (keep current one). New query in `queries.ts`: `deleteOtherSessions(userId, keepSessionId)`.
- `logAuth('pwchange_ok' | 'pwchange_fail', ...)`.

`src/routes/settings/security/+page.svelte` + `+page.server.ts`:

- Simple form: current password, new password, confirm.
- `+page.server.ts` `load` redirects to `/login` if unauthenticated.
- Submit hits the JSON endpoint with the CSRF header.

### 6. CSRF on JSON routes

For each of the five API routes, insert at the top of each non-GET handler:

```ts
import { verifyCsrf } from '$lib/server/auth/csrf.js';
...
export const POST: RequestHandler = async ({ request, cookies, locals }) => {
  verifyCsrf(request, cookies); // throws → 403
  ...
};
```

Ensure `verifyCsrf` throws a `error(403, 'csrf')` so SvelteKit short-circuits cleanly. Update the function in `src/lib/server/auth/csrf.ts` if it currently just returns `boolean`.

Client side: in the fetch wrapper (find via `rg -n "fetch\\(" src/lib/client`), read `maw_csrf` from `document.cookie` and attach `x-csrf-token` header to every non-GET request. If there is no central wrapper, add one now — do not spray `document.cookie` reads across callsites.

### 7. WebSocket Origin check

`src/lib/server/ws/hub.ts` around the existing upgrade auth block (~line 305):

```ts
const origin = req.headers.origin;
const allowed = getConfig().publicOrigin; // new config: e.g. "https://maw.remesch.at"
if (origin && origin !== allowed) {
  ws.close(4403, 'origin');
  return;
}
```

Config: `MAW_PUBLIC_ORIGIN` env, required in prod, optional in dev (log a warning if unset). If the prod server sits behind Caddy, `allowed` must match the browser-visible origin, not the upstream.

### 8. Session cookie → `SameSite=strict`

`src/lib/server/auth/session.ts` line 53: `sameSite: 'lax'` → `'strict'`. Test: opening a bookmarked `/` in a new tab should still work after login (strict allows same-site top-level nav; it only blocks cross-site requests carrying the cookie).

### 9. Security headers + password-gate redirect

`src/hooks.server.ts` — add to the existing `handle`:

```ts
const response = await resolve(event);
response.headers.set('Content-Security-Policy',
  "default-src 'self'; img-src 'self' data:; style-src 'self' 'unsafe-inline'; script-src 'self'; connect-src 'self' wss:; frame-ancestors 'none'");
response.headers.set('Strict-Transport-Security', 'max-age=63072000; includeSubDomains; preload');
response.headers.set('X-Content-Type-Options', 'nosniff');
response.headers.set('Referrer-Policy', 'strict-origin-when-cross-origin');
response.headers.set('Permissions-Policy', 'geolocation=(), microphone=(), camera=()');
return response;
```

Also in `handle`, before `resolve`: if `event.locals.user?.must_change_password` and the path is not `/settings/security`, `/api/user/password`, `/login`, or a static asset → `throw redirect(303, '/settings/security')`.

Svelte needs `'unsafe-inline'` for its style hydration; revisit once CSP nonces are wired. Do **not** add `'unsafe-inline'` to `script-src`. Run the app and watch DevTools Console for violations; iterate.

## Verification

Run on the workstation, not the prod host, against a local dev server (`pnpm dev`) or the staging build:

- `pnpm check` — must be clean.
- Log in with bootstrap password → redirected to `/settings/security`. Change it → redirected to `/`. Log out, log in again with new password.
- `sqlite3 data/maw.sqlite 'select event,ip,username from auth_events order by ts desc limit 10'` — sees the events.
- `tail -n 5 data/auth.log` — lines match the fail2ban regex: `echo '<line>' | fail2ban-regex - deploy/fail2ban/filter.d/maw-auth.conf` reports matches.
- `for i in $(seq 1 12); do curl -s -o /dev/null -w '%{http_code}\n' -d 'username=bad&password=bad' http://localhost:5173/login; done` — after ~10 requests returns 429.
- Cross-origin CSRF: from `http://localhost:8000` scratch page, `fetch('http://localhost:5173/api/projects', {method:'POST', credentials:'include', ...})` → 403.
- WS: `websocat 'ws://localhost:5173/ws' -H 'Origin: https://evil.example' -H 'Cookie: maw_session=...'` → closes with 4403.
- DevTools: no CSP violations on normal app use.
- `curl -I https://maw.staging.remesch.at/` shows HSTS, CSP, X-Content-Type-Options, Referrer-Policy, Permissions-Policy.
