# Plan: JSONL transcript as the source of truth for reconnect history

## Context

Claude Code (and Codex/Gemini, with their own formats) repaint their TUI
into tmux's main buffer on every streaming tick. Tmux scrollback ends up
full of byte-unequal ghost renders, which is why
`cli-adapters/claude-code.jsonc` falls back to `scrollbackMode: "visible"`
(`-S 0`) â€” capture only what is on screen *right now*. That works for
"resume the live view" but throws away all conversation history that has
scrolled out of the visible pane.

The user's earlier proposal (anchor on a stored startup banner, trim
scrollback before its last occurrence) was analyzed and rejected â€” see
prior revision of this file. Summary: the banner is itself repainted on
every tick, mutates byte-by-byte, and the region after it still contains
ghost renders. It strictly regresses from `-S 0`.

This plan proposes the alternative hinted at in that analysis: stop
trying to reconstruct history from the terminal stream, and read it from
the CLI's own session log instead.

## Why this works

Claude Code persists every conversation turn as JSONL at:

```
~/.claude/projects/<encoded-cwd>/<sessionId>.jsonl
```

Encoding maps `/` â†’ `-` (verified locally). One line per event. Relevant
event types (observed in a real session):

| `type`                  | Meaning                                       |
|-------------------------|-----------------------------------------------|
| `user`                  | User prompt (`message.content` is the text)   |
| `message` (assistant)   | Assistant response (text + tool_use blocks)   |
| `permission-mode`       | Mode change (`plan`, `default`, â€¦)            |
| `file-history-snapshot` | File state checkpoint                         |
| `system`, `skill_listing`, `deferred_tools_delta` | Bookkeeping        |

This is a **lossless, append-only** record â€” exactly what the terminal
stream is not. It survives repaints, expand/collapse widgets, scrolling,
and reattach. It is what `claude --resume` itself reads.

## Design

### 1. Adapter capability: `historySource`

Add an optional adapter descriptor field alongside `scrollbackMode`:

```jsonc
// cli-adapters/claude-code.jsonc
"historySource": {
  "kind": "claude-jsonl",
  "path": "~/.claude/projects/{{cwdEncoded}}/{{sessionId}}.jsonl"
}
```

`kind` is a discriminator so Codex/Gemini can plug their own readers
later (`codex-jsonl`, `gemini-history`, etc.) without touching the WS
protocol. Adapters without `historySource` keep current behavior.

### 2. Capturing `sessionId`

`claude` accepts `--session-id <uuid>` (confirmed in installed version's
`--help`). `AgentRuntime` generates a UUID at spawn time, passes it via
adapter spawn args, and persists it on the agent row as `cli_session_id`.
The JSONL path is then deterministic:
`~/.claude/projects/<encoded-cwd>/<cli_session_id>.jsonl` â€” no watch, no
sniff, no race.

Plumb the UUID via a new template variable (e.g. `{{agent.cliSessionId}}`)
so the adapter JSON stays declarative:

```jsonc
"spawn": {
  "command": "claude",
  "args": ["--session-id", "{{agent.cliSessionId}}", "--dangerously-skip-permissions"],
  ...
}
```

`cwdEncoded` is derived from the worktree path with the `/` â†’ `-` rule.

### 3. Server-side renderer

New module `src/lib/server/agents/history/ClaudeJsonlHistory.ts`:

- `loadSince(filePath, sinceLineOffset)` â†’ returns parsed events + new
  offset (so subsequent reads are incremental).
- `renderToAnsi(events, opts)` â†’ produces a deterministic ANSI/text
  rendering: one block per `user` and assistant `message`, with simple
  separators. Tool-use blocks render as collapsed summaries
  (`â–¸ Bash: pnpm test (exit 0)`) â€” full payloads on demand later.

Keep the renderer dumb: no markdown, no syntax highlighting in v1.
xterm.js will display the ANSI text as-is.

### 4. WS protocol addition

Bump protocol to v3 with one new message:

- `history_snapshot` (server â†’ client): `{ format: 'text', body: string,
  truncated: boolean }`. Sent on subscribe **before** the existing
  `terminal_snapshot` (which still carries the live `-S 0` capture).

Client renders `history_snapshot` into the xterm scrollback first, then
the live capture. Result: scrolling up shows real conversation history;
the bottom of the screen is the live TUI, unchanged.

## Risks / open questions

- **JSONL schema drift**: Anthropic can change the format. Mitigate with
  defensive parsing (skip unknown `type`s) and a smoke test fixture
  checked into `tests/fixtures/claude-jsonl/`.
- **Multiple sessions per agent**: if the user runs `/clear` or starts a
  new session inside the same tmux pane, a new JSONL file is created.
  Re-run discovery on `permission-mode` reset or expose a "rescan"
  control. Defer to v0.3 unless it bites in testing.
- **Codex/Gemini parity**: out of scope for this plan; the
  `historySource.kind` discriminator leaves room for them.
- **Privacy**: JSONL contains full conversation text. It already lives
  on the same disk under the same user, so no new exposure â€” but worth
  noting the WS payload now carries it explicitly.

## Out of scope

- Codex / Gemini history readers (separate plans).
- Rich rendering (markdown, syntax highlighting, collapsible tool
  blocks) â€” text first, polish later.
- Editing or replaying history back into the running session.
