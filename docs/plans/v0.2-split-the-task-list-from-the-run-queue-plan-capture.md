# Split the Task List from the Run Queue (+ Plan capture)

## Context

Today every task the user creates is automatically eligible for the scheduler: the moment a task lands in [`queue_entries`](migrations/010_queue_entries.sql), the next [scheduler tick](src/lib/server/queue/Scheduler.ts:305) classifies it and promotes it to a running agent as soon as a slot opens. There is no way to say "I want this on my list, but I'm not done thinking about it — don't run it yet."

The user wants two distinct concepts living side by side:

- **Task list (Backlog)** — inventory. Items the user is still working on (writing the prompt, sketching a plan, etc.). The scheduler ignores these.
- **Queue** — the subset the user has explicitly green-lit. The scheduler picks these up when capacity opens.

The user explicitly drives the transition: every new task lands in Backlog by default; the user clicks **Run** to send it to the Queue.

A natural part of this workflow is reusing a plan an agent has already produced. We add a **Copy markdown** button to the plan viewer so the user can grab a plan with one click, and give tasks a real `plan_md` field so the plan travels with the task (and reaches the next agent as context when it runs).

## Recommended Approach

Two coupled changes:

1. **Queue admission boolean.** Add `queued INTEGER NOT NULL DEFAULT 1` to `queue_entries`. The scheduler only looks at rows where `queued = 1`. Backlog rows (`queued = 0`) sit in `pending` / `blocked` / `ready` but the scheduler skips them entirely until the user flips the bit. A boolean keeps the *intent* axis (auto-run?) orthogonal to the *lifecycle* axis (`pending → ready → running → done`).

2. **Plan-on-task.** Add `plan_md TEXT NULL` and `plan_source_path TEXT NULL` to `queue_entries`. `plan_md` is the immutable snapshot stored with the task. `plan_source_path` is an optional reference (e.g., `docs/plans/v0.3-foo.md`) for traceability — schema-only for now, no UI to set it manually; future automation can populate it. When the agent spawns, the prompt becomes `body + "\n\n## Plan\n\n" + plan_md` (only when `plan_md` is non-empty).

The spawn dialog gets **two submit buttons** instead of one: `Save` (to Backlog) and `Run` (to Queue). The plan viewer gets a **Copy markdown** button in its header.

## Data Model

### New migration: [`migrations/011_queue_entries_queued_and_plan.sql`](migrations/011_queue_entries_queued_and_plan.sql)

```sql
ALTER TABLE queue_entries
  ADD COLUMN queued INTEGER NOT NULL DEFAULT 1 CHECK (queued IN (0,1));
ALTER TABLE queue_entries
  ADD COLUMN plan_md TEXT;
ALTER TABLE queue_entries
  ADD COLUMN plan_source_path TEXT;

CREATE INDEX idx_queue_entries_user_queued_status
  ON queue_entries(user_id, queued, status);
```

Existing rows get `queued = 1` (preserves today's behavior) and `NULL` for the plan columns.

## Scheduler

Single gate, single code change in [`Scheduler.ts:308`](src/lib/server/queue/Scheduler.ts:308):

```ts
const entries = listSchedulableQueueEntries(userId, { onlyQueued: true });
```

Widen [`listSchedulableQueueEntries`](src/lib/server/db/queries.ts:1377) to accept `{ onlyQueued?: boolean }`. Default `false` so [`reconcileOrphanedRunning`](src/lib/server/queue/Scheduler.ts:227) keeps seeing every non-terminal row.

### Dependency UX touch (in scope, ~3 lines)

In [`classifyOpenEntry`](src/lib/server/queue/Scheduler.ts:371) at line 401, when a dependency exists but has `queued === 0`, return `last_error: 'dependency in backlog: <id>'` instead of `null`. Without this, a queued task depending on a Backlog item sits silently in `blocked`.

## Spawn pipeline

In [`spawnFromInputs.ts`](src/lib/server/agents/spawnFromInputs.ts) where the initial prompt is composed for the adapter (the function that combines title/body before invoking `supervisor.spawn`), append the plan:

```ts
const initialPrompt = plan_md
  ? `${body ?? ''}\n\n## Plan\n\n${plan_md}`.trimStart()
  : (body ?? '');
```

Pass `initialPrompt` to the adapter exactly where `body` flows today. No adapter changes required — they all already accept a single text blob.

## API

### Two new endpoints for queue admission (mirror [`promote/+server.ts`](src/routes/api/queue/[id]/promote/+server.ts))

- `POST /api/queue/[id]/queue` — sets `queued = 1`. Valid on `pending`/`blocked`/`ready`. 409 on `running` or terminal.
- `POST /api/queue/[id]/backlog` — sets `queued = 0`. Same status guard.

Both call new `QueueScheduler` methods (`queueEntry` / `backlogEntry`) modeled on [`cancelEntry`](src/lib/server/queue/Scheduler.ts:161): single-column update via a new `setQueueEntryQueued` query helper, emit `change`, call `scheduleTick()`.

### Plan content for tasks (used by the Tasks page plan badge)

- `GET /api/queue/[id]/plan` — returns `{ markdown: string, html: string, source_path: string | null }` for the task's `plan_md`. HTML is sanitized server-side using the same path the agents plan viewer uses ([`src/lib/server/plans/agentPlans.ts:renderAgentPlan`](src/lib/server/plans/agentPlans.ts)). 404 when `plan_md` is null.

### Extend the agent plan endpoint for clipboard

[`/api/agents/[id]/plan?file=...&source=...`](src/routes/api/agents/[id]/plan/+server.ts) currently returns `{ name, html }`. Add `markdown: string` to the response body (the raw file contents pre-sanitization). The plan viewer uses it for the Copy button without re-fetching.

### Existing endpoints

- `POST /api/queue` — payload gains `queued: boolean`, `plan_md: string | null`, `plan_source_path: string | null`. The handler passes them through to `insertQueueEntry`.
- `POST /api/queue/[id]/promote` ("Run now") — no semantic change. Continues to work on Backlog entries: bypasses caps, does **not** flip `queued` (preserves the user's intent).
- `PUT /api/queue/[id]` — extend the existing edit path to accept `plan_md` and `plan_source_path`. (No `queued` here; users use the dedicated endpoints.) This is the route the user uses if they want to edit the plan later.

## UI

### [`PlanViewerModal.svelte`](src/lib/client/components/PlanViewerModal.svelte) — Copy markdown button

Add a small button in the modal header (alongside the existing file switcher when present, otherwise standalone in `headerRight`):

```
[Copy markdown]
```

Stores the latest `markdown` returned by the API (the new field). On click: `await navigator.clipboard.writeText(markdown)`, then briefly swap the label to `Copied ✓` for ~1.5s (local `$state` flag). Disabled when `view.kind !== 'viewing'`. No new permissions — `navigator.clipboard.writeText` works in secure contexts (HTTPS / localhost).

In the file-switcher case, the button copies the *currently selected* plan, not the original auto-picked one.

### [`+page.svelte`](src/routes/queue/+page.svelte) — Backlog section + new actions + plan badge

Update `groupEntries` (line 31) to produce five buckets:

```ts
{ running, ready, blocked, backlog, completed }
```

- `running` / `ready` / `blocked` keep their current filters **AND** require `e.queued === 1`.
- `backlog` is `e.queued === 0 && !TERMINAL.has(e.status)`.
- `completed` is unchanged (terminal status; queued or backlog rows land here when cancelled/failed/done).

Render the new `<section>` between Blocked and Completed (mirror lines 240-278). Sort inside Backlog by `priority DESC, created_at ASC` for consistency with the scheduler.

**Per-row actions:**

| Section | Buttons |
|---|---|
| Running | Open Agent, Cancel |
| Ready | **Run now**, **Send to backlog**, Cancel |
| Blocked | **Run now**, **Send to backlog**, Cancel |
| **Backlog** | **Queue**, **Run now**, Cancel |
| Completed | Open Agent (if any) |

`Send to backlog` → `POST /api/queue/:id/backlog`. `Queue` → `POST /api/queue/:id/queue`. Both call `invalidateAll()` on success.

**Plan badge:**

For any entry where `plan_md` is non-null, render a small chip after the title: `📄 Plan` (use the document icon already in `AgentMenu.svelte:96-101` for visual consistency). Clicking the chip opens a generalized plan viewer modal seeded from `GET /api/queue/[id]/plan`.

To avoid duplicating PlanViewerModal logic, generalize it minimally: add a `source` prop with two shapes:

```ts
source: { kind: 'agent'; agentId: string } | { kind: 'task'; taskId: string }
```

The fetch URLs branch on `source.kind`. The file switcher only renders for `kind === 'agent'` (tasks have exactly one plan). Existing callers continue to pass the agent shape unchanged.

### Sidebar badge

Keep [`countOpenQueueEntries`](src/lib/server/db/queries.ts:1407) counting Backlog rows too. The badge reflects "items on my plate," which is the user's whole point. No split.

## Out of Scope

- **AgentMenu "Create Task" item.** Dropped — the Copy markdown button + manual paste flow is the user's preferred path.
- **Auto-extraction of `plan_source_path` from the plan viewer.** Column exists in schema, no UI to set it for v0.3. Add later if there's a clear use case (e.g., one-click "create task from this plan" that fills both).
- **"Agent makes a plan and returns to the list" automation.** Out — replaced by the Copy-and-paste flow. Agents still mark their task `done` on normal exit.
- **Edit-mode for existing tasks in the UI.** The PUT endpoint accepts plan fields but no row-level edit button is added in this PR.
- **User-setting to flip the default submit action** (Save vs Run). Defer.
- **Splitting the sidebar badge** into queued/backlog counts. Single number stays.
