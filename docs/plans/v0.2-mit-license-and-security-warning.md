# Plan: MIT License + Security Warning in README

## Context

Two things missing from the repo's public-facing docs:

1. **No license.** Neither a `LICENSE` file at the root nor a `"license"` field in `package.json`. Without an explicit license, default copyright applies — nobody can legally fork, reuse, or contribute. User wants MIT.
2. **No security warning.** MAW exposes shell access (literally — see `cli-adapters/shell.jsonc`) and lets any authenticated user spawn CLIs that inherit the host user's filesystem and network privileges. That's powerful and dangerous. The README talks about fail2ban and TLS but never spells out the top-level rule: *don't run this as root, don't expose it naked on the internet*. User supplied draft text (partly in German — "ohne Einschränkungen" = "without restrictions") and asked me to improve it and propose further additions.

Outcome: a README that can ship to strangers — clear legal terms, clear warnings, no surprises.

### 3. README.md — new **Security** callout near the top

Placement: **directly after the "Two driving goals" list (line 15)** and before `## PWA & push notifications`. It is the first thing a reader sees after understanding what MAW is, which is the right moment for a warning. Rendered as a blockquote with a bold lede so GitHub renders it visually distinct.

Proposed replacement/improvement of the user's draft:

```markdown
## Security — read this before deploying

> **⚠️ MAW exposes a shell to every authenticated session.** Anyone who
> logs in can spawn agents that run arbitrary commands on the host — the
> `shell` adapter is literally a bare `bash`, and Claude Code / Codex /
> Gemini agents can read and write any file the host user can, make
> outbound network requests, and invoke any installed CLI. Treat an
> authenticated session as equivalent to SSH access to the MAW user
> account.

The non-negotiables:

- **Never run MAW as `root` or under any account with `sudo` rights.**
  Create a dedicated unprivileged system user (`maw` is the convention
  used throughout these docs). Everything MAW spawns inherits that
  user's permissions — that isolation *is* the sandbox.
- **Use a strong password** for every MAW account, and rotate the
  `MAW_BOOTSTRAP_PASSWORD` seeded on first boot.
- **Enable [fail2ban](#fail2ban-prod)** before exposing MAW to any
  untrusted network. The shipped jail bans repeat `login_fail`,
  `pwchange_fail`, `rate_limited`, and `ws_origin_reject` offenders.
- **Always terminate TLS in front of MAW.** Session cookies, WebSocket
  traffic, and Web Push subscriptions must not travel over plain HTTP.
  HTTPS is also required for PWA install and Web Push to function at
  all (except on `localhost` for dev).
- **Set `MAW_PUBLIC_ORIGIN` in production** so the WebSocket upgrade
  rejects cross-origin handshakes. Without it MAW falls back to dev
  behavior and accepts any `Origin`.
- **Set `MAW_TRUST_PROXY=1` when behind a reverse proxy** so the auth
  log records the real client IP, not the proxy's. The fail2ban jail
  is useless without this.
- **Rotate `MAW_SESSION_SECRET` if you suspect compromise** — it signs
  every session cookie, and changing it invalidates all active logins.
- **Only grant MAW accounts to people you would give SSH to.** There
  is no per-agent sandbox, no per-user filesystem isolation, no rate
  limit on command execution. Permission boundaries end at the host
  user.

For extra paranoia: disable `cli-adapters/shell.jsonc` in production
(move or rename the file; the registry hot-reloads) so users can only
spawn adapter-vetted CLIs rather than a free-form bash.
```

Why the improvements:

- Drops the German word "Einschränkungen" for English "restrictions"
  (rephrased as "that isolation *is* the sandbox" which reads better).
- Replaces the vague "logged-in user" with "every authenticated
  session" and explains the blast radius concretely (filesystem +
  network + any installed CLI).
- Turns the three-sentence paragraph into a skimmable bullet list —
  each bullet a single rule. Readers who skim docs (everyone) will at
  least see the section headings.
- Links the existing `#fail2ban-prod` and `#environment` anchors so
  the warning points at the config the reader needs.
- Adds the six additions the original draft omitted (TLS, public
  origin, trust-proxy, session-secret rotation, "SSH-equivalent
  access" framing, shell-adapter disable tip) because each corresponds
  to a real trust boundary the app already exposes — not hypothetical
  future hardening.
