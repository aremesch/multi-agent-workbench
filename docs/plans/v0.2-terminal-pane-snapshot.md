# Terminal reconnect via tmux pane snapshot

## Context

The v0.2 clean-slate revert (`docs/plans/v0.2-terminal-revert.md`) put terminal
reconnect back on raw byte-log replay from `terminal_log`. That carries a known
stacked-banner bug on first connect to any TUI CLI without an alt-screen
buffer (claude-code, codex, gemini): the replay bursts 3–4 accumulated redraw
cycles into a freshly `reset()`-ed xterm grid and each one paints its banner
one screen lower than the last.

The user noticed that **directly after spawn, the same claude-code renders
cleanly** — because the byte log is short and the TUI just finished drawing,
so the final state of the tmux pane and what xterm renders match. The moment
the modal closes, `AgentTerminalPanel.onDestroy` calls
`getMawWsClient().unsubscribe(agentId)` → `subs.delete(agentId)`
([ws.ts:127](src/lib/client/ws.ts:127)). On reopen a fresh `Sub` is created
with `maxSeenSeq=0` so the client sends no `lastSeq` and the server replays
from seq=0 — the same stacking path as a page reload.

Solution: on every subscribe, ship the **tmux pane's current rendered grid**
(what a live viewer would see *right now*) instead of the raw byte history.
`tmux capture-pane -p -e -S 0` already does exactly that — the existing
`TmuxSession.capturePane` helper [tmux/TmuxSession.ts:207](src/lib/server/tmux/TmuxSession.ts:207)
documents `-S 0` as "the right mode for a reconnect snapshot." The bytes that
stream in after the snapshot paint on top via the existing `seq` dedup path.

## Design

**Protocol bump v4 → v5.** Replace the server's `scrollback` burst on
`subscribe_agent` with a single `pane_snapshot` message carrying the
visible-pane ANSI + a seq watermark. No other message types change.

### Server flow ([hub.ts:109–135](src/lib/server/ws/hub.ts:109))

```
handleSubscribe(agentId, _lastSeq_ignored):
  snapshotSeq = peekLatestTerminalSeq(agentId)          // SELECT MAX(seq)
  register onOutput listener (unchanged)                // live bytes → send
  ansi = await Tmux.capturePane(runtime.tmux_session, 0)
  send({ type: 'pane_snapshot', agentId, ansi, seq: snapshotSeq })
  catchUp = listTerminalChunksSince(agentId, snapshotSeq)
  for chunk in catchUp:
    send({ type: 'output', agentId, seq: chunk.seq, b64: chunk.b64 })
```

`lastSeq` from the client is now ignored — the snapshot is the authoritative
reset. The listener is registered *before* `capturePane` resolves so bytes
that land during the await still reach the client (client dedupes via the
existing `seq <= maxSeenSeq` drop at [ws.ts:205](src/lib/client/ws.ts:205)).
The explicit catch-up on lines 6–8 covers the (rare) window between reading
`snapshotSeq` and the listener going live.

### Client flow

`MawWsClient.dispatch` [ws.ts:196](src/lib/client/ws.ts:196) swaps the
`scrollback` case for `pane_snapshot`:

```
case 'pane_snapshot': {
  const sub = this.subs.get(msg.agentId); if (!sub) break;
  if (msg.seq > sub.maxSeenSeq) sub.maxSeenSeq = msg.seq;
  sub.handlers.onPaneSnapshot(msg);
  break;
}
```

`AgentTerminalPanel.svelte` handler becomes:

```
onPaneSnapshot: ({ ansi }) => {
  term.reset();
  term.write(ansi);
}
```

The live `output` path stays identical — `seq > maxSeenSeq` is still the
guard, so any pre-snapshot output bytes that raced the listener registration
get dropped harmlessly.

## Out of scope / known trade-offs

- **Visible pane only.** `-S 0` loses everything that scrolled off. Full
  scrollback (`-S -`) is not a drop-in fix because tmux's history for a TUI
  pane is stuffed with intermediate redraws — it would reintroduce the
  stacking. A follow-up can add per-adapter `captureScrollback: 'visible' | 'full'`
  once a non-TUI adapter actually needs it; default stays `visible`.
- **Alt-screen apps** (vim, btop): `capture-pane` captures whichever buffer
  is currently active, which matches what a live viewer sees — no regression.
- **Dead agents** are served by `/api/agents/[id]/log`; the hub snapshot path
  only runs for `running` agents where a tmux session exists. `capturePane`
  returns `''` on missing session ([TmuxSession.ts:219](src/lib/server/tmux/TmuxSession.ts:219))
  as a defensive no-op.
