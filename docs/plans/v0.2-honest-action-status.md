# Plan: surface fail() status on the wire for non-browser clients

## Context

You ran from `sweub1`:

```sh
for i in {1..6}; do
  curl -s -o /dev/null -w "Attempt $i: %{http_code}\n" \
    -X POST 'https://maw.remesch.at/login?/login' \
    -H 'Origin: https://maw.remesch.at' \
    --data-urlencode 'email=nobody@example.com' \
    --data-urlencode 'password=wrong'
done
# → all six attempts: 200
```

The nginx in front of MAW is a clean pass-through (no
`proxy_intercept_errors`, no `error_page`):

```nginx
location / {
    proxy_pass http://192.168.178.15:3000;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    # WebSocket
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
}
```

So nginx isn't the rewriter. The 200 really comes from MAW.

I traced the SvelteKit source and found the cause: it's the documented
behaviour of form actions for clients that don't put `text/html` first
in their `Accept` header (curl, bots, fetch with defaults). The login
action returns `fail(401, …)` correctly, but for those clients SvelteKit
serializes the result as JSON with an HTTP status of **200**. Real
browsers send `Accept: text/html,…,*/*;q=0.8`, take the HTML render path,
and see 401 on the wire — which is why this never surfaced before.

This plan adds a small response-rewriter hook so the wire status matches
the action's `fail()` status for the JSON path too. fail2ban (which
parses MAW's own `auth.log`) already works regardless of this — see
"Why bother" below.

## Root cause (verified against the installed @sveltejs/kit ^2.20.0)

### How content negotiation chooses the path

[`node_modules/@sveltejs/kit/.../page/actions.js:14`](../../node_modules/@sveltejs/kit/src/runtime/server/page/actions.js):

```js
export function is_action_json_request(event) {
    const accept = negotiate(event.request.headers.get('accept') ?? '*/*', [
        'application/json',
        'text/html'
    ]);
    return accept === 'application/json' && event.request.method === 'POST';
}
```

`negotiate()` returns the first match in the priority list when q-values
tie ([`utils/http.js:23-37`](../../node_modules/@sveltejs/kit/src/utils/http.js)).
With curl's default `Accept: */*` both `application/json` and `text/html`
match at q=1.0, but `application/json` is listed first → JSON path wins.

### What the JSON path does with `fail()`

[`actions.js:60-89`](../../node_modules/@sveltejs/kit/src/runtime/server/page/actions.js):

```js
if (data instanceof ActionFailure) {
    return action_json({
        type: 'failure',
        status: data.status,           // ← in the body
        data: stringify_action_response(...)
    });                                  // ← no `init` passed
}
```

[`actions.js:137-139`](../../node_modules/@sveltejs/kit/src/runtime/server/page/actions.js):

```js
function action_json(data, init) {
    return json(data, init);
}
```

`json(data)` with no init defaults the HTTP status to 200. The fail
status survives only inside the body — `use:enhance` reads it from
there. For a non-`use:enhance` client, the wire is 200.

### What the HTML path does with `fail()`

[`page/index.js:84-86`](../../node_modules/@sveltejs/kit/src/runtime/server/page/index.js):

```js
if (action_result?.type === 'failure') {
    status = action_result.status;     // → render_response gets 401
}
```

Browsers (Accept prioritises `text/html`) take this path → wire 401. ✓

### Quick proof you can run on `sweub1` to confirm

```sh
# JSON path (what you ran) → 200
curl -s -o /dev/null -w "%{http_code}\n" \
  -X POST 'https://maw.remesch.at/login?/login' \
  -H 'Origin: https://maw.remesch.at' \
  --data-urlencode 'username=nobody' --data-urlencode 'password=wrong'
# → 200

# HTML path → 401
curl -s -o /dev/null -w "%{http_code}\n" \
  -X POST 'https://maw.remesch.at/login?/login' \
  -H 'Origin: https://maw.remesch.at' \
  -H 'Accept: text/html' \
  --data-urlencode 'username=nobody' --data-urlencode 'password=wrong'
# → 401
```

(Side note on your repro: it sent `email=…` but the action reads
`username` ([`+page.server.ts:28`](../../src/routes/login/+page.server.ts)),
so it actually hits `fail(400)` at line 41, not `fail(401)`. Doesn't
change the diagnosis — both 400 and 401 are pinned to wire-200 by the
JSON path. The repro above uses `username=` to land on `fail(401)`.)

## Why bother — and a sanity check before doing the change

The change below only matters if you care about non-browser clients
(curl, bots, fetch with default Accept) seeing the real status code.

- **fail2ban**: already works without this fix.
  [`deploy/fail2ban/filter.d/maw-auth.conf`](../../deploy/fail2ban/filter.d/maw-auth.conf)
  matches structured lines in MAW's `auth.log`, written by
  `logAuth('login_fail', …)` at
  [`+page.server.ts:45`](../../src/routes/login/+page.server.ts) and
  `:50-55`. The wire HTTP status is not part of the regex.
  Verify by tailing the file while you re-run your curl loop:

  ```sh
  tail -f /var/lib/maw/auth.log    # adjust to MAW_DATA_DIR
  ```

  You should see six `login_fail` lines.

- **HTTP-status-based monitoring / WAF rules / nginx access-log
  alerting**: those will currently see 200 for every bot-driven
  password-spray attempt, masking the attacks. This is the actual win
  from the fix below.

If only fail2ban matters to you, **no code change is needed** and we
should stop here. The rest of the plan assumes you want the wire status
to be honest for all clients.

## Proposed change

Single small `handle` hook addition in
[`src/hooks.server.ts`](../../src/hooks.server.ts), composed with the
existing one. After `resolve(event)`, when the response is a SvelteKit
form-action JSON failure (`Content-Type: application/json`, body
`{ type: 'failure', status: 4xx, … }`), copy `body.status` onto the wire
and forward the body unchanged.

### Sketch

```ts
const honestActionStatus: Handle = async ({ event, resolve }) => {
  const response = await resolve(event);

  // SvelteKit form actions pin the wire status at 200 when the client
  // takes the JSON-action path (Accept negotiates to application/json).
  // Re-emit the failure status from the body so non-browser clients
  // (curl, bots, monitoring probes) see real 4xx codes.
  if (
    event.request.method !== 'POST' ||
    response.status !== 200 ||
    !response.headers.get('content-type')?.startsWith('application/json')
  ) {
    return response;
  }

  const text = await response.clone().text();
  let parsed: { type?: string; status?: number } | null = null;
  try { parsed = JSON.parse(text); } catch { return response; }

  if (
    parsed?.type === 'failure' &&
    typeof parsed.status === 'number' &&
    parsed.status >= 400 && parsed.status < 600
  ) {
    return new Response(text, {
      status: parsed.status,
      headers: response.headers
    });
  }

  return response;
};

// Compose with the existing handle via `sequence`
import { sequence } from '@sveltejs/kit/hooks';
export const handle = sequence(existingHandle, honestActionStatus);
```

Notes on the implementation:

- Cost is one extra `text()` parse on JSON action POSTs only —
  negligible vs. argon2.
- Does not affect successful actions (those are `type: 'success'` with
  `status: 200|204`) or redirects (a redirect short-circuits before
  ever hitting this code path; `redirect_response` is sent directly
  from `render_page`).
- Only kicks in when the SvelteKit handler itself returned
  `Content-Type: application/json` AND body type=`failure`. JSON
  responses from `+server.ts` API routes that return `{ status: 4xx }`
  by accident in their body are not touched (they wouldn't have
  `type: 'failure'`).

### Where exactly the new hook lives

Edit [`src/hooks.server.ts`](../../src/hooks.server.ts):

1. Add `import { sequence } from '@sveltejs/kit/hooks';`
2. Rename the existing `export const handle: Handle = …` to

  follow-up.
