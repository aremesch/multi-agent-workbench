# Context

Goal: Let the user spawn a "Browser" agent that displays an iframe preview of a localhost
dev server (e.g., `localhost:5173`) inside MAW's dashboard, primarily for **visual mobile
testing of responsive design** via the existing PWA. The agent is a formal first-class
agent type — listed alongside CLI agents, persisted in the DB, archivable — but skips
tmux/FIFO entirely since there is no CLI process to drive.

# Why the original plan needed revision

After validating the first draft against the actual codebase, three blockers and several
smaller gaps surfaced. They are addressed in this revision.

### Blocker 1 — `localhost` is unreachable from a phone

The phone's browser resolves `localhost` to **its own** loopback, not the MAW server's.
A literal `<iframe src="http://localhost:5173">` therefore loads nothing on mobile.
Even on desktop, mixed content (HTTPS MAW + HTTP dev server) is blocked.

**Fix:** add a same-origin reverse-proxy route (`/preview/<agentId>/...`) on MAW's HTTP
listener that forwards to the per-agent target URL — including the WebSocket upgrade
needed for Vite HMR. The iframe then loads `/preview/<agentId>/`, which is same-origin
with MAW (no mixed content, no CORS, mobile reachable).

### Blocker 2 — `repos.origin_url` is a git URL, not a preview URL

The first plan suggested using `repos.origin_url` as the iframe src. That field stores
git remotes (`git@github.com:foo/bar.git`), not HTTP endpoints. Reusing it is wrong.

**Fix:** add a per-agent `target_url` (and `target_port` for proxy use), captured at
spawn from a new form field. No `repos` migration required for Phase 1; per-repo
defaults can be added later if multiple agents in the same repo need DRY config.

### Blocker 3 — `AgentSupervisor` assumes every agent has a tmux session

`spawn()`, `reap()`, `kill()`, `startExitWatcher()`, `finishAsExited()`, and the
reattach-on-boot logic all touch `tmux_session` directly. A NULL session would crash
or get marked as exited on first reap.

**Fix:** introduce `isBrowserAgent(cliKind)` and explicitly gate every tmux/FIFO
touchpoint. Browser agents stay `running` until the user archives them.

### Smaller gaps closed

| Gap | Fix |
|---|---|
| Spawn form requires task body + role | Add a "Browser" role (cli_kind=`browser`) with empty system prompt; make task body optional when the selected role's kind is `browser`. Title is reused as the session label. |
| Worktree creation is wasteful | Set `createWorktree: false` on the browser adapter — no branch, no worktree dir. |
| Dashboard modal always renders `<Terminal>` | Branch on `cli_kind` inside `AgentTerminalPanel` — render `<BrowserView>` for browser agents. |
| CSP blocks cross-origin iframes | Same-origin proxy makes this a non-issue. No CSP edits needed. |
| Proxy auth | The `/preview/<agentId>/...` route requires the same session cookie as the rest of MAW, and verifies `agent.user_id === session.userId`. |

# Architecture (Phase 1)

```
┌─────────────────────────────────────────────────────────────────┐
│  Phone PWA  ── HTTPS ──>  MAW (server.js)                       │
│                              │                                  │
│   iframe src=                │  /preview/<agentId>/*  ──proxy──> http://localhost:<port>/*
│   "/preview/<id>/"           │  WS upgrade /preview/<id>/* ────> ws://localhost:<port>/* (Vite HMR)
│                              │                                  │
│                              │  Everything else → SvelteKit handler
└─────────────────────────────────────────────────────────────────┘
```

The browser agent has no tmux session, no FIFO, no pattern matcher. It is purely a
DB row + an iframe view backed by the proxy.

# Implementation

### 1. Browser adapter — `cli-adapters/browser.jsonc`

```jsonc
{
  "kind": "browser",
  "displayName": "Browser",
  "createWorktree": false,
  "spawn": {
    "command": "/bin/true",       // Never executed; supervisor short-circuits.
    "args": [],
    "env": {},
    "initialInput": ""
  },
  "input": { "encoding": "literal", "submitKey": "Enter", "promptAnswers": {} },
  "patterns": [],
  "idleDetection": { "method": "inactivity", "inactivityMs": 999999 },
  "mobileQuickKeys": []
}
```

The adapter is intentionally inert — its config exists so the registry treats `browser`
as a known kind, but the supervisor's branch never invokes spawn/ingest paths.

### 2. Supervisor branching — `src/lib/server/agents/AgentSupervisor.ts`

Add a top-level helper:

```typescript
const BROWSER_KIND = 'browser';
function isBrowserKind(kind: string) { return kind === BROWSER_KIND; }
```

Branch in:

- **`spawn()`** — if `role.cli_kind === 'browser'`: insert agent row with `tmux_session = NULL`, `status = 'running'`, persist `target_url`/`target_port`, skip `Tmux.newSession`, `AgentRuntime.start`, `startExitWatcher`. Return immediately.
- **`reap()`** — filter browser agents out of the tmux-comparison loop. They never transition to `exited` automatically.
- **`kill()`** — for browser agents, just call `finishAsExited(id, 'killed')` and skip `Tmux.killSession`.
- **`finishAsExited()`** — for browser agents, skip `runtime.stop()` and `snapshotAgentCommits` (no commits attributable). Just update DB row + emit `state` event.
- **Reattach-on-boot** — exclude browser agents from tmux session-existence checks. They survive restarts unconditionally.

### 3. Per-agent target capture — schema + spawn form

**Migration `007_agent_browser_target.sql`** (additive, nullable for non-browser agents):

```sql
ALTER TABLE agents ADD COLUMN target_url TEXT NULL;
ALTER TABLE agents ADD COLUMN target_port INTEGER NULL;
```

`target_url` stores the full URL (e.g., `http://localhost:5173`) for transparency in the UI.
`target_port` is the canonical port the proxy forwards to (parsed from `target_url`).
Both NULL for non-browser agents.

**Spawn form (`src/routes/agents/new/+page.server.ts` + form component):**

When the selected role's kind is `browser`:
- Hide the "Task body" textarea.
- Show a new "Preview URL" field (default `http://localhost:5173`, validated as `http://localhost:<1-65535>` or `http://127.0.0.1:<port>`).
- Title remains required (it's the session label).
- `repo_id` remains required (used for archive grouping and proxy authorization scope).

**Built-in Browser role:** seeded once at bootstrap (or via migration) — `cli_kind = 'browser'`, empty system prompt, empty tool config. Always available in the role dropdown so users don't have to create it manually.

### 4. Reverse proxy — `server.js`

Use Node's built-in `http.request` (no new dependency) to forward HTTP and `upgrade`
requests under `/preview/<agentId>/*` to the agent's `target_port` on `127.0.0.1`.

**Required behavior:**

- **Auth:** Parse `SESSION_COOKIE` from the request, look up the session, and verify the agent row belongs to that user. Reject `401` otherwise. Reuse the same helpers `WsHub.attach` already uses.
- **HTTP forwarding:** Strip `/preview/<agentId>` from the path, copy headers (drop `host`), pipe request body, pipe response back. Set a sensible timeout (10s).
- **WebSocket forwarding:** On `upgrade` events with `/preview/<agentId>/*` paths, open a TCP socket to `127.0.0.1:<target_port>`, forward the HTTP upgrade handshake bytes verbatim, and pipe the resulting socket bidirectionally. This is what makes Vite HMR work.
- **Same-origin guarantees:** Because the iframe URL is same-origin with MAW, no CSP/CORS gymnastics are needed. The browser treats it as a normal subresource.
- **Resolve target_port at request time:** look up the agent row in the DB on each request rather than caching — this is sub-millisecond on SQLite and keeps things stateless.

This logic goes between the existing `/ws` guard and the SvelteKit `handler(req, res)`
fallback. The `upgrade` listener gets a parallel branch alongside the `/ws` branch.

### 5. New component — `src/lib/components/BrowserView.svelte`

Renders the iframe and viewport controls. Top toolbar:

- **URL bar** (read-only) — shows `target_url`. A small reload button next to it that bumps a `key` on the iframe to force a reload.
- **Viewport selector** — segmented control with: Mobile (375×812), Tablet (768×1024), Desktop (1280×800), Custom (two number inputs).
- **Rotate** — swaps width and height of the current preset.
- **Open externally** — opens `/preview/<agentId>/` in a new tab.

The iframe lives in a centered, fixed-size container (the chosen viewport). It uses
`sandbox="allow-scripts allow-same-origin allow-forms allow-popups allow-modals"` —
permissive enough for typical SvelteKit/Vite apps without granting top-level navigation.

The iframe's `src` is `/preview/<agentId>/` (note trailing slash so relative asset URLs
resolve correctly). On error (proxy returns 502 / dev server not running), the component
shows a "Dev server not reachable on `<target_url>`" placeholder with a Retry button —
this is by far the most common failure mode and worth a friendly message.

### 6. Modal routing — `src/lib/components/AgentTerminalPanel.svelte`

Pass `cli_kind` into the panel and branch:

```svelte
{#if agent.cli_kind === 'browser'}
  <BrowserView agentId={agent.id} targetUrl={agent.target_url} />
{:else}
  <Terminal ... />
{/if}
```

Thumbnail tile (in the gridstack dashboard): browser agents render a static globe/browser
icon plus the target URL — no live preview. A live thumbnail would require a server-side
screenshot pipeline, which is Phase 2 territory.

### 7. Archive view

Browser agents land in the archive table just like CLI agents when the user explicitly
archives them. The "View logs" button is hidden for browser kind (no terminal_log to
replay); the row instead shows the captured `target_url` and total session duration.

# Open Decisions (please confirm before I start coding)

1. **Default port** — hardcode `5173` (Vite default), or expose as a per-user setting? My
   recommendation: hardcode the default in the form, let users override per agent.

2. **Listening interface** — should the dev server's port be reachable beyond `127.0.0.1`?
   The proxy approach assumes the dev server binds to `127.0.0.1` or `0.0.0.0`. If your
   dev server runs as a different user / inside a container, we may need to surface a
   host field too. I'm assuming same-host single-user for now.

3. **Per-agent vs per-repo target URL** — start per-agent only, defer per-repo defaults
   to a follow-up. OK?

4. **Iframe sandbox strictness** — the proposed sandbox attributes are permissive enough
   for typical Svelte/Vite apps. Tighter sandboxing (e.g., dropping `allow-popups`) is
   safer but may break some frameworks. I'd start permissive and tighten if needed.

5. **Phase 2 (Playwright monitoring)** — skip entirely from this plan, file a separate
   plan if/when wanted? Keeps Phase 1 small.

# Verification

1. **Unit tests** — `AgentSupervisor` browser branches: spawn skips tmux, reap ignores
   browser rows, kill writes `exited` without tmux call, reattach-on-boot leaves browser
   agents `running`.
2. **Proxy unit test** — mock dev server on a random port; assert `/preview/<id>/path`
   forwards correctly, including a WebSocket echo.
3. **E2E (Playwright)** — spawn a browser agent against a fixture HTTP server on a
   random port; assert the iframe URL responds 200 and renders expected content.
4. **Manual mobile smoke** — open MAW PWA on phone, spawn browser agent, navigate to
   `localhost:5173` (Vite running on the MAW host), confirm the app renders, scroll/tap
   works, viewport selector reflows correctly at 375×812.
5. **Vite HMR** — edit a `.svelte` file in the running dev server, confirm HMR updates
   propagate through the proxy without a full reload.
