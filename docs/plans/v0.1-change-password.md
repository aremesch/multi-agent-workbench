# Change Password

## Context

MAW has single-user auth (argon2id over `users.password_hash`) but no way to
rotate the password after initial signup — today the only path is editing the
DB by hand. This plan adds a minimal authenticated "Account" page with a
change-password form, reusing the existing hash/verify + session utilities.

## Approach

Add an `/account` route (SvelteKit form action) with a single form: current
password, new password, confirm new password. On success:

1. Verify `current` against `user.password_hash` with `verifyPassword`.
2. Re-hash `new` with `hashPassword`, update `users.password_hash`.
3. Delete all of the user's sessions **except the current one**, so other
   devices get logged out but the submitting tab stays signed in.
4. Return a success flag; form clears.

Errors surfaced via `fail(400/401, { error })` — same pattern as
`src/routes/login/+page.server.ts:24`.

### Validation rules (manual, matching login route style)

- All three fields required.
- `new !== current` (avoid no-op rehash).
- `new === confirm`.
- `new.length >= 8` (minimum; matches argon2 threat model — no complex rules).
- Current must verify.

## Files

### New

- `src/routes/account/+page.server.ts` — `load` (requires `locals.user`, else
  `throw redirect(303, '/login')`); `actions.changePassword` implementing the
  flow above.
- `src/routes/account/+page.svelte` — form with three password inputs,
  `enhance`-ed, error + success messaging. Match login page styling.

### Modified

- `src/lib/server/db/queries.ts` — add two helpers (this is the only file
  permitted to build SQL, per header comment):
  - `updateUserPasswordHash(userId: string, hash: string): void` —
    `UPDATE users SET password_hash = ?, updated_at = ? WHERE id = ?`.
  - `deleteSessionsForUserExcept(userId: string, keepSessionId: string): void`
    — `DELETE FROM sessions WHERE user_id = ? AND id != ?`.
- `src/routes/+layout.svelte` — add an "Account" menu item above "Archive"
  (lines 76–79) linking to `/account`; close the menu on click.

### Reused as-is

- `hashPassword` / `verifyPassword` from `src/lib/server/auth/password.ts`.
- `SESSION_COOKIE` + `locals.session` (resolved by `hooks.server.ts`) to
  identify the current session when purging others.
- CSRF: SvelteKit form actions are already CSRF-protected by the project's
  existing double-submit token (`src/lib/server/auth/csrf.ts`) — no extra
  work.

## Verification

1. `pnpm check` — expect 0/0.
2. `pnpm dev`, sign in, click hamburger → **Account**. Confirm the form
   renders and `/account` 303s to `/login` when unauthenticated.
3. Submit with wrong current password → "Current password incorrect".
4. Submit with mismatched confirm → "New passwords do not match".
5. Submit with `new.length < 8` → length error.
6. Submit valid change → success banner; log out; re-login with new password
   succeeds, old password fails.
7. Open a second browser/incognito session, change password in tab A; tab B's
   next navigation should bounce to `/login` (session row deleted), while
   tab A stays signed in.
