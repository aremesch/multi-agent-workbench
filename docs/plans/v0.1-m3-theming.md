# M3 Design Polish + Theme Selection

## Context

The app currently uses hand-picked hex colors scattered through scoped
`<style>` blocks (sidebar, topbar, hamburger menu, page bg). There's no
token system, no light mode, and the placeholder `/settings` page has no
actual settings. The user wants the UI — especially the left sidebar —
polished against the [Material Design 3](https://m3.material.io/) guide,
plus a real Settings dialog that lets the user pick from several themes.

We already have the plumbing for per-user preferences (`user_settings`
key-value table used today for `ui.sidebar.collapsed`), so theme
persistence piggy-backs on that without new migrations.

Goal: introduce an M3-aligned token system, restyle the shell (topbar,
sidebar, menu) to use it, ship a handful of built-in themes, and give
the user a Settings page where they can switch themes (persists
per-user, applies instantly, SSR-safe so there's no flash).

## Approach

### 0. Introduce Tailwind CSS v4

Tailwind is currently only a dev-dependency for Prettier. Promote it to
a first-class styling layer, CSS-first config (no JS config file):

- Add deps: `tailwindcss@^4`, `@tailwindcss/vite`.
- Register the Vite plugin in `vite.config.ts`.
- Create `src/app.css` with `@import "tailwindcss";` followed by an
  `@theme` block that maps Tailwind's color/radius/spacing names onto
  the M3 CSS variables defined in step 1 (so `bg-surface`,
  `text-on-surface`, `bg-primary-container`, `rounded-md`, etc. resolve
  through the token layer and react to `data-theme` swaps).
- Import `app.css` once from `src/routes/+layout.svelte`.
- Scoped `<style>` blocks stay allowed where they read better
  (animations, pseudo-elements, xterm/gridstack overrides) — the
  default for new shell markup is Tailwind utilities.

This is intentionally scoped to the files already in this plan
(topbar, hamburger, sidebar, settings page) plus `app.css`. Other
components keep their current scoped CSS until touched in later work.

### 1. M3 token system (CSS custom properties)

Create `src/lib/client/styles/themes.css` defining the M3 role tokens
we actually use as CSS variables on `:root`, with theme overrides
scoped by `[data-theme="<name>"]` on `<html>`. `app.css` imports this
file before the `@theme` block so Tailwind's color names alias the
tokens.

Token set (subset of M3 roles, enough for this shell):

- **Surfaces / elevation**: `--md-sys-color-surface`,
  `--md-sys-color-surface-container-lowest/low/high/highest`,
  `--md-sys-color-surface-variant`
- **On-surface**: `--md-sys-color-on-surface`,
  `--md-sys-color-on-surface-variant`
- **Primary**: `--md-sys-color-primary`, `--md-sys-color-on-primary`,
  `--md-sys-color-primary-container`,
  `--md-sys-color-on-primary-container`
- **Outline**: `--md-sys-color-outline`,
  `--md-sys-color-outline-variant`
- **State**: `--md-sys-color-error`, `--md-sys-color-success`,
  `--md-sys-color-warning` (semantic extras for status dots)
- **Shape**: `--md-sys-shape-corner-xs/sm/md/lg` (4/8/12/16px)
- **Motion**: `--md-sys-motion-duration-short`,
  `--md-sys-motion-easing-standard`
- **Elevation**: `--md-sys-elevation-level-1/2` (box-shadow stacks)

Import the stylesheet once from `src/routes/+layout.svelte`.

### 2. Built-in themes

Define five presets, covering dark/light + one vibrant expressive option:

| Name              | Mode  | Feel                                    |
| ----------------- | ----- | --------------------------------------- |
| `dark-slate`      | Dark  | Current near-black, M3-tokenized        |
| `dark-midnight`   | Dark  | Deep blue-tinged neutrals, indigo accent|
| `amoled`          | Dark  | True-black surfaces, high contrast      |
| `light-default`   | Light | M3 baseline light with violet accent    |
| `expressive-plum` | Dark  | M3 "expressive" vibe, plum + teal       |

Each theme = one `[data-theme="..."]` block in `themes.css` overriding
the token set. `dark-slate` is the default and also populates `:root`
so SSR without a cookie looks right.

### 3. Persistence & SSR

- New key: `ui.theme` in the existing `user_settings` table
  (`src/lib/shared/dashboard.ts` — add constant `USER_SETTING_THEME`).
- Extend `src/routes/+layout.server.ts` to load the theme alongside
  `ui.sidebar.collapsed` and pass it to the layout.
- In `src/routes/+layout.svelte`, set `data-theme` on
  `document.documentElement` on mount from the server-provided value
  (and keep it in a `$state` so switching updates immediately).
- Set `data-theme` server-side into `<html>` too via a small hook in
  `src/hooks.server.ts` (transform the `%sveltekit.html%` response) OR
  inject it through the layout data + an inline `<script>` in
  `app.html` stub — pick the `hooks.server.ts` route to avoid FOUC.
- Logged-out users fall through to the default theme.

Extend the existing `PUT /api/user/sidebar-state` → rename handler file
is risky; instead add a sibling: `src/routes/api/user/theme/+server.ts`
(`PUT { theme: string }`, validates against allowed set, writes via
`setUserSetting`).

### 4. Settings page

Replace the placeholder `src/routes/settings/+page.svelte` with a real
page:

- Header: "Settings" (M3 headline-small).
- Section: **Appearance → Theme**.
- Render each theme as a selectable card showing a mini
  preview (three colored swatches: primary, surface, surface-variant)
  with the theme name + a brief description. Selected card gets the
  M3 "filled card + outline" treatment.
- On click: PUT `/api/user/theme`, then update the `data-theme`
  attribute live via a small shared store
  (`src/lib/client/stores/theme.ts`, exporting `currentTheme` plus a
  `setTheme` helper).
- Use the existing `Modal.svelte` pattern if the user prefers dialog
  over page — but keep it as a route page so it deep-links.

### 5. Sidebar restyle (`RepoTreeSidebar.svelte`)

Rewrite the scoped `<style>` block to consume tokens instead of
hard-coded hexes:

- Sidebar bg → `--md-sys-color-surface`.
- Section headers ("Repositories", "Archive") → M3 label-medium,
  `--md-sys-color-on-surface-variant`, letter-spacing 0.5px.
- Rows: 40px tall, 12px horizontal padding, 12px gap, rounded-full on
  hover (M3 navigation-drawer item spec). Hover bg =
  `--md-sys-color-surface-container-high` at 60% opacity. Active/selected
  row bg = `--md-sys-color-secondary-container` with
  `--md-sys-color-on-secondary-container` text.
- Disclosure caret: 20px, rotates 90°, transition uses motion tokens.
- Agent rows under a repo indent with a subtle
  `--md-sys-color-outline-variant` left rule (1px) except when
  CLAUDE.md says "no left-rail connector on agents" — so keep the
  current indent-only treatment, just swap colors to tokens.
- Status dots → semantic tokens (`success`, `warning`, `primary`,
  `error`, `outline`). Add a 2px ring using token colors for the
  M3 "dot" look.
- Collapsed rail (1.75rem) stays width-wise but gets token colors +
  an M3 icon-button hover circle (28px) on the hamburger.

### 6. Topbar + hamburger menu (`+layout.svelte`)

- Topbar bg → `--md-sys-color-surface-container`, border →
  `--md-sys-color-outline-variant`.
- App title uses M3 title-medium.
- Hamburger menu becomes an M3 menu: rounded 8px corners, elevation
  level 2, 8px vertical padding, 48px menu items with 12px icon slot,
  hover uses `surface-container-highest`. Items: **Settings**,
  **Logout** (unchanged set).
- All existing hex values removed in favor of tokens.

### 7. Files to modify / create

Create:
- `src/app.css` — `@import "tailwindcss"` + `@theme` mapping to M3 vars.
- `src/lib/client/styles/themes.css` — tokens + theme overrides.
- `src/lib/client/stores/theme.ts` — `currentTheme` store, `setTheme`.
- `src/routes/api/user/theme/+server.ts` — PUT handler.
- `src/hooks.server.ts` — set `data-theme` on `<html>` from user
  setting (read via existing auth → `getUserSetting`).

Modify build/config:
- `package.json` — add `tailwindcss@^4` + `@tailwindcss/vite` as
  regular dependencies (Tailwind moves out of dev-only Prettier use).
- `vite.config.ts` — register `@tailwindcss/vite` plugin.

Modify:
- `src/routes/+layout.svelte` — import tokens, consume theme store,
  restyle topbar + hamburger.
- `src/routes/+layout.server.ts` — return `theme` alongside existing
  layout data.
- `src/lib/client/components/RepoTreeSidebar.svelte` — swap hex for
  tokens, tighten to M3 navigation-drawer spec.
- `src/routes/settings/+page.svelte` — real Appearance → Theme UI.
- `src/lib/shared/dashboard.ts` — add `USER_SETTING_THEME`,
  `ThemeId` union, `DEFAULT_THEME`, `ALL_THEMES` metadata list
  (id, label, description, swatches).
- `src/app.html` — add `data-theme` placeholder on `<html>` so the
  hook can substitute without flash.

Reused existing utilities: `getUserSetting` / `setUserSetting`
(already backing sidebar collapsed state), `Modal.svelte` pattern if
needed, existing form action + PUT API conventions.

## Verification

1. `pnpm check` — 0/0.
2. `pnpm build` then `node build/server.js` — app boots, no CSP errors.
3. Log in, open `/settings`, click each theme in turn — sidebar,
   topbar, hamburger, and main content recolor instantly with no
   full-page reload.
4. Hard-refresh after each selection — SSR serves the correct
   `data-theme`, no flash of the default theme before hydration.
5. Log out + log in again — last-chosen theme persists for that user.
6. Open an agent terminal modal in each theme — terminal still
   readable (xterm bg stays its own theme; token contrast remains ok).
7. Collapse sidebar → reopen — collapsed rail uses token colors,
   hover states look M3-correct.
8. Inspect `<html data-theme="...">` in devtools; toggle the attribute
   manually to confirm every surface the shell owns is tokenized
   (no stray hex colors remain in `RepoTreeSidebar.svelte` or the
   topbar/hamburger CSS blocks).
