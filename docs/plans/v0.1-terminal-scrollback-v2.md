# Replace xterm-persistence with deeper tmux snapshot on reopen

> Rename to `v0.1-terminal-scrollback-v2.md` on approval (per CLAUDE.md convention).

## Context

`v0.1-terminal-persistence` kept xterm alive across modal open/close by reparenting the host DOM node between a hidden pool and the active panel, so closing and reopening the modal preserved xterm's 10k-line scrollback verbatim. In practice this has two problems:

1. **TUI CLIs pollute the persisted scrollback.** Claude Code (the primary target) doesn't use the alt-screen buffer. Every streaming tick re-emits the full UI (banner + prompt + accumulated output). xterm faithfully appends each redraw into its scrollback, so after a minute of use the user scrolls up and sees N copies of the welcome banner stacked on top of each other. The first screenshot in testing showed exactly this.
2. **Reattach-triggered resizes redraw again.** Every pool↔modal round trip triggers `fit()` → `term.resize` → `sendResize` → tmux SIGWINCH → Claude Code redraws → another banner in scrollback. Partially mitigated with a `attachedTo`-gated ResizeObserver, but a sync `fit()` in `attachTo` racing Svelte's `$effect` ordering for `dialog.showModal()` produces 2×1 or 7-col measurements on a `display: none` ancestor, which then ship to tmux as a genuine tiny resize. Second screenshot showed the 7-col aftermath.

Both failure modes share a root cause: for a CLI that redraws on every tick, "scrollback" is not meaningful history — it's a pile of superseded renderings of the current frame. Nothing xterm-side can clean that up.

The user's proposed alternative: forget about xterm-side persistence; let tmux hold the backing scrollback (up to 500 lines); on every modal open, capture the full 500 lines, run it through the existing `collapseRepeatingTailBlocks` dedup helper, and ship it as the initial snapshot. xterm's viewport shows the last N rows (= visible pane); the rest lives in xterm's own scrollback for the current session. When the user closes and reopens, they see a fresh, deduped, up-to-date snapshot — no stale redraw stack.

This approach has three strictly-better properties vs the persistence plan:

- **Line-based CLIs get real history.** Shell/REPL sessions keep 500 lines of actual output that the user can scroll through, because tmux's scrollback for non-redrawing CLIs is clean.
- **TUI CLIs get cleaner history than they have today.** `collapseRepeatingTailBlocks` greedily folds repeating tail blocks (≥ 2 lines) out of the capture. Claude Code's banner-and-prompt redraws produce large repeating blocks — exactly the shape the helper was built for. Residual pollution is bounded to whatever fraction of redraws have subtle jitter (progress counters, spinner frames) that breaks byte-equality.
- **Every open is fresh.** No stale state persists across close/reopen, so there's no window for the reattach-resize-redraw race to accumulate junk.

The infrastructure is already there:

- `src/lib/server/tmux/TmuxSession.ts:146` — `static async capturePane(session: string, startLine = -10000)`. Accepts negative `startLine` and passes it straight to `tmux capture-pane -S <startLine>`.
- `src/lib/server/ws/hub.ts:362-385` — `export function collapseRepeatingTailBlocks(text: string): string`. Greedy O(n²) repeating-tail-block fold. **Not currently called anywhere on the live path** — dead code waiting for a caller.
- Tmux default `history-limit` is 2000 lines (no `set-option` overrides anywhere in the spawn path — confirmed in `TmuxSession.newSession`), so 500 fits with headroom.

## Approach

Revert `v0.1-terminal-persistence` and switch the reconnect-snapshot path to a 500-line deep capture piped through `collapseRepeatingTailBlocks`. Keep the **shared `MawWsClient` singleton** from that plan — it's an independent improvement (one ws connection per tab instead of one per panel) and costs nothing to preserve.

## Files

### Revert: `src/lib/client/terminalRegistry.ts`
Delete the file. All of its responsibility (xterm lifecycle, fit, subscribe, dispose) folds back into `Terminal.svelte` + `AgentTerminalPanel.svelte`.

### Restore: `src/lib/client/components/Terminal.svelte`
Recreate the thin xterm wrapper that was deleted in the persistence plan. Same shape as before:

- Dynamic-imports `@xterm/xterm` + `@xterm/addon-fit` in `onMount`
- `term.open(container)`, wires `onData`/`onResize`, initial `fit.fit()` guarded with try/catch
- Exposes `write(data)`, `clear()`, `reset()` via `bind:this`
- Queues writes that arrive before the dynamic import resolves (`pending` array)
- Owns a `ResizeObserver` on the container; `window` resize listener as a fallback
- No reparenting, no pool, no persistence

The previous file is recoverable from `git show HEAD:src/lib/client/components/Terminal.svelte` — reuse it verbatim. `xterm/xterm/css/xterm.css` import lives here again (the layout-level global CSS rules move back out).

### Rewrite: `src/lib/client/components/AgentTerminalPanel.svelte`
Back to the pre-persistence shape, but using `getMawWsClient()` (shared singleton) with per-agent handler registration:

### Keep as-is: `src/lib/client/ws.ts`

### Changed: `src/lib/server/ws/hub.ts`
Two lines in `sendReconnectSnapshot`:

```ts
// Before
const snapshot = await Tmux.capturePane(runtime.tmuxSession, 0).catch(() => '');
if (!snapshot) return;
const normalized = snapshot.replace(/\r?\n/g, '\r\n');
```

After:

```ts
// Capture ~500 lines of tmux scrollback (bounded by the default
// history-limit of 2000). For line-based CLIs this is real history;
// for TUI CLIs that redraw on every tick, collapseRepeatingTailBlocks
// folds byte-equal repeating banner/prompt blocks out before we ship
// it. Residual redraw jitter is unavoidable but bounded.
const raw = await Tmux.capturePane(runtime.tmuxSession, -500).catch(() => '');
if (!raw) return;
const snapshot = collapseRepeatingTailBlocks(raw);
const normalized = snapshot.replace(/\r?\n/g, '\r\n');
```

`collapseRepeatingTailBlocks` is already exported from the same file (`hub.ts:362`), so no import changes.

### Unchanged
- `src/routes/api/agents/[id]/snapshot/+server.ts` — dashboard thumbnail endpoint. Thumbnails stay at `capturePane(..., 0)` (visible pane only); a thumbnail is a few lines of card context and doesn't benefit from the deeper capture.
- `src/lib/server/tmux/TmuxSession.ts` — no spawn-side changes. Default `history-limit=2000` covers the 500-line ask with 4× headroom. If we later discover the default is lower in some tmux build, add `set-option -g history-limit 2000` in `Tmux.newSession`; not needed now.
- `src/lib/shared/protocol.ts` — `PROTOCOL_VERSION` stays at 2. The `scrollback` message shape doesn't change; only the content does.

## Subtleties

1. **Snapshot size vs. xterm scrollback cap.** xterm's `scrollback: 10_000` option in `Terminal.svelte` caps how many lines it remembers. A 500-line snapshot fits trivially. If we ever grow the tmux capture to 2000+, the xterm cap still holds as the hard ceiling.

2. **Dedup is byte-equal only.** `collapseRepeatingTailBlocks` collapses repeating lines where each line matches the previous block exactly. Claude Code's redraws will mostly match (banner is static, prompt text is stable) but progress counters and spinner frames (`⠋`, `⠙`, `⠹`, …) break equality. Expect some residual pollution on live-updating CLIs. Acceptable for v0.1; revisit with a diff-based collapser in v0.2 if it's noisy enough to matter.

3. **Reconnect path preserves behavior.** `MawWsClient.onopen` re-issues every active subscription on reconnect. With the new snapshot depth, a genuine ws reconnect will also receive a deeper snapshot — strictly better than today, no regression.

4. **Thumbnail divergence.** The dashboard thumbnails and the terminal modal now capture at different depths. Documented above as intentional; if this becomes confusing, unify later by making the depth a per-caller argument to a shared `Tmux.capturePane` wrapper. Not in scope.

5. **`getMawWsClient()` in the layout.** Keep it — the panel's first subscribe benefits from the connection being already `OPEN`, and there's no downside to kicking the handshake early. It's a 2-line win from the persistence plan worth preserving.

6. **No registry-style dispose-on-death path.** With per-modal lifetime, the panel cleans up on `onDestroy`. Agent death is detected by the panel's existing `onStatusChange` → dashboard `$effect` auto-close path (unchanged). No global agent_state listener needed.

## Verification

1. **Typecheck & build**
   - `pnpm check` — must stay 0/0.
   - `pnpm build` — SvelteKit prod build.

2. **TUI CLI (Claude Code) open/close**
   - Spawn a Claude Code agent. Open modal. Ask it for 50+ lines of output.
   - Close modal. Reopen. The snapshot should show the current Claude Code frame + some pre-dedup lines of history above it in xterm's scrollback. No stacked banner duplicates visible; if `collapseRepeatingTailBlocks` is doing its job at least one banner-block's worth of noise should be gone compared to raw capture.
   - Spot-check: resize the browser window, close, reopen at the new size — pane renders cleanly at new dims on every open.

3. **Line-based CLI (shell adapter)**
   - Spawn the shell smoke adapter (`cli-adapters/shell.jsonc`). Run a few commands producing distinct lines. Close modal, reopen. All prior output should be present in xterm's scrollback (deduped only if there are literal repeating tail blocks, which a shell session normally doesn't have).

4. **Multi-modal / shared ws**
   - Open agent A's modal. Open agent B's modal (close A first — one-at-a-time). Verify in browser devtools that only one `/ws` connection exists across both opens (the shared `MawWsClient` singleton at work).
   - Dedicated route: visit `/agents/<id>`. Terminal renders normally; no `pinTerminal` needed.

5. **Agent death cleanup**
   - With a modal open, kill the agent externally (tmux-kill or supervisor reaper). The `agent_state=exited` message should trigger the dashboard's auto-close `$effect` (`+page.svelte:32-38`, unchanged). Panel unmounts, calls `unsubscribe`, shared ws drops its handler entry.

6. **Reconnect**
   - Take the network offline with DevTools while a modal is open. Wait for backoff. Bring it back. `ws.onopen` should re-subscribe and the server should ship a fresh 500-line deduped snapshot. Client `term.reset()`s before applying — live output resumes.

## Non-goals

- **xterm-side persistence across modal open/close.** Explicitly dropped. If we later want it for line-based CLIs only, layer it as a per-CLI-kind opt-in in a follow-up.
- **Page-reload persistence.** Out of scope as in the original persistence plan.
- **Server-side headless xterm emulator** for clean logical history — v0.2 item, revisit for the LLM overseer.
- **Adaptive dedup.** A diff-based collapser that handles progress-counter jitter. If the byte-equal dedup proves insufficient in practice, add later — don't gold-plate preemptively.
- **Thumbnail capture depth.** Stays at visible-pane-only.
