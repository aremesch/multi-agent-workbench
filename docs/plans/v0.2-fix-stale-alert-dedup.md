# Fix: Push notifications show "ERROR rate_limit" instead of prompt question

## Context

When Claude Code shows a plan approval prompt ("Would you like to proceed?"), the
user receives no `prompt_detected` notification. Instead, they see an old
"ERROR rate_limit" notification on their phone — a stale match from earlier
terminal output that is still sitting in the adapter's 8 KB tail buffer.

Two bugs work together to produce this:

1. **Missing pattern** — the `claude-code.jsonc` adapter has no pattern for
   Claude Code's plan-approval prompt (`"Would you like to proceed?"`), so
   `prompt_detected` never fires.
2. **Stale tail re-matching** — once error text like "rate limit" enters the
   rolling tail buffer, `exec()` re-matches it on *every* subsequent `ingest()`
   call. The 30-second dedup window in `maybeAlert()` throttles but doesn't
   prevent repeated alerts from the same static text.

## Changes

### 1. Add plan-approval pattern to `claude-code.jsonc`

**File:** `cli-adapters/claude-code.jsonc`

Add a new pattern before the existing `tool_permission_prompt`:

```jsonc
{
  "id": "plan_approval_prompt",
  "kind": "prompt_detected",
  "regex": "Would you like to proceed\\?",
  "scope": "tail",
  "choices": ["1", "2", "abort"],
  "severity": "warning",
  "description": "Claude Code plan approval prompt"
}
```

Choices `["1", "2", "abort"]` map to the adapter's `promptAnswers` block
(option 1 = yes+bypass, option 2 = yes+manual, abort = Ctrl-C).

### 2. Deduplicate stale tail matches in `ConfigDrivenAdapter.ingest()`

**File:** `src/lib/server/agents/adapters/ConfigDrivenAdapter.ts`

Add a `firedMatches: Map<string, string>` (patternId → last `m[0]`) to the
adapter. On each ingest, for alert-producing kinds (`prompt_detected`, `error`,
`task_done`):

- **Skip** if `firedMatches.get(patternId) === m[0]` (same text still in tail).
- **Record** `firedMatches.set(patternId, m[0])` when firing.
- **Cleanup** after updating the tail: delete entries whose text is no longer
  in the tail (the old content scrolled out, so a fresh occurrence should fire).

State-tracking kinds (`ready`, `working`) are **not** deduplicated — they need
to re-fire for the state machine.

```typescript
// new field
private firedMatches = new Map<string, string>();

// in ingest(), after tail update:
for (const [id, text] of this.firedMatches) {
  if (!this.tail.includes(text)) this.firedMatches.delete(id);
}

// in the pattern loop, after finding a match:
const ALERT_KINDS = new Set(['prompt_detected', 'error', 'task_done']);
if (ALERT_KINDS.has(p.cfg.kind)) {
  if (this.firedMatches.get(p.cfg.id) === m[0]) continue;
  this.firedMatches.set(p.cfg.id, m[0]);
}
```

### 3. Improve `rate_limit` pattern with named capture group

**File:** `cli-adapters/claude-code.jsonc`

Change the regex to capture the matched text so the notification body is
descriptive instead of the generic "Agent encountered an error.":

```diff
-"regex": "rate.limit|quota exceeded",
+"regex": "(?<message>rate.limit|quota exceeded)",
```
