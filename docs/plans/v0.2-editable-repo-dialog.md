# Make existing repos editable (hover pencil → repo dialog)

## Context

Repos are created from the spawn-agent form (inline, via `POST /api/repos`).
Once created, they can never be edited — so a user who forgot to paste the
origin URL at create time has no way to add it later without touching the DB
directly. CLAUDE.md notes `origin_url` is optional; the natural expectation
is an edit affordance for it.

Goal: hover a repo row in the left sidebar, reveal a small pencil icon to
the right of the count, click it to open a dialog pre-populated with the
repo's current values and save the changes (primarily `origin_url`).

## UX shape

- Pencil icon appears **only on row hover** (CSS `opacity: 0` → `1`), placed
  **after** the agent count inside `.row` (outside `.row-link`, because an
  `<a>` can't contain a button).
- Clicking opens `RepoEditDialog` — a new modal that shows:
  - **Path** (read-only, disabled input — changing it would break worktrees
    and running agents)
  - **Origin URL** (editable, optional)
- Save → `PUT /api/repos/[id]` → dialog closes, sidebar reflects nothing
  visible (path/name unchanged), but subsequent edits see the new value.
- Scope is the active Repositories list only. The Archive list is a filtered
  view of the same repos; no separate button there.

## Files to modify

### 1. DB helper — `src/lib/server/db/queries.ts`

Add `updateRepo` next to `insertRepo` (queries.ts:235). Updates only the
mutable columns plus `updated_at`:

```ts
export function updateRepo(row: {
  id: string;
  user_id: string;
  origin_url: string | null;
}): boolean {
  const res = prep<[string | null, number, string, string], never>(
    'UPDATE repos SET origin_url = ?, updated_at = ? WHERE id = ? AND user_id = ?'
  ).run(row.origin_url, now(), row.id, row.user_id);
  return res.changes > 0;
}
```

Ownership is enforced by the `user_id = ?` predicate so a foreign user who
guessed an ID can't mutate. No migration needed — all columns already exist
(migration 004 already made `project_id` nullable, `default_branch`
nullable).

### 2. API — new route `src/routes/api/repos/[id]/+server.ts`

Two handlers:

- `GET` — returns `{ id, path, origin_url, default_branch, projectName }`
  for the dialog to pre-fill. Auth + ownership check via `getRepo` +
  `user_id` compare.
- `PUT` — CSRF (`verifyCsrf`), auth check, JSON body
  `{ origin_url?: string | null }`. Normalize empty string → `null`.
  Call `updateRepo`; 404 if no row changed, else `{ ok: true }`.

Pattern mirrors `POST /api/repos` (src/routes/api/repos/+server.ts:13) —
same `verifyCsrf` + `locals.user` guard + `t(locals.locale, …)` error
strings. Do **not** accept `path` in the PUT body (server ignores it).

### 3. New component — `src/lib/client/components/RepoEditDialog.svelte`

Props: `{ open: boolean; repoId: string | null; onClose: () => void;
onSaved: (updated: { id: string; origin_url: string | null }) => void }`.

Wraps the existing `Modal.svelte` (src/lib/client/components/Modal.svelte).
On open (`$effect` watching `open && repoId`), fetch `GET /api/repos/[id]`
via `apiFetch` into local `$state` bindings. Submit via
`apiFetch('/api/repos/[id]', { method: 'PUT', … })` — same pattern as

`<input>`, origin URL as an editable `<input>`, error line, Cancel/Save
buttons. Match the existing `.inline-form` CSS vibe so it feels familiar.

## Tests

- Vitest unit: `updateRepo` — happy path, wrong user (no row changes),
  null vs. non-null origin URL. Mirror the existing `insertRepo` test
  (search `insertRepo` under `tests/`).
- Vitest unit: `PUT /api/repos/[id]` handler — 401 without auth, 403/404
  for foreign user, 200 on success. Match the pattern of existing

- Skip Playwright for now — the feature is visual but the logic is
  already covered by the unit tests. A regression would show up as a
  401 or missing button, both easy to spot.

## Verification

1. `pnpm check` — must stay 0/0.
2. `pnpm test` — new unit tests green, existing suite unchanged.
3. Manual preview flow:
   - Start dev server (`pnpm dev`).
   - Create a repo **without** an origin URL via the spawn form.
   - Hover the new repo in the sidebar — pencil appears after count.
   - Click pencil — dialog opens with path filled (disabled) and origin
     URL empty.
   - Enter a URL, Save — dialog closes.
   - Reopen the dialog — URL persists.
   - Reload the page — URL still persists (DB round-trip).
   - Foreign-user attempt via curl to `PUT /api/repos/<otherId>` → 404.

## Out of scope (not in this change)

- Editing `path` — touching it would invalidate every worktree rooted at
  that repo. If the user wants to move a repo, that's a dedicated
  "relocate" flow with its own validation (re-resolve git dir, re-check
  branches, update worktree paths). Flag in a separate plan if needed.
- Editing `default_branch` — affects future agent spawns only but isn't
  part of the create form either, so leaving it out keeps edit and create
  symmetric.
- Delete/archive repo from the sidebar — separate destructive action with
  its own confirmation UX.
- Refactor: extract a shared `RepoFormFields` component used by both
  create and edit. Not justified for two fields today.
