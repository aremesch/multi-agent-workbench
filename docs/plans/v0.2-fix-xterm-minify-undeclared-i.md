# Show-log crash: `ReferenceError: assignment to undeclared variable i`

## Context

Occasionally, opening **Show Log** for a still-running coding-CLI agent (claude-code,
codex, gemini) on `/repos/<id>/archive` or via the agent-window kebab menu paints a
blank black panel and Firefox prints:

```
Uncaught ReferenceError: assignment to undeclared variable i
    Immutable 43
    async* https://maw.remesch.at/repos/01KP1AFMXNP98TK2JA5SZG7427/archive:33
    promise callback* https://maw.remesch.at/repos/01KP1AFMXNP98TK2JA5SZG7427/archive:32
BqvuqXEL.js:24:103434
```

Bash/shell agents never trigger it; large or "rich-TUI" logs do. This plan documents
the root cause and proposes a fix.

---

## Root cause

**The bug is in our Vite production bundle of [`@xterm/xterm@6.0.0`](https://www.npmjs.com/package/@xterm/xterm), not in MAW source.**

### Trace decoded

`BqvuqXEL.js` is the xterm.js chunk in `build/client/_app/immutable/chunks/`
(verified locally — the file starts with the xterm copyright header). At column
103434 of its single 135 KB minified line is the body of `InputHandler.requestMode`,
which handles incoming **DECRQM** sequences (`CSI Pm $ p` / `CSI ? Pm $ p` —
"Request DEC private mode"):

```js
// build/client/_app/immutable/chunks/BqvuqXEL.js (line 24, col ~103300)
requestMode(e,t){
  (g=>(g[g.NOT_RECOGNIZED=0]="NOT_RECOGNIZED",
       g[g.SET=1]="SET",
       g[g.RESET=2]="RESET",
       g[g.PERMANENTLY_SET=3]="PERMANENTLY_SET",
       g[g.PERMANENTLY_RESET=4]="PERMANENTLY_RESET")
  )(void 0||(i={}));   // <-- `i` is never declared in this function scope
  let s=this._coreService.decPrivateModes, ...
}
```

`(void 0||(i={}))` evaluates `i = {}` because `void 0` is falsy. Since `i` was
never declared with `let`/`var`/`const`, ES module strict mode throws
`ReferenceError`.

### How the bug got there

The upstream source in [`node_modules/@xterm/xterm/lib/xterm.mjs`](../../node_modules/@xterm/xterm/lib/xterm.mjs) is **valid** — same column, search for `NOT_RECOGNIZED`:

```js
requestMode(e,i){
  let r;                                      // <-- valid declaration
  (P=>(P[P.NOT_RECOGNIZED=0]="NOT_RECOGNIZED", ...))
  (r||={});                                   // r ||= {}  → r = r || {}
  let n=this._coreService.decPrivateModes, ...
}
```

This is a TypeScript `enum ModeReportType` declared inside a function. TypeScript
emits the standard `var X; (function(X){...})(X || (X = {}));` pattern, then xterm's
own webpack build modernizes it to `let r; ... (r ||= {});` using ES2021 logical
assignment. Note that `r` is **only ever written** inside the IIFE call — the
enum members aren't read anywhere else in the function (the `requestMode` body
just compares against magic numbers like `f === 2`, `f === 4`, …).

### The narrow esbuild bug

esbuild itself handles the `let r; ...(r ||= {})` pattern correctly when the
target supports `||=` natively. The bug only fires when **all four** of these
hold simultaneously:

1. **The build target predates ES2021.** Logical-assignment operators
   (`||=`, `&&=`, `??=`) are ES2021. SvelteKit's adapter-node sets the client
   build target to `['es2020', 'edge88', 'firefox78', 'chrome87', 'safari14']`
   (verified locally — `firefox78` is the limiter; Firefox didn't ship `||=`
   until 79). So esbuild **must** transpile `r ||= {}` for that target.
2. **`minify: true`** is on (Vite's default for `vite build`).
3. **`treeShaking: true`** is on (Vite passes this with minify).
4. **The variable on the LHS is never read after the assignment.** xterm 6.0
   creates an enum object via the IIFE and then never references it again
   in this function — the enum's only purpose was the IIFE's named-binding side
   effect.

Under those four conditions, esbuild's transpile-then-minify pipeline does:

1. Recognize `r ||= {}` is unsupported by the target → expand to
   `r || (r = {})`.
2. Notice `r`'s only read is the LHS of the new `||`, and `let r;` makes
   `r` definitionally `undefined` at that point → constant-fold to
   `void 0 || (r = {})`.
3. Notice `r`'s only remaining reference is now the *assignment* RHS in
   `r = {}`. The minifier's DCE deems `let r;` "unread" and drops it.
4. Identifier-mangle `r` → `i` (the original parameter `i` was already renamed
   to `t`, freeing the slot).

Result: `(void 0||(i={}))` with **no `let i;`**. The assignment binds against
nothing, and ESM strict mode throws `ReferenceError`.

This is verifiable with esbuild 0.25.12 (Vite 6.4's bundled version) directly:

```sh
# With es2020 target → buggy output (drops `let i;`)
esbuild --minify --tree-shaking=true --target=es2020 --format=esm in.mjs
# With es2022 target → correct output (preserves `let i;`)
esbuild --minify --tree-shaking=true --target=es2022 --format=esm in.mjs
```

So this is **not** a "esbuild silently swallows variable declarations" bug
that would hit every app — it's a corner case in the transpile+minify+DCE
interaction that only fires when the target forces logical-assignment expansion
AND the source uses a write-only `let` pattern. Web-app builds that target
modern browsers (es2022+, or the current SvelteKit defaults raised) never see
it; library/CLI builds targeting Node 14 or older browsers can.

### Why intermittent

`requestMode` runs only when xterm parses a DECRQM byte sequence. Plain bash
output never sends DECRQM; rich TUIs (claude-code's permission UI, codex,
gemini, anything ncurses-ish) probe terminal capabilities at startup with
exactly this sequence, so their **captured `terminal_log`** contains it, and
the one-shot replay through `term.write(bytes)` in
[`ArchivedAgentLogModal.svelte`](../../src/lib/client/components/ArchivedAgentLogModal.svelte#L62-L66)
re-feeds those bytes into the broken handler, crashing xterm and leaving the
modal blank.

The live terminal modal fires `requestMode` too, but only when the CLI sends
DECRQM in the moment — which doesn't always happen during a viewing session,
hence "sometimes works".

### Versions in play

- `@xterm/xterm` **6.0.0** (latest as of January 2026)
- Vite **6.4.2**, with Vite-bundled esbuild **0.25.12** (used for `vite build`)
- Top-level esbuild **0.28.0** (used by `scripts/bundle-server.mjs` — server-side
  only, not the client chunk)

The buggy chunk comes from the **vite build → Vite-bundled esbuild** path.

## Verification of the diagnosis (already done)

- Inspected `BqvuqXEL.js`: at column ~103253 the `requestMode(e,t){` function
  body opens; between `{` and `i={}` (179 chars) there's no `let i;` or
  `var i;`. Confirmed via brace-balanced backward walk from the error site.
- Captured what Vite actually feeds esbuild minify by monkey-patching
  `esbuild.transform` in `node_modules/.pnpm/esbuild@0.25.12/.../lib/main.js`:
  options object includes `target: ['es2020', 'edge88', 'firefox78',
  'chrome87', 'safari14']`, `minify: true`, `treeShaking: true`. Input still
  has the readable `let r; ...(r||={})`. Output has the buggy
  `(void 0||(i={}))`.
- Reproduced with a direct `esbuild.transform` call passing the captured
  options on the captured input — `target: es2020` produces the bug,
  `target: es2022` does not.
- `pnpm exec esbuild --minify --target=es2022 --format=esm <file>` on the
  full xterm `requestMode` produces correct output (`let T;...(T||={})`).
- Upstream `node_modules/@xterm/xterm/lib/xterm.mjs` source is valid:
  `let r;(P=>(...))(r||={})`.
- `grep -rEn '\bi\s*=\s*0\b|for\s*\(\s*(?:var\s+)?i\s*=' src/` — every match
  uses `let i`. No undeclared `i` in MAW source.

## Recommended fix: switch the client minifier from esbuild to terser

Vite supports both. Terser doesn't have this transpile+DCE interaction bug —
its handling of unsupported logical-assignment + DCE keeps the binding around.
Build time goes up by ~5–10 s, acceptable for a self-hosted app where dev
iteration runs on `pnpm dev` (no minification) and only `vite build` takes
the hit.

### Changes

1. **Add dep**: `pnpm add -D terser`
2. **[`vite.config.ts`](../../vite.config.ts)** — add to `defineConfig({ ... })`:
   ```ts
   build: { minify: 'terser' }
   ```
   Default terser compress handles the pattern correctly; no options needed.

### Why not the alternatives

| Option | Why rejected |
|---|---|
| Bump `build.target` to drop `firefox78`/`chrome87`/`safari14` (so esbuild sees `\|\|=` as native) | Cleanest semantic fix, but the Vite/SvelteKit defaults are owned by the adapter. Override would need to be re-checked on every SvelteKit upgrade. Also drops support for older browsers we still nominally cover. Worth considering as a follow-up. |
| `supported: { 'logical-assignment': true }` in vite.config | Tells esbuild "treat `\|\|=` as supported regardless of target". Fixes this *one* pattern but doesn't immunize against other transpile+minify interactions. Terser sidesteps the whole class. |
| `pnpm add patch-package` + edit `xterm.mjs` to `let r = {}; ...(r);` | Adds a runtime dep, ties us to xterm 6.0.0 layout, re-applies on every install. Heavier than a one-line minifier swap. |
| `build: { minify: false }` | Bundle size jumps ~2×. Acceptable but loses identifier-shortening wins; also doesn't catch other minify bugs that may surface in unrelated chunks. |
| Pin `@xterm/xterm` to 5.x | 5.x didn't ship the `\|\|=` pattern, but loses 6.0 features (synchronized output, etc.) and we already verified 6.0 works fine through terser. |

### Optional follow-up: file the esbuild bug upstream

Minimal repro for [evanw/esbuild](https://github.com/evanw/esbuild/issues)
(needs `target: es2020` to force `||=` transpilation):

```js
// in.mjs — `r` is written but never read after, hits the DCE path
export function f() {
  let r;
  ((P) => (P[P.X = 0] = "X"))(r ||= {});
}
```

```sh
# Triggers the bug — drops `let r;` and emits `(void 0||(i={}))` undeclared
esbuild --minify --tree-shaking=true --target=es2020 --format=esm in.mjs

# Correct — preserves the binding
esbuild --minify --tree-shaking=true --target=es2022 --format=esm in.mjs
```

Not blocking the fix.

## Summary

`Show Log` crashes intermittently because Vite's esbuild minifier mishandles the
ES2021 logical-assignment pattern `let r; ...(r ||= {})` inside xterm 6.0's
`requestMode`, dropping the `let r;` declaration but keeping the renamed
assignment `i = {}`. Switching `vite build`'s client minifier to **terser** (one
config line + one dev dep) sidesteps the bug without touching xterm or our
source. Verification is a re-grep of the rebuilt chunk plus the existing test
suites.
