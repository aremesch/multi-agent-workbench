# v0.1 Security Hardening — Phase 2 (remaining items)

## Context

The original [`v0.1-security-hardening.md`](v0.1-security-hardening.md)
identified six pre-pentest hardening priorities. Only one shipped — the
change-password flow at `/account` (commit `5a43fe6`). The rest are still
open and the external pentest is approaching, so we want the trivially
fixable gaps closed now. This plan re-scopes the remaining five priorities
into a single executable sequence, with **fail2ban integration** as a
named goal (drop-ins consume the auth log produced in step 2).

Everything here is small and surgical — no PWA, no 2FA, no overseer,
no admin UI. Out-of-scope items from the original plan (TOTP, session
binding, audit-log UI, distributed rate limiting, admin user mgmt) stay
out of scope.

## Step-by-step (one commit per step)

### 1. Migration + types
`migrations/004_auth_events.sql`:
```sql
ALTER TABLE users ADD COLUMN must_change_password INTEGER NOT NULL DEFAULT 0;
ALTER TABLE users ADD COLUMN password_updated_at INTEGER;
CREATE TABLE auth_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  ts INTEGER NOT NULL,
  event TEXT NOT NULL,        -- login_ok|login_fail|pwchange_ok|pwchange_fail|session_revoked|rate_limited|ws_origin_reject
  user_id TEXT, username TEXT, ip TEXT, user_agent TEXT, detail TEXT
);
CREATE INDEX idx_auth_events_ts ON auth_events(ts);
CREATE INDEX idx_auth_events_ip_ts ON auth_events(ip, ts);
```
Extend `UserRow` and add the three queries listed above. Bootstrap-user
insert flips `must_change_password=1` only when the password came from
`MAW_BOOTSTRAP_PASSWORD`.

### 2. Auth log + client-IP helper + rate limiter
Three small files. `clientIp` reads `X-Forwarded-For` only when
`cfg.trustProxy`; otherwise `event.getClientAddress()`. `authLog.logAuth()`
sanitizes (`/[\r\n\t]/g` → space, slice 256), appends one line in a
fixed format (`<iso> <event> user=<u> ip=<ip> ua="<ua>"`), and
inserts a row — both wrapped in try/catch so logging never throws.
`rateLimit.checkRate(key, count, windowSec)` — sliding window, occasional
GC when map > 10k.

Reuse the existing fetch-wrapper pattern; do not introduce a second one.

### 3. Login + change-password hardening
`src/routes/login/+page.server.ts`:
- Resolve `ip = clientIp(event)`, `ua = headers.get('user-agent')` first.
- `if (!checkRate('login:'+ip, ...)) { logAuth('rate_limited', ...); return fail(429, ...) }`.
- On wrong creds: keep constant-time `verifyPassword`, then `logAuth('login_fail', ...)`.
- On success: `logAuth('login_ok', ...)`. If `user.must_change_password`, `redirect(303, '/account')`.
- `logout` action: `logAuth('session_revoked', ...)`.

`src/routes/account/+page.server.ts` (already exists): on success, also
`setMustChangePassword(user.id, 0)` + `markPasswordUpdated(user.id)` and
`logAuth('pwchange_ok'|'pwchange_fail', ...)`.

### 4. CSRF on JSON routes
For each of the five `+server.ts` files listed above, at the top of every
non-GET handler:
```ts
verifyCsrf(request, cookies); // throws error(403,'csrf')
```
Update `csrf.ts` so `verifyCsrf` throws (don't return bool — easier to
forget). Client wrapper attaches `x-csrf-token` from the `maw_csrf`
cookie. If no central wrapper exists, add one — do not sprinkle
`document.cookie` reads across callsites.

### 5. WebSocket Origin check
`src/lib/server/ws/hub.ts`, in the upgrade handler next to the session-cookie
check:
```ts
const origin = req.headers.origin;
const allowed = getConfig().publicOrigin;
if (allowed && origin && origin !== allowed) {
  logAuth('ws_origin_reject', { ip, userAgent: ua, detail: origin });
  ws.close(4403, 'origin'); return;
}
```
`MAW_PUBLIC_ORIGIN` env: required in prod (warn if unset), optional in dev.
Behind Caddy, this is the browser-visible origin, not the upstream.

### 6. SameSite=strict + security headers + must-change gate
- `session.ts`: `sameSite: 'lax'` → `'strict'`. Verify bookmarked navs still work post-login (strict allows top-level same-site, blocks cross-site cookied requests).
- `hooks.server.ts` `handle`:
  - Before `resolve`: if `event.locals.user?.must_change_password` and the path is not `/account`, `/login`, `/logout`, or a static asset → `redirect(303, '/account')`.
  - After `resolve`: set CSP, HSTS, X-Content-Type-Options, Referrer-Policy, Permissions-Policy. Svelte hydration needs `style-src 'unsafe-inline'`; do **not** add `'unsafe-inline'` to `script-src`. Watch DevTools for CSP violations and iterate.

### 7. fail2ban drop-ins
`deploy/fail2ban/filter.d/maw-auth.conf`:
```ini
[Definition]
failregex = ^\S+ login_fail user=\S+ ip=<HOST> ua=".*"$
            ^\S+ pwchange_fail user=\S+ ip=<HOST> ua=".*"$
            ^\S+ rate_limited user=\S+ ip=<HOST> ua=".*"$
            ^\S+ ws_origin_reject user=\S+ ip=<HOST> ua=".*"$
ignoreregex =
```
`deploy/fail2ban/jail.d/maw.conf`:
```ini
[maw]
enabled = true
port    = http,https
filter  = maw-auth
logpath = /var/log/maw/auth.log
findtime = 600
maxretry = 5
bantime  = 3600
```
README deploy note: symlink `${MAW_DATA_DIR}/auth.log` →
`/var/log/maw/auth.log`, then `fail2ban-client reload`.
