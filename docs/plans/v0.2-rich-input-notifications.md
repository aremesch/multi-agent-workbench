# Rich input-prompt notifications & foreground toast

## Context

The user wants notifications when any running agent is waiting on input that:

1. Fire in the browser **and** the PWA.
2. Tap/click forwards directly to the agent's terminal pane.
3. Identify *which* agent needs interaction.
4. Describe *what* is being asked, when the adapter can extract it.
5. Cover all CLI adapters where feasible (else claude-code only).

**Most of this already ships in the v0.2 push stack** ‚Äî what's missing is the
content quality of the alert and the loop closing when the user answers it.

### Gaps this plan closes

1. **Detection is regex-only and brittle.**
   Today every CLI adapter relies on `ConfigDrivenAdapter` regex-matching
   ANSI-stripped terminal output. The shipped patterns are admitted starter
   regexes ([cli-adapters/claude-code.jsonc:18](cli-adapters/claude-code.jsonc:18))
   and the real claude-code TUI uses box-drawing prompts that may not match.
   For **claude-code specifically**, there is a vastly more reliable channel:
   the documented `Notification` hook (event `notification_type:
   permission_prompt` / `idle_prompt`), plus `PermissionRequest` and
   `PreToolUse` hooks. Each delivers structured JSON
   (`session_id`, `tool_name`, `tool_input`) on stdin to a per-agent
   shell command ‚Äî no regex tuning, no TUI-format coupling.

2. **Notification title hides *which* agent.**
   `alertReason()` formats `"${cli_kind}: Permission needed: ${tool}"`. With
   several agents running, the user can't tell which task is blocked. The
   instance title (`agents.title`, set at spawn) is never used.

3. **Notification body is generic.**
   `alertBody()` returns `"Choices: yes, no"` ‚Äî useless. Even with the regex
   path's named groups (`tool`, `args`, `cmd`, `action`), the body formatter
   ignores them. With hooks we get *richer* detail straight from claude-code
   (`tool_input.command` for Bash, `tool_input.file_path` for Write, etc.).

4. **No in-app foreground notification.**
   When the dashboard tab is foregrounded, only the sidebar dot reacts. The
   OS-level web-push still fires, but it can be missed when the user has
   the tab visible. A small toast "Agent X needs input ‚Äî Bash: rm -rf
   /tmp/foo ¬∑ [Open]" closes that gap and is the literal "in the browser"
   half of the request.

5. **Alerts never get acknowledged.**
   [`acknowledgeAlert()`](src/lib/server/db/queries.ts:862) exists but no
   client code calls it. The sidebar warning dot is sticky until the
   adapter itself emits a different state event, and the 30-s dedup window
   may suppress a follow-up prompt that *should* re-alert.

## Goals (single PR)

- For **claude-code**, prompt detection comes from the official
  `Notification`/`PermissionRequest` hooks. Existing regex stays in place
  as a defensive fallback if the hooks didn't write or fired late.
- For **codex / gemini**, regex stays primary (no equivalent hook API);
  they still benefit from the new payload formatter and the foreground
  toast.
- `prompt_detected` notifications (push + foreground) carry the agent's
  instance title and a useful one-line summary of *what* it asked.
- Same payload shape works for `task_done`, `error`, `exited` ‚Äî no
  regression on the existing kinds.
- Tapping/clicking either the OS push or the foreground toast lands the
  user on the correct agent's terminal pane.
- Visiting an agent that has unacknowledged alerts marks them acked; the
  sidebar warning dot clears when no alerts are pending.
- Shell / browser adapters stay no-op as today.

## Design

### 1 ¬∑ Claude Code hooks as primary detection (claude-code only)

**Mechanism.** Claude Code reads `.claude/settings.json` and
`.claude/settings.local.json` from the project (worktree) it's running
in. We write a per-agent settings file at spawn that registers a
`Notification` hook (covering `permission_prompt`, `idle_prompt`,
`elicitation_dialog`) plus a `PermissionRequest` hook for the explicit
permission-dialog channel. Each hook runs a shell command that pipes
the hook event JSON (delivered on stdin by Claude Code) to a localhost
MAW endpoint with a per-agent bearer token.

**Worktree settings file** ‚Äî written by `WorktreeManager.create` after
`git worktree add` succeeds, only when `cli_kind === 'claude-code'`:

```jsonc
// <worktree>/.claude/settings.local.json (gitignored, never tracked)
{
  "hooks": {
    "Notification": [
      { "command": "curl -fsS -X POST -H 'Authorization: Bearer ${TOKEN}' -H 'Content-Type: application/json' --data-binary @- 'http://127.0.0.1:${PORT}/api/internal/claude-hook'" }
    ],
    "PermissionRequest": [
      { "command": "curl -fsS -X POST -H 'Authorization: Bearer ${TOKEN}' -H 'Content-Type: application/json' --data-binary @- 'http://127.0.0.1:${PORT}/api/internal/claude-hook'" }
    ]
  }
}
```

`${TOKEN}` and `${PORT}` are substituted at write-time. The token is
generated once per agent (cryptographically random, stored in a new
`agents.hook_token` column ‚Äî migration `008_agent_hook_token.sql`).

**Token plumbing.** Token is also passed as `MAW_AGENT_TOKEN` env to the
spawned tmux session so any future hook variants don't need re-write of
the settings file. The hook command above uses literal substitution
because Claude Code does not expand env in the `command` string.

**Internal endpoint** ‚Äî `POST /api/internal/claude-hook` in
`src/routes/api/internal/claude-hook/+server.ts`:

1. Reject any request whose `request.url`'s host isn't loopback
   (defense-in-depth ‚Äî even if the server later binds publicly).
2. Read `Authorization: Bearer <token>`; look up agent by token (new
   `getAgentByHookToken(token)` query). Constant-time string compare.
3. Parse JSON body. Extract `session_id`, `hook_event_name`,
   `notification_type`, `tool_name`, `tool_input`.
4. Map to `AdapterEvent`-compatible shape and feed it through the same
   `insertEvent` + `maybeAlert` path used by the regex pipeline. The
   alert payload `detail` includes `tool_name`, `tool_input.command`
   (Bash) or `tool_input.file_path` (Write) etc.
5. Always respond 204 quickly ‚Äî Claude Code waits for the hook command
   to exit before showing the prompt to the human, so this must be
   fast (<100 ms goal).

**Dedup.** The same fired-match map lives on regex; for hook events we
key dedup on `(agentId, session_id, tool_use_id ?? hook_event_name)` so
the regex fallback doesn't double-fire when both paths fire on the same
prompt. Add a small `claudeHookDedup` map alongside the existing
30-second alert dedup.

**Defensive fallback.** Regex patterns stay enabled in
`cli-adapters/claude-code.jsonc`. If the hooks file failed to write
(disk full, permissions, etc.) or if Claude Code is invoked with a flag
that disables hooks, the regex still detects the prompt ‚Äî just less
richly. The `(agentId, patternId)` dedup window already prevents both
paths from creating two alerts for the same prompt.

**Threat model.** The bearer token lives in plaintext in
`<worktree>/.claude/settings.local.json` and in the agent's tmux env.
Anyone with file-read access to the worktree can hit
`/api/internal/claude-hook` for that one agent and forge events. For a
self-hosted single-user workbench this is acceptable ‚Äî file system
ACLs already protect the worktree dir. Do not use this token-only
scheme without the loopback restriction; together they form
defence-in-depth.

**Loopback-only enforcement.** Use SvelteKit's `event.getClientAddress()`
to drop any request whose remote IP isn't `127.0.0.1` or `::1`. Reject
*before* the bearer check so token-comparison time isn't a side channel.

### 2 ¬∑ Richer alert content (server-side only)

Edit [src/lib/server/agents/AgentRuntime.ts:253-277](src/lib/server/agents/AgentRuntime.ts:253):

```ts
function alertReason(agent: AgentRow, ev: AdapterEvent): string {
  const who = agent.title?.trim() || agent.cli_kind;
  switch (ev.kind) {
    case 'prompt_detected': {
      const what = ev.detail?.tool ?? ev.detail?.action;
      return what ? `${who} ¬∑ Permission needed: ${what}` : `${who} ¬∑ Permission needed`;
    }
    case 'task_done':  return `${who} ¬∑ Task complete`;
    case 'error':      return ev.patternId ? `${who} ¬∑ ${ev.patternId}` : `${who} ¬∑ Error`;
    default:           return `${who} ¬∑ ${ev.kind}`;
  }
}
```

`alertBody` gets a priority chain that surfaces the most useful detail
field present on the event:

```ts
function alertBody(ev: AdapterEvent): string {
  if (ev.kind === 'prompt_detected') {
    const d = ev.detail ?? {};
    if (d.cmd)    return String(d.cmd).slice(0, 200);     // claude-code shell_command_prompt
    if (d.args)   return `${d.tool ?? 'tool'}(${String(d.args).slice(0, 180)})`;
    if (d.action) return String(d.action).slice(0, 200);  // codex / gemini
    if (ev.choices?.length) return `Choices: ${ev.choices.join(', ')}`;
    if (ev.raw)   return ev.raw.slice(0, 200);            // last-resort: matched text
    return 'Agent needs your input.';
  }
  // task_done / error branches unchanged.
}
```

The push payload's `data` block grows two fields:

```ts
data: {
  agentId, alertId, url,
  agentTitle: this.agent.title ?? null,
  severity            // already passed via SC_Alert; useful for the toast
}
```

`PushPayload` type updated in
[src/lib/server/push/PushService.ts](src/lib/server/push/PushService.ts).
Service worker payload type widens accordingly
([src/service-worker.ts:115-119](src/service-worker.ts:115)).
No DB-schema or migration change ‚Äî `agents.title` is already stored.

### 3 ¬∑ Foreground in-app toast

The WS hub already emits `SC_Alert` per-agent. For a dashboard-wide toast
we need a user-scoped channel so the toast fires regardless of which
panel is currently subscribed.

- New client-‚Üí-server protocol message
  [src/lib/shared/protocol.ts](src/lib/shared/protocol.ts):
  `CS_SubscribeUserAlerts { type: 'subscribe_user_alerts' }` (and
  `unsubscribe_user_alerts`). Reuses the existing `SC_Alert` shape on the
  reply path ‚Äî *no* server-‚Üí-client protocol change.
- [src/lib/server/ws/hub.ts](src/lib/server/ws/hub.ts) keeps a
  `Map<userId, Set<WSConnection>>` and an `onAnyAlert(userId, alertPayload)`
  hook. When `AgentRuntime` emits `alert`, the hub already routes it to
  per-agent subscribers; we add a second fanout into the user-scoped set.
  ~40 lines.
- [src/lib/client/ws.ts](src/lib/client/ws.ts) (`MawWsClient`) gains
  `subscribeUserAlerts(handler)`. The shared singleton means one socket
  per tab regardless of how many components subscribe.
- New Svelte store
  `src/lib/client/stores/alertToasts.ts` holding an array of active
  toasts keyed by `alertId`, with `push(toast)` and `dismiss(id)`.
- New component `src/lib/client/components/AlertToast.svelte` ‚Äî renders
  a stack top-right with title (= `reason`), body, severity colour, an
  **Open agent** button (`goto('/repos/{repo}?agent={id}')`), and a
  close X. Auto-dismiss after 12 s for `info`, manual-only for
  `warning`/`error`/`critical`. The store also dedupes by `alertId` so
  a re-fire of the same alert doesn't stack.
- Mount one `<AlertToastContainer>` and subscribe to user-alerts in
  [src/routes/+layout.svelte](src/routes/+layout.svelte) right after the
  existing `registerPush(...)` call.

The toast **payload** carries the same fields the push payload now
ships (title, body, severity, url, agentId), so we have one renderer.

### 4 ¬∑ Auto-acknowledge on visit

- DB helper `acknowledgeAgentAlerts(agentId, userId)` in
  [src/lib/server/db/queries.ts](src/lib/server/db/queries.ts) ‚Äî single
  UPDATE that flips `acknowledged_at` for every unacked alert of that
  agent, scoped by `user_id`.
- New route `POST /api/agents/[id]/alerts/ack` calling the helper. Returns
  count of rows acked; idempotent.
- [src/routes/repos/[id]/+page.svelte](src/routes/repos/[id]/+page.svelte)
  ‚Äî inside the existing `?agent=` effect that opens the modal, fire-and-
  forget `apiFetch('/api/agents/{id}/alerts/ack', { method: 'POST' })`.
- After a successful ack, also `dismiss(alertId)` any matching toast in
  the store and clear `agents.attention_at` server-side so the sidebar
  warning dot turns off if no other unacked alerts remain. The hub then
  broadcasts an `agent_state` change so all open tabs converge.

### 5 ¬∑ Out of scope ‚Äî codex / gemini pattern tuning

For the non-claude-code adapters there is no equivalent hook API ‚Äî
detection stays on regex, and those patterns are still admitted starter
regexes that need recordings of the real TUI to confirm they fire.
Adapter tuning is empirical, not architectural ‚Äî once the regexes
capture the right named groups, the new body formatter surfaces them
automatically. Track as a follow-up plan.
