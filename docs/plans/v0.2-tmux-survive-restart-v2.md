# v0.2 — tmux survives maw restart, take 2

## Context

v0.2's first attempt (see [`v0.2-tmux-survive-restart.md`](v0.2-tmux-survive-restart.md))
moved the tmux server onto a dedicated `-L maw` socket and tried to keep it
out of maw's cgroup by launching it with
`systemd-run --user --scope --unit=maw-tmux -- tmux -L maw start-server`.

On the production host (`user@3002.service`, Debian-ish) this does **not**
survive `systemctl --user restart maw`. Diagnostics:

- `systemctl --user status maw-tmux.scope` → *"Unit maw-tmux.scope could not be found"* (the scope never stuck).
- `pgrep -af "tmux.*-L maw"` → one server running.
- `/proc/<tmux-pid>/cgroup` → `…/app.slice/maw.service`.
- On `systemctl --user restart maw` → `tmux -L maw list-sessions` =
  *"no server running"*. All agents lost.
- `maw.service` has default `KillMode=control-group`, `TimeoutStopSec=5`,
  and the stop phase escalates to SIGKILL of every PID in the cgroup.

### Why `systemd-run --scope` doesn't work here

`systemd-run --scope <cmd>` creates a transient scope whose *main process*
is `<cmd>`. `tmux start-server` forks the real daemon (double-fork to
detach) and the CLI returns immediately. systemd observes the scope's main
process exit, declares the scope done, and tears it down — which kills the
freshly-forked tmux daemon. `ensureServer()` then sets `serverEnsured=true`
and returns, oblivious that no server is actually running. The very first
`tmux -L maw new-session -d …` auto-spawns a **new** server as a child of
the node process, inheriting `maw.service`'s cgroup — exactly what we see
in `/proc/<pid>/cgroup`.

`--scope` is the wrong primitive for a fork-and-exit daemon. The right
primitive is a **forking systemd service unit** owned by the user manager,
independent of `maw.service`.

## Fix

Ship a dedicated user unit `maw-tmux.service` that owns the tmux server.
`maw.service` becomes a plain client that connects to the `-L maw` socket;
it no longer tries to birth the server itself.

### A. New user unit — `deploy/systemd/maw-tmux.service`

```ini
[Unit]
Description=MAW dedicated tmux server
After=default.target

[Service]
Type=forking
# tmux -L maw start-server forks the daemon and exits; PIDFile isn't
# reliable for tmux, so GuessMainPID handles it.
ExecStart=/usr/bin/tmux -L maw start-server
ExecStop=/usr/bin/tmux -L maw kill-server
# tmux daemon double-forks; don't let systemd kill it as if it were the CLI.
RemainAfterExit=yes
# Keep the server alive across maw restarts — only user logout tears it down.
KillMode=mixed
Restart=on-failure
RestartSec=2

[Install]
WantedBy=default.target
```

Install once, per host:

```
mkdir -p ~/.config/systemd/user
cp deploy/systemd/maw-tmux.service ~/.config/systemd/user/
systemctl --user daemon-reload
systemctl --user enable --now maw-tmux.service
```

### B. `maw.service` — declare dependency, soften stop

Edit `~/.config/systemd/user/maw.service` on the host:

```ini
[Unit]
Description=Multi-Agent Workbench
After=default.target maw-tmux.service
Wants=maw-tmux.service

[Service]
Type=simple
WorkingDirectory=/home/maw
EnvironmentFile=/home/maw/.env
Environment=UV_THREADPOOL_SIZE=64
Environment=NODE_ENV=production
ExecStart=/usr/bin/node build/server.js
# Only kill the node main PID. If anything leaks into the cgroup (it shouldn't
# once tmux is out), we still don't collateral-kill it. Belt & braces.
KillMode=process
Restart=on-failure
RestartSec=5
TimeoutStopSec=10

[Install]
WantedBy=default.target
```

Note: `Wants=` not `Requires=` so a broken tmux unit doesn't block maw.
`KillMode=process` is the backup; the primary fix is that tmux is no
longer in this cgroup.

### C. Code — drop `ensureServer()`, add a sanity probe

`src/lib/server/tmux/TmuxSession.ts`:

- Delete `ensureServer()`, `serverRunning()`, `hasSystemdRun()`, and their
  call sites. The systemd unit owns server lifecycle now.
- Keep every `-L maw` call (the socket plumbing in the file today is
  correct and must stay).
- Add `Tmux.assertServerRunning()` — runs `tmux -L maw list-clients` once;
  if it errors with "no server running", log a loud warning pointing to
  the install docs. Don't throw — dev (macOS, no systemd) can still
  auto-spawn tmux on first `new-session`.

`src/lib/server/bootstrap.ts`:

- Replace the `await Tmux.ensureServer()` block with
  `await Tmux.assertServerRunning()`. Keep it in the same slot (before
  `supervisor.init()`).

## Verification

Run on the maw host.

1. **Install the tmux unit.**
   ```
   cp deploy/systemd/maw-tmux.service ~/.config/systemd/user/
   systemctl --user daemon-reload
   systemctl --user enable --now maw-tmux.service
   systemctl --user status maw-tmux.service --no-pager
   ```
   Expect `active (running)`, PID visible, cgroup under
   `…/user@3002.service/app.slice/maw-tmux.service` — **not**
   `maw.service`.

2. **Cgroup check.**
   `cat /proc/$(pgrep -f 'tmux.*-L maw')/cgroup` shows
   `…/maw-tmux.service`.

3. **Update maw.service**, `daemon-reload`, `restart maw`.

4. **Spawn the smoke `shell` agent**, type a few commands. Confirm the
   tmux session appears in `tmux -L maw list-sessions`.

5. **The restart.** `systemctl --user restart maw`. Then:
   - `tmux -L maw list-sessions` → still lists `maw-agent-…`.
   - `journalctl --user -u maw.service -n 40` shows fresh boot with
     `supervisor: 1 agents reattached, 0 crashed`.
   - Reload the dashboard → agent is in **Running**, modal opens with live
     scrollback (reconnect snapshot path from previous plans still works).

6. **Exit detection still works.** Type `exit` in the smoke modal → modal
   closes within ~100 ms, agent moves to archive. (This path is
   orthogonal to the current fix but regressing it would be bad.)

7. **`pnpm check`** clean.
