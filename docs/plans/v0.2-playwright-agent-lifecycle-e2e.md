# v0.2 — Playwright agent-lifecycle e2e test

## Why

Commit `33c4089` fixed a cascading-reap bug in `AgentSupervisor.startExitWatcher`:
the per-session `session-closed` tmux hook signalled the wrong agent's
exit channel, so quitting any one agent reaped every other live agent. The
five unit tests in `TmuxSession.test.ts` pin the fix at the point of bug
(hook is `-g`, uses `run-shell` + `#{hook_session_name}`, channel derived
from session name). What they don't guarantee is that the full chain
holds together:

```
tmux session closes
  → global hook fires with hook_session_name expanded
  → run-shell bounces to `tmux wait-for -S maw-exit-<session>`
  → MAW's wait-for subprocess resolves
  → AgentSupervisor.finishAsExited runs for the correct agent
  → DB row flips to 'exited'
  → WS `state` event + snapshot 410 propagate
  → dashboard stops showing the card
```

A refactor that leaves each unit correct but mis-wires the handoff (e.g.
indexes the wrong runtime from the exit-waiters map, or drifts the
session→channel derivation between `startExitWatcher` and
`ensureGlobalSessionClosedHook`) would sail through the unit tests and
still ship the regression. An e2e test that spawns two shell agents and
closes one of them exercises the whole chain; it'd catch the original
bug *and* generalises to any future "reap the wrong agent" bug.

## What the test does

One test case in a new `tests/e2e/agent-lifecycle.spec.ts` file:

1. **Fixtures** (`beforeAll`):
   - Create an empty host dir at `/tmp/maw-e2e/test-repo` (inside the data
     dir the webServer already nukes pre-start, so cleanup is free)
   - `POST /api/repos` → repo row. MAW auto-`git init`s the empty dir.
   - `POST /api/roles` with `cli_kind: 'shell'` → role row. The `shell`
     adapter is already in `cli-adapters/shell.jsonc` and spawns `bash -i`,
     which responds to a plain `exit` keystroke.
2. **Test body**:
   - `POST /agents/new` twice via form-action, extract each `agentId` from
     the 303 redirect target (`/agents/<id>`)
   - Poll `GET /api/agents/<id>/snapshot` for both — wait for 200 (tmux
     session up and pipe-pane streaming)
   - `tmux -L maw send-keys -t maw-agent-<id2> -l 'exit'` + `Enter` to quit
     the 2nd agent's bash
   - Poll `GET /api/agents/<id2>/snapshot` until it returns **410 Gone**
     (MAW's exit-watcher has run finishAsExited + the snapshot route
     noticed the dead session).
   - **Load-bearing assertion**: read agent 1's `status` column directly
     from `/tmp/maw-e2e/maw.db` (via `better-sqlite3` in readonly mode) and
     assert it's still `'running'`. The DB — not the snapshot route — is
     what catches the cascade regression; see "Why the DB, not the
     snapshot route" below.
   - Double-check at the tmux level: `has-session -t maw-agent-<id1>` still
     succeeds.
3. **Cleanup** (`afterAll`):
   - `tmux -L maw kill-session -t maw-agent-<id>` for every spawned agent,
     swallowing "can't find session" (already reaped). No tmux session leaks
     even if an assertion fails mid-test.

## Architecture choices

### Socket isolation — just reuse `-L maw`

The existing `-L maw` socket is hardcoded in `TmuxSession.ts:20`. Adding
a `MAW_TMUX_SOCKET` env var would be the architecturally pure fix, but
it's scope creep. Observed risk with the shared socket is low:

- **Session-name collisions**: agents are named `maw-agent-<ULID>`. ULIDs
  are monotonically sortable + random, so concurrent dev + test cannot
  collide.
- **Global hook replacement**: both servers install the *same* hook
  string (`set-hook -g … run-shell -b "tmux -L maw wait-for -S
  maw-exit-#{hook_session_name}"`); `set-hook -g` without `-a` is an
  idempotent replace. Harmless.
- **Cross-visibility on `listMawSessions`**: each supervisor only acts on
  sessions whose agent rows live in ITS OWN sqlite DB
  (`listLiveAgents()` → filter to matching session names). Dev's sessions
  are ignored by the test supervisor and vice versa.
- **Test cleanup missing**: `afterAll` handles that. Worst case a crashed
  test leaks a `maw-agent-<ULID>` tmux session; harmless, and
  `tmux -L maw list-sessions` makes it obvious for the developer.

The only real footgun is "don't run `tmux -L maw kill-server` from
something else while the e2e test is running". That's acceptable.

### HTTP path, not UI path

The repo + role creation paths via the UI are finicky to drive from
Playwright (the spawn dialog's nested role-create form uses labels rather
than ids, and the Svelte reactive state doesn't round-trip through raw
`input.value = ...` cleanly). Using the JSON APIs directly via
`page.request` is simpler, faster, and keeps the test focused on the
actual regression (tmux/supervisor chain), not the spawn-form UX.

The **spawn** itself still goes through the real form action
(`POST /agents/new`) because that's the only spawn entry point and it
exercises the worktree creation + supervisor.spawn() path we need.

### Triggering the exit via `tmux send-keys`, not the UI terminal

Same reasoning: the terminal modal's keystroke-forwarding path
(xterm.js → WS `send_keys` → `Tmux.sendLiteral`) is not what's being
tested. Sending the `exit` + `Enter` directly via `tmux send-keys` skips
the UI entirely and produces a deterministic trigger for the server-side
behaviour the test cares about.

### Why `GET /api/agents/:id/snapshot` for agent 2 readiness/exit

The snapshot route returns 410 the moment MAW notices the session is
gone (it also calls `reapAgent` as a side effect, which is idempotent
with the exit-watcher's own `finishAsExited` — whichever wins, the other
short-circuits via the map-delete guard). Polling 200 → 410 is
measurable in tens of milliseconds; waiting for the dashboard to
re-render the same state involves a page invalidate cycle and adds
flake surface for zero coverage gain.

### Why the DB, not the snapshot route, is the regression signal for agent 1

First-draft test asserted agent 1's snapshot stayed `200`. Against the
pre-fix code, that assertion **passed** — so the test failed to catch
the bug it was written to guard.

Reason: the cascade bug only flips agent 1's DB `status` column to
`exited` (and tears down its runtime/FIFO); it does NOT kill agent 1's
tmux session. The snapshot route determines alive/dead by asking tmux
(`Tmux.hasSession`) — it never consults the DB status. So agent 1
shows up as a zombie: DB row says `exited`, tmux says `alive`, snapshot
returns 200.

The DB is the single source of truth for "the supervisor believes this
agent is alive". The test reads it directly with `better-sqlite3` in
readonly mode — cheap, no server interaction, and impossible to fool
with a zombie tmux session. Under the pre-fix code the test now fails
exactly where the bug manifests (`Expected: "running", Received:
"exited"` on agent 1's row).

## Scope: single case only

One test case. No matrix, no parametrisation, no per-CLI-adapter
coverage. The regression characterisation is "close any one agent, the
others stay up"; a single two-agent close-the-second case demonstrates
that at the chain level. Expanding to 3+ agents or multiple adapters
would add runtime without changing the class of bug detected.

## Out of scope

- **Reattach-after-restart behaviour**: the existing periodic reaper in
  `bootstrap.ts` handles this, and it's tangential to the cascade fix.
- **LLM-CLI adapters** (claude-code, codex, gemini): their exit behaviour
  depends on external binaries that may not be installed in CI. The
  `shell` adapter is enough to exercise tmux's session-closed path.
- **`MAW_TMUX_SOCKET` env var**: worth doing later for true concurrent
  dev/test isolation. Not needed to land this test; see the risk analysis
  above.
