# v0.2 — tmux survives restart + working exit detection

> **Superseded by [`v0.2-tmux-survive-restart-v2.md`](v0.2-tmux-survive-restart-v2.md).**
> The `systemd-run --user --scope` approach in section A did not work on the
> production host: scopes die when their main process exits, and
> `tmux start-server` fork-and-exits, so the daemon ended up orphaned into
> `maw.service`'s cgroup anyway. v2 ships a dedicated `maw-tmux.service`
> user unit instead. Section B (exit-watcher hardening) was implemented and
> still applies.


## Context

Two regressions, root-cause linked:

1. **All agents die on `systemctl --user restart maw`.** Originally tmux
   sessions outlived a service restart. They no longer do.
2. **End-of-agent detection is broken.** The terminal modal does not
   auto-close when the CLI exits (e.g. `Ctrl-D` in a shell, `/exit` in
   claude code).

### Root cause #1 — tmux server is in maw's systemd cgroup

`~/.config/systemd/user/maw.service` uses default `KillMode=control-group`.
`Tmux.newSession` (`src/lib/server/tmux/TmuxSession.ts:41`) calls plain
`execa('tmux', ['new-session', '-d', …])`. The auto-daemonized tmux
server lands in maw's cgroup, so `systemctl stop maw` SIGTERMs the tmux
server too. On reboot, `Tmux.listMawSessions()` returns `[]`, so
`AgentSupervisor.init()` (`AgentSupervisor.ts:135-140`) flips every live
row to `crashed`.

This used to work because either (a) the tmux server already existed
from another login session on the default socket, or (b) maw was started
outside systemd. Neither is true on the current production setup.

### Root cause #2 — exit-watcher stale-channel guard masks real exits

`startExitWatcher` (`AgentSupervisor.ts:75-108`) installs a
`session-closed → wait-for -S maw-exit-<id>` hook and forks a blocking
`tmux wait-for` client. When it resolves it calls `Tmux.hasSession`
(line 91) and returns early if the session still exists. This was added
in commit `63576de` to suppress spurious wakeups caused by tmux signals
that survived a maw restart (which only happen because of root cause #1).

Problem: tmux fires the `session-closed` hook **before** the session is
fully reaped from `list-sessions`/`has-session`. The guard therefore
short-circuits legitimate exits, `finishAsExited()` never runs,
`runtime.emit('state', 'exited')` never fires, the WS hub never tells
the client, and the modal stays open. The 5s reaper in `bootstrap.ts:83`
eventually mops up the DB row but the modal is already wedged.

## Fix

### A. Isolate the tmux server from maw's cgroup

Use a dedicated tmux socket and start the server outside the cgroup.

1. **Dedicated socket.** Add `private static SOCKET = 'maw'` to `Tmux`
   and prefix every `execa('tmux', …)` call in
   `src/lib/server/tmux/TmuxSession.ts` with `['-L', Tmux.SOCKET, …]`.
   One helper method `tmuxArgs(args: string[])` keeps it tidy.

2. **Detached server start.** Add `Tmux.ensureServer()` that runs
   `systemd-run --user --scope --quiet --unit=maw-tmux -- tmux -L maw
   start-server` (idempotent — `systemd-run` errors if the unit already
   exists; swallow `Unit maw-tmux.scope already exists`). Call it once
   from `bootstrap.ts` between steps 4 and 5, before `supervisor.init()`.
   The `--scope` puts the tmux server in its own transient scope unit,
   so `systemctl stop maw` does not touch it.

   Fallback when `systemd-run` is unavailable (dev on macOS, CI):
   spawn tmux with `detached: true` + `setsid` via Node's `child_process`
   so it leaves the process group. Detect by trying `systemd-run
   --version`; cache the result.

3. **Systemd unit doc.** Add a note to `README` (or
   `docs/plans/v0.2-…`) recommending `KillMode=process` as a belt-and-
   braces backup. Don't ship a unit file edit — leave that to the user.

### B. Drop the stale-channel guard, harden the watcher

Once (A) lands, channel signals can no longer survive a maw restart
(the tmux server is the same one across restarts, but maw never sees
stale state because the new socket means startExitWatcher always
re-installs the hook on a server that didn't lose it).

1. **Remove the guard** at `AgentSupervisor.ts:91-96`. The watcher
   resolves only when tmux fires `session-closed`, which by definition
   means the session is gone.

2. **Idempotent re-install on reattach.** `setSessionClosedSignal`
   already overwrites; keep it. Also re-arm the channel by calling
   `tmux wait-for -S <channel>` once at boot **only if** the session is
   missing — but with (A) this isn't needed. Skip.

3. **Belt-and-braces.** Wrap the `finishAsExited` call in the watcher
   `then` so it can't double-fire if the periodic reaper gets there
   first (`AgentSupervisor.reap()` and `reapAgent()` already check
   `runtimes.has(agentId)` implicitly via `runtimes.get`; verify the
   delete in `finishAsExited:223` happens before any `await` that could
   yield to the reaper — it currently does not, since
   `runtime.stop()` awaits before `runtimes.delete`. Move
   `this.runtimes.delete(agentId)` and `this.stopExitWatcher(agentId)`
   to the top of `finishAsExited` so concurrent reap paths see it gone
   immediately).

## Files to modify

- `src/lib/server/tmux/TmuxSession.ts` — `-L maw` socket on every call,
  add `ensureServer()`.
- `src/lib/server/bootstrap.ts` — call `Tmux.ensureServer()` before
  `supervisor.init()`.
- `src/lib/server/agents/AgentSupervisor.ts` — drop stale guard at
  L91-96; reorder `finishAsExited` to delete-first.
- `README.md` — document `KillMode=process` recommendation.

## Verification

1. `pnpm check` clean.
2. **Restart survival.** Spawn the smoke `shell` agent, type a few
   commands, `systemctl --user restart maw`, reload the dashboard. The
   agent appears in **Running**, modal opens with live scrollback. From
   the host, `tmux -L maw list-sessions` shows the session both before
   and after the restart.
3. **Exit detection — fast path.** Open the smoke agent modal, type
   `exit` + Enter. Modal auto-closes within ~100 ms; agent moves to the
   archive on the sidebar. Server log shows
   `[AgentSupervisor] exit-watcher fired` (or equivalent) followed by
   the push notification (if enabled).
4. **Exit detection — claude-code.** Same with a real claude-code
   agent: `/exit` inside the CLI must close the modal without waiting
   for the 5 s reaper.
5. **No double-reap.** Kill an agent via the kill button while the
   watcher is armed; check logs for no "stop failed" / no duplicate
   `finishAsExited` calls.
