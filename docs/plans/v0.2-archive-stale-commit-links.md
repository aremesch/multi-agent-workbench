# Fix stale commit links in archive after history rewrites

## Context

The archive page at `/repos/[id]/archive` renders per-commit links as
`{remote.webBase}/commit/{sha}`. Commit SHAs are snapshotted once at agent
exit into the `agent_commits` table (migration 006) and never revisited.
If the user later rebases, squashes, force-pushes, or resolves conflicts
in a way that recreates the commit, the stored SHA no longer matches the
upstream SHA — the link 404s.

Concrete case that prompted this fix: agent committed under
`96534ce...`. The user later merged that work into `development` by
recommitting (single-parent commit whose message says "Integrates
maw/... (commit 96534ce)"), producing `a82cb199...`. `96534ce` is gone
from the local object DB (`git cat-file -e` fails) and from the remote.
The archive page still links to the dead SHA because:

- `snapshotAgentCommits` only runs at `finishAsExited` / `kill` and on
  one-shot legacy back-fill.
- There is no user-visible mechanism to re-snapshot.
- There is no detection of "stored SHA no longer reachable".
- `replaceAgentCommits` is destructive (DELETE+INSERT), so a naive
  auto-refresh risks wiping historical rows when the branch has been
  deleted post-merge.

Intended outcome: users can recover working links after a rebase/merge,
and the system never silently destroys attribution metadata.

## Approach

Three-part fix, kept deliberately small:

1. **User-triggered refresh** — per-row "Refresh commits" button on the
   archive page that calls the existing `POST /api/agents/[id]/commits`.
2. **Non-destructive refresh semantics** — when the new snapshot would
   be empty AND both the agent branch and `base_sha` are unreachable,
   preserve existing rows instead of wiping them.
3. **Reachability annotation** — at archive load time, batch-check each
   stored SHA with `git cat-file --batch-check`; mark unreachable ones
   in the UI with a muted/strikethrough style and a tooltip. Keep the
   link clickable (the upstream may still have it under the same SHA
   even when the local copy does not).

Out of scope: patch-id tracking, Gitea/GitHub REST lookups, cross-repo
SHA translation, automatic refresh on page load, auto-retry. Those are
viable follow-ups but too ambitious for now.

## Changes

### `src/lib/server/git/agentCommits.ts`

Add three small helpers (all `-C <repoPath>` shelling out to git):

- `revParseQuiet(repoPath, ref): Promise<boolean>` —
  `git rev-parse --verify --quiet <ref>`.
- `catFileExists(repoPath, sha): Promise<boolean>` — `git cat-file -e <sha>`.
- `checkShaReachability(repoPath, shas: string[]): Promise<Set<string>>` —
  single `git cat-file --batch-check=%(objectname)` fed all SHAs on
  stdin; returns the subset that resolved (lines not containing
  " missing"). One process call per repo, not per SHA.

### `src/lib/server/git/commitSnapshot.ts`

Extend the result union:

```ts
export type SnapshotResult =
  | { captured: number; source: AgentCommitSource }
  | { preserved: number; reason: 'branch-missing' | 'base-missing' | 'empty-unreachable' }
  | { error: string };
```

Before calling `replaceAgentCommits`:

```ts
if (commits.length === 0) {
  const existing = listPersistedAgentCommits(agent.id);
  const branchExists = await revParseQuiet(repo.path, `refs/heads/${wt.branch}`);
  const baseExists = agent.base_sha
    ? await catFileExists(repo.path, agent.base_sha)
    : false;
  const anchorsGone = !branchExists && (!agent.base_sha || !baseExists);
  if (existing.length > 0 && anchorsGone) {
    updateAgentCommitSnapshot(agent.id, null, Math.floor(Date.now() / 1000));
    return { preserved: existing.length, reason: 'empty-unreachable' };
  }
}
replaceAgentCommits(...);
updateAgentCommitSnapshot(...);
return { captured: commits.length, source };
```

Genuine "agent ran but made no commits" stays destructive (branch still
exists, empty is correct). Only the anchors-gone case preserves.

### `src/routes/api/agents/[id]/commits/+server.ts`

Pass `preserved`/`reason` from `SnapshotResult` through to the JSON
response so the UI can show a distinct toast.

### `src/routes/repos/[id]/archive/+page.server.ts`

- After building `rows` for all agents, collect every `sha` across all
  agents and call `checkShaReachability(repo.path, allShas)` **once**
  (batch across agents; one git call per archive load).
- `rowToCommit(row, reachableSet)` sets `reachable = reachableSet.has(row.sha)`.

### `src/routes/repos/[id]/archive/+page.svelte`

- Refresh button in the actions column (next to View logs, before Delete):

  ```ts
  let refreshing = $state<Record<string, boolean>>({});
  async function refreshCommits(agentId: string): Promise<void> {
    refreshing[agentId] = true;
    try {
      const res = await apiFetch(`/api/agents/${agentId}/commits`, { method: 'POST' });
      // TODO(toast): surface preserved/reason/error from body
      if (res.ok) await invalidateAll();
    } finally {
      refreshing[agentId] = false;
    }
  }
  ```

- Commit SHA anchor gets `class:stale={!c.reachable}` plus
  `title={!c.reachable ? t('archive.commit.stale') : ...}`. Still an

- CSS: `.sha.stale { opacity: .55; text-decoration: line-through dotted; }`.

## Verification

1. `pnpm check` clean.
2. `pnpm test` — new unit tests pass; coverage thresholds hold.
3. On the live instance:
   - Pick an archived agent whose commits have been rewritten upstream
     (e.g. the one where `96534ce` is stored but upstream is
     `a82cb199`).
   - Visit `/repos/<id>/archive` — the stale SHA should render with
     the `.stale` style and tooltip; the link should still be
     clickable.
   - Click **Refresh commits** — if the rewritten commit is
     discoverable by committer-email (agents with preserved
     attribution), the row updates to the live SHA. If anchors are
     gone and the committer filter finds nothing, the toast reads
     "Kept existing commits; branch or base is gone."
   - Visit a non-stale archive — no visual change, refresh is a no-op
     that reports `{ captured: N }`.
