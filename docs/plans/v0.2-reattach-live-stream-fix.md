# Fix: live terminal stream dead after `systemctl --user restart maw`

## Context

Agent `wrong-git-assignments` (tmux session `maw-agent-01KPQDZH1W8TWEN9JZKRT34DYZ`) is the live reproducer. After `systemctl --user restart maw`:

- tmux session survives (runs under separate `maw-tmux.service`). ✓
- Dashboard thumbnail keeps updating (uses `tmux capture-pane`, not the FIFO). ✓
- Reopening the terminal modal shows a fresh `capture-pane` snapshot — including keystrokes typed after the restart — so tmux DID receive the input. ✓
- But in the open modal: new keystrokes don't echo, host output doesn't stream. ✗

Root cause (confirmed empirically on the live system):

1. `maw.service` has `TimeoutStopSec=5`. SIGTERM handler in `server.js:72-81` does NOT call `supervisor.shutdown()`, so `rt.stop()` / `Tmux.stopPipePane` / `FifoStreamer.stop` never fire. The old maw process is SIGKILL'd 5 s later, leaving:
   - stale FIFO file on disk (two already orphaned under `/tmp/maw-fifos/`),
   - `cat >> fifo` subprocess (tmux's child) still alive on the tmux side whenever it was idle on stdin during the shutdown window — so tmux still has `wp->pipe_fd != -1`.
2. On boot, `AgentSupervisor.init()` reattaches: `AgentRuntime.start()` calls `Tmux.pipePane()` which runs `tmux pipe-pane -o -t <session> 'cat >> <fifo>'`.
3. The `-o` flag is a TOGGLE, not "replace cleanly" as the docstring claims. Per tmux's `cmd-pipe-pane.c`: if an old pipe exists, destroy it AND return without opening a new one. So on the survived-`cat` path, reattach kills the old pipe and leaves `pane_pipe=0` — pane output is piped to nothing, forever.

Live evidence (while the bug is active, before the fix):

```
$ tmux -L maw display -p -t maw-agent-01KPQDZH1W8TWEN9JZKRT34DYZ '#{pane_pipe}'
0
$ lsof /tmp/maw-fifos/fifo-01KPQDZH1W8TWEN9JZKRT34DYZ

$ ps -eo pid,cmd | grep -E 'cat.*fifo' | grep -v grep
(nothing)

Modify: 2026-04-21 20:36:49 (14 minutes BEFORE the SIGTERM at 20:51:12)
```

Node has the FIFO open as reader. tmux is not piping. The pipe was toggled off on reattach and never re-opened.

## The fix

Minimal, two-part change. Both are small and isolated.

### Part A — Remove the `-o` flag in `Tmux.pipePane`

`src/lib/server/tmux/TmuxSession.ts:119-128`

Change:

```ts
/** `tmux pipe-pane -o 'cat >> <fifo>'` — open-ended (-o) so restarts replace cleanly. */
static async pipePane(session: string, fifoPath: string): Promise<void> {
  await execa('tmux', t([
    'pipe-pane',
    '-o',
    '-t',
    session,
    `cat >> ${JSON.stringify(fifoPath)}`
  ]));
}
```

to:

```ts
/**
 * `tmux pipe-pane 'cat >> <fifo>'` — no `-o`.
 *
 * Default (no `-o`) semantics destroy any existing pipe and open a new
 * one. That is what reattach wants: if the previous maw process was
 * SIGKILL'd before it could call `stopPipePane`, tmux may still have
 * `wp->pipe_fd != -1` from a surviving `cat` child. With `-o` that
 * becomes a TOGGLE-OFF and the new pipe is never opened — pane output
 * goes to /dev/null, live terminal stream silently dies.
 *
 * See `docs/plans/i-have-my-project-parallel-crystal.md` for the full
 * post-mortem.
 */
static async pipePane(session: string, fifoPath: string): Promise<void> {
  await execa('tmux', t([
    'pipe-pane',
    '-t',
    session,
    `cat >> ${JSON.stringify(fifoPath)}`
  ]));
}
```

`stopPipePane` is unchanged. `pipePane` is only called from `AgentRuntime.start()` (grep-confirmed), so removing `-o` cannot regress any unrelated caller.

### Part B — Defensive fresh FIFO in `FifoStreamer.create`

`src/lib/server/tmux/FifoStreamer.ts:54-60`

Change:

```ts
async create(): Promise<void> {
  mkdirSync(dirname(this.path), { recursive: true });
  if (!existsSync(this.path)) {
    await execa('mkfifo', [this.path]);
  }
}
```

to:

```ts
async create(): Promise<void> {
  mkdirSync(dirname(this.path), { recursive: true });
  // Always start from a fresh FIFO: if a previous maw process was
  // SIGKILL'd before it could call stop(), the stale FIFO on disk may
  // still have a surviving tmux-side `cat` writer bound to it. Replacing
  // the inode decouples the new reader from any stray state.
  if (existsSync(this.path)) {
    try { unlinkSync(this.path); } catch { /* ignore — racing cleanup is fine */ }
  }
  await execa('mkfifo', [this.path]);
}
```

Already-imported symbols (`existsSync`, `unlinkSync`) at FifoStreamer.ts:33. Side-effect: unrelated orphan FIFOs elsewhere in the dir are untouched (we only replace our own path).

**Layer 2 — real-tmux integration test.** New file `src/lib/server/tmux/TmuxSession.reattach.integration.test.ts`. This is the regression-proofing test the user specifically asked for. It runs real tmux on a throwaway socket, reproduces the live bug directly, and fails loudly if `-o` ever comes back through any code path.

Shape:

```ts
import { describe, it, expect, beforeAll, afterAll } from 'vitest';
import { execa } from 'execa';
import { mkdtempSync, rmSync, openSync, closeSync, readSync } from 'node:fs';
import { tmpdir } from 'node:os';
import { join } from 'node:path';

const SOCK = `maw-test-${process.pid}`;
const t = (...args: string[]) => execa('tmux', ['-L', SOCK, ...args]);

// Skip when tmux isn't on PATH — honours dev laptops without tmux AND
// CI images that choose not to install it. CI (Ubuntu) has tmux.
const hasTmux = await execa('which', ['tmux']).then(() => true, () => false);
const d = hasTmux ? describe : describe.skip;

d('Tmux.pipePane — reattach robustness (real tmux)', () => {
  let fifoDir: string;
  const session = 'maw-agent-reattach-test';

  beforeAll(async () => {
    fifoDir = mkdtempSync(join(tmpdir(), 'maw-fifo-test-'));
    await t('-f', '/dev/null', 'new-session', '-d', '-s', session, 'bash');
  });

  afterAll(async () => {
    await t('kill-server').catch(() => {});
    rmSync(fifoDir, { recursive: true, force: true });
  });

  it('back-to-back pipePane calls leave the pane with an active pipe (no toggle)', async () => {
    // Use the real Tmux wrapper but point it at the test socket. This
    // import path forces a fresh module with our SOCK env — done via an
    // internal harness helper so we don't mutate the real SOCKET const.
    //
    // If the -o flag is ever re-introduced, the second pipePane call
    // toggles the pipe OFF and pane_pipe comes back '0'.
    const fifo = join(fifoDir, 'reattach.fifo');
    await execa('mkfifo', [fifo]);

    // Keep a reader open (O_RDWR) so `cat` never SIGPIPEs between calls.
    const rfd = openSync(fifo, 'r+');

    await t('pipe-pane', '-t', session, `cat >> "${fifo}"`);
    const first = await t('display', '-p', '-t', session, '#{pane_pipe}');
    expect(first.stdout.trim()).toBe('1');

    // Simulate a restart reattach: second pipePane with the OLD pipe
    // still live (we never called stopPipePane).
    await t('pipe-pane', '-t', session, `cat >> "${fifo}"`);
    const second = await t('display', '-p', '-t', session, '#{pane_pipe}');
    expect(second.stdout.trim()).toBe('1');          // ← fails under `-o`

    // Data actually flows: poke the shell prompt and read a byte.
    await t('send-keys', '-t', session, 'echo hi', 'Enter');
    await new Promise((r) => setTimeout(r, 200));
    const buf = Buffer.alloc(4096);
    const n = readSync(rfd, buf, 0, buf.length, null);
    expect(n).toBeGreaterThan(0);
    expect(buf.slice(0, n).toString()).toMatch(/hi/);

    closeSync(rfd);
  });
});
```

CI note: `.github/workflows/` already has a vitest workflow. Ubuntu runners include `tmux` by default; if not, `apt-get install -y tmux` in the workflow's setup step (a one-line addition in the existing Ubuntu image). The `describe.skip` guard keeps the test from breaking dev machines without tmux installed.

Failure-signal test: before committing the fix, run this integration test against the pre-fix `-o` code and confirm it fails at `expect(second.stdout.trim()).toBe('1')` with `Received: "0"`. That's how we know the test actually guards the bug.

## Explicitly NOT in this change

- Wiring `supervisor.shutdown()` into the SIGTERM handler in `server.js`. The current omission is deliberate — see the comment at `server.js:65-71`. With parts A + B, reattach is robust against ungraceful shutdown, so tightening shutdown hygiene is no longer load-bearing and would risk a different regression class (stopPipePane races with tmux SIGTERM under `KillMode=control-group`).

- **The 5 s SIGKILL delay on restart.** journalctl shows `[maw] received SIGTERM` at T+0 and systemd `Killing process … with signal SIGKILL` at T+5 s — meaning the 2 s `setTimeout(…, 2000).unref()` fallback in `server.js:79` is failing to force-exit the process within its window, so systemd has to escalate to SIGKILL at `TimeoutStopSec=5`. **Impact is UX-only**: every `systemctl restart maw` takes ~5 s instead of <2 s, and feels janky; but the broken state on disk (stale FIFO, possibly-surviving `cat` child) is identical whether we exited at T+2 s or T+5 s. With parts A + B of this fix, reattach is idempotent against both paths, so whatever is preventing the fallback timer from firing stops mattering for correctness. If we ever want the cosmetic fix, it belongs in a separate patch with its own test that measures actual exit time under real load; it is not load-bearing for this bug.

- Any change to adapter logic, WS hub, or history snapshot path — all of those are working correctly.
