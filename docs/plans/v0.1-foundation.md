# Multi-Agent Workbench (MAW)

## Context

You want a self-hosted web "workbench" to orchestrate multiple LLM coding-agent CLIs (Claude Code, Codex, Gemini CLI) in parallel across one or more git repos, with pre-configured roles, inter-agent supervision, and 24/7 mobile access via a PWA. The two driving goals are: **(1)** managing multi-agent dev work from your phone wherever you are, and **(2)** an everyday browser workbench you can log into and resume seamlessly.

The directory `/home/ar/smb/ar/workspace/multi-agent-workbench` is empty — this is greenfield. The plan below is the locked-in design after exploring trade-offs.

## Locked-in decisions

| Area | Choice | Reason |
|---|---|---|
| Stack | **SvelteKit fullstack** + TypeScript (strict) | One framework end-to-end. `$lib/server/*` keeps server-only code out of client bundles. |
| Runtime | Node.js 22 LTS via `@sveltejs/adapter-node` | Mature, great child_process / fs, native `ws` |
| Custom server entry | `server.js` wrapping the adapter-node handler + `ws` server on the same HTTP listener | Lets us mount WebSockets cleanly and own the boot sequence (singletons, FIFO cleanup, reattach) |
| Singleton bootstrap | `src/hooks.server.ts` with module-level lazy init | Canonical SvelteKit pattern; runs once per server boot |
| WebSocket | **raw `ws`** attached to the same HTTP server in `server.js` | Low overhead, simple framing for byte streams. In dev, use a tiny Vite plugin (`vite-plugin-svelte-server`-style hook) or hit a sibling port — covered in §6a |
| Database | **`better-sqlite3` directly, no ORM** | Single-host, single-user; sync API; ~13 simple tables don't need an abstraction layer |
| Migrations | Hand-written `migrations/NNN_*.sql` files run on boot via a tiny migrator (~30 LOC) that tracks applied versions in a `_schema_migrations` table | Less magic than Drizzle, easy to read/diff/review |
| Type safety for rows | Hand-maintained TS row interfaces in `src/lib/server/db/types.ts` + typed query helper functions in `queries.ts` | You write SQL, you write types, they meet at one parameterized helper per query |
| UI kit | Tailwind + **shadcn-svelte** + bits-ui | Mobile-first touch components, no runtime lock-in |
| Terminal view | `@xterm/xterm` + `xterm-addon-fit` + `xterm-addon-webgl` | De-facto browser terminal |
| State (fe) | `@tanstack/svelte-query` for server state, Svelte stores for local | Cache + reconnect handling for WS-backed data |
| PWA | `@vite-pwa/sveltekit` (Workbox) | First-class SvelteKit integration, generates SW + manifest |
| Push | `web-push` (VAPID) | Spec-compliant, works on iOS 16.4+ and Android |
| Auth | Argon2id (`@node-rs/argon2`) password + signed httpOnly session cookie set in a `+layout.server.ts`/hooks pair | Strong defaults; passkey/WebAuthn deferred to v0.4 |
| CSRF | SvelteKit's built-in form-action CSRF + double-submit token on JSON `+server.ts` mutating endpoints | Native to SvelteKit |
| Git | Shell out to `git` via `execa` | Worktrees, dirty states, and edge cases are easier to debug at the CLI level |
| Tmux | Shell out to `tmux` via `execa`; pipe-pane → FIFO; send-keys for input | No `libtmux` for Node — own a thin TS wrapper. Boring beats `tmux -CC` for one developer |
| LLM oversight | `@anthropic-ai/sdk` (+ optional `openai`, `@google/generative-ai`) | Use whichever key the user configured |

| Process supervision | systemd unit on the user's host | Auto-restart, journal logs |
| Testing | Vitest + Playwright | Unit/integration + one PWA smoke test |
| Multi-user | **Single user now, schema carries `user_id` everywhere** | Future migration, not a rewrite |

## Architecture overview

A single SvelteKit fullstack app, served by `adapter-node` behind a thin custom `server.js` that also mounts a `ws` WebSocket server on the same HTTP listener. On boot, `hooks.server.ts` lazy-initializes the **AgentSupervisor** (singleton), which reads SQLite, scans tmux for `maw-agent-*` sessions, and reattaches to any that survived. Each live agent has an **AgentRuntime** that owns: a `pipe-pane`-fed FIFO reader, a per-CLI **adapter** state machine (Claude Code / Codex / Gemini), a serialized input queue → `tmux send-keys`, an append-only `terminal_log` writer, and a fan-out `EventEmitter` that any connected WebSocket subscribes to. SvelteKit `+server.ts` routes handle REST (auth, CRUD, push subscriptions); WebSocket handles realtime (terminal stream, input, alerts). An alert pipeline watches adapter events and delivers Web Push to your phone. Because all state lives in tmux + SQLite, the backend can crash, redeploy, or upgrade and **reattach to existing agents on boot** without losing them.

## Data model (raw SQLite, hand-written migrations)

Every table has `id TEXT PRIMARY KEY` (ULID), `created_at INTEGER NOT NULL`, `updated_at INTEGER NOT NULL`, and `user_id TEXT NOT NULL` (forward-compat). Times stored as Unix epoch seconds. Migrations live in `migrations/NNN_*.sql`; the migrator runs them in order on boot, tracking applied versions in `_schema_migrations(version, applied_at)`.

- **users** `(id, username, password_hash)` — single row in v0.1.
- **sessions** `(id, user_id, expires_at, user_agent)`
- **projects** `(id, user_id, name, default_branch)`
- **repos** `(id, project_id, path, origin_url)`
- **worktrees** `(id, repo_id, path, branch, status)` — `status in ('active','orphaned','removed')`
- **roles** `(id, user_id, name, system_prompt, cli_kind, default_args_json, tool_config_json, repo_scope_json)`
- **agents** `(id, user_id, role_id, repo_id, worktree_id, cli_kind, tmux_session, status, last_attention_at, current_task_id)` — `status in ('spawning','running','waiting_input','idle','exited','crashed')`
- **agent_runs** `(id, agent_id, started_at, ended_at, exit_code, reason)`
- **tasks** `(id, agent_id, title, body, status, assigned_by_agent_id, completed_at)` — `status in ('queued','active','done','cancelled')`
- **terminal_log** `(id, agent_id, seq, ts, chunk BLOB)` — append-only, pruned past N MB per agent
- **events** `(id, agent_id, kind, payload_json, ts)` — adapter events (`prompt_detected`, `task_done`, `stderr_burst`, …)
- **messages** `(id, from_agent_id, to_agent_id, kind, body, read_at, ts)` — inter-agent bus
- **alerts** `(id, agent_id, severity, reason, payload_json, ts, acknowledged_at)`
- **push_subscriptions** `(id, user_id, endpoint, p256dh, auth, ua)`
- **llm_oversight_verdicts** `(id, agent_id, verdict, rationale, model, tokens_in, tokens_out, ts)`

Indexes: `(agent_id, seq)` on `terminal_log`, `(to_agent_id, read_at)` on `messages`, `(agent_id, ts)` on `events`.

**Query layer pattern** — No ORM. Each query is a small typed function:

```ts
// src/lib/server/db/types.ts
export interface AgentRow {
  id: string; user_id: string; role_id: string; repo_id: string;
  worktree_id: string; cli_kind: 'claude-code'|'codex'|'gemini';
  tmux_session: string; status: AgentStatus;
  last_attention_at: number | null; current_task_id: string | null;
  created_at: number; updated_at: number;
}

// src/lib/server/db/queries.ts
import { db } from './index';
const _getAgent = db.prepare<[string], AgentRow>('SELECT * FROM agents WHERE id = ?');
export const getAgent = (id: string): AgentRow | undefined => _getAgent.get(id);

const _listLiveAgents = db.prepare<[], AgentRow>(
  `SELECT * FROM agents WHERE status IN ('running','waiting_input','idle','spawning')`
);
export const listLiveAgents = (): AgentRow[] => _listLiveAgents.all();
```

`better-sqlite3`'s prepared statements are typed via generics so `get`/`all`/`run` return your row interface. Queries are prepared once at module load (singleton-friendly). This gives you ~95% of an ORM's type safety with zero abstraction overhead.

### Drizzle migration readiness (rules to follow from day 1)

We start raw but want to keep the door open to switching to Drizzle around v0.3 if the manual sync becomes painful. To make that migration a couple-hour job rather than a refactor, follow these rules from the first commit:

1. **Single funnel**: `src/lib/server/db/queries.ts` is the **only** file in the entire codebase that imports `better-sqlite3` or builds SQL strings. No exceptions — not in routes, not in the supervisor, not in adapters, not in tests. Routes call `getAgent(id)`, never `db.prepare(...)`.
2. **Function-shaped API**: every query is exposed as a named exported function with explicit parameter and return types (`getAgent(id: string): AgentRow | undefined`). Drizzle migration rewrites function *bodies*; signatures stay identical so callers don't change.
3. **Row types live in `types.ts`, queries import from there**: when we switch to Drizzle, `types.ts` becomes a re-export of `InferSelectModel<typeof agents>` etc. Everything downstream still imports the same names.
4. **Migrations are pure SQL files** in `migrations/NNN_*.sql` — no JS/TS migration scripts, no programmatic schema mutations. Drizzle's migration format is also raw SQL files; the historical files become Drizzle's history with one rename pass on the tracking table.
5. **Migration tracking table is named `__drizzle_migrations`** (Drizzle's default) from day 1, with columns `id INTEGER PRIMARY KEY, hash TEXT NOT NULL, created_at INTEGER NOT NULL`. Our boot-time migrator writes this format, so switchover is zero-friction.
6. **No hand-rolled joins returned as flat rows** with renamed columns (e.g. `SELECT a.id AS agent_id, r.id AS role_id`). If a query needs joined data, return two separate calls or a `{agent: AgentRow, role: RoleRow}` shape — that's how Drizzle's relational queries shape results, so migrating later is mechanical.
7. **No raw SQL inside transactions outside `queries.ts`**. Transactions are exposed as wrapper helpers like `withTx(fn)` that take a callback receiving the same query helpers. Drizzle's `db.transaction(tx => ...)` slots in identically.
8. **No SQLite-specific JSON1 trickery** (`json_extract`, `json_set` in queries) unless absolutely necessary. Drizzle supports it but the API differs; sticking to plain columns or stringified JSON blobs decoded in TS keeps the migration trivial. The columns we already mark as `_json` (`tool_config_json`, `repo_scope_json`, etc.) are read as strings and parsed in TS — this is the right pattern.
9. **No dynamic SQL construction** (`'SELECT * FROM ' + table`). Every query is a fixed prepared statement. Drizzle's query builder is composable but our migration is dramatically simpler if there's nothing dynamic to translate.
10. **Tests hit `queries.ts`, never the DB directly**. This means test fixtures don't need rewriting on the switchover.

A `// TODO(drizzle): ...` comment convention can flag any spot where we deliberately took a raw-SQL shortcut that will need attention during migration — but the goal is zero such comments.

## CLI adapter interface (the contract)

```ts
// src/lib/shared/adapterTypes.ts
export type CliKind = string; // free-form: 'claude-code', 'codex', 'gemini', or any new tool

export interface SpawnSpec {
  command: string;
  args: string[];
  env: Record<string, string>;
  cwd: string;                  // worktree path
}

export interface AdapterEvent {
  kind: 'ready' | 'working' | 'prompt_detected' | 'task_done' | 'error' | 'exited';
  at: number;
  patternId?: string;           // which config pattern fired
  choices?: string[];           // for prompt_detected
  detail?: Record<string, unknown>;
  raw?: string;
}

export interface InputEncoding {
  encode(text: string): string[];                          // ordered keystrokes
  answerPrompt(choice: string | number): string[];        // for active permission prompts
}

export interface CliAdapter {
  kind: CliKind;
  displayName: string;
  buildSpawnSpec(opts: {
    role: { systemPrompt: string; toolConfig: unknown };
    worktreeCwd: string;
    task: { title: string; body: string } | null;
  }): SpawnSpec;
  ingest(chunk: Buffer): AdapterEvent[];   // stateful state machine
  input: InputEncoding;
  isIdleWaiting(state: AdapterRuntimeState): boolean;
}
```

There is **one default implementation**: `ConfigDrivenAdapter`. It's instantiated from a JSONC config file at boot. The Claude Code, Codex, and Gemini "adapters" are config files, not TypeScript classes. Custom TS adapters remain available as an escape hatch (`CliAdapter` is the interface, `ConfigDrivenAdapter` is one concrete implementation).

## Config-driven CLI adapters (the extensibility story)

**Goal**: adding a new CLI tool = drop a new JSONC file in `cli-adapters/`. No recompile, no code changes. Hot-reloaded in dev, watched in prod.

**Format**: JSONC (JSON-with-comments) validated by a Zod schema (`adapter.config.schema.ts`) at load time. Validation errors point to the file with line/column and a human message.

**Example**: `cli-adapters/claude-code.jsonc`

```jsonc
{
  "$schema": "../schemas/adapter.schema.json",
  "kind": "claude-code",
  "displayName": "Claude Code",

  // How to launch the CLI inside its tmux pane.
  // Variables: {{worktree}}, {{role.systemPrompt}}, {{role.toolConfig}}, {{task.title}}, {{task.body}}, {{env.X}}
  "spawn": {
    "command": "claude",
    "args": ["--dangerously-skip-permissions"],
    "env": {
      "ANTHROPIC_API_KEY": "{{env.ANTHROPIC_API_KEY}}"
    },
    // Optional: text typed into the pane after the CLI is ready, instead of CLI args
    "initialInput": "{{task.body}}"
  },

  // How user input becomes tmux keystrokes
  "input": {
    "encoding": "literal",        // -> tmux send-keys -l -- "<text>"
    "submitKey": "Enter",
    // Named answers used by `answer_prompt {choice}` from the WS protocol
    "promptAnswers": {
      "yes":   ["y", "Enter"],
      "no":    ["n", "Enter"],
      "1":     ["1", "Enter"],
      "2":     ["2", "Enter"],
      "abort": ["C-c"]
    }
  },

  // Patterns matched against an ANSI-stripped rolling 8KB tail buffer.
  // Order matters: most-specific first. First match per scan wins per pattern kind.
  "patterns": [
    {
      "id": "tool_permission_prompt",
      "kind": "prompt_detected",
      // Named groups become AdapterEvent.detail (e.g. detail.tool, detail.cmd) and feed alert payloads
      "regex": "Do you want to (?:run|proceed with) (?<tool>[a-zA-Z_]+)(?:\\((?<args>[^)]*)\\))?\\?",
      "flags": "m",
      "scope": "tail",            // 'tail' (whole buffer) | 'tail_line' (last non-empty line)
      "choices": ["yes", "no"],
      "severity": "warning",
      "description": "Standard tool-use permission prompt"
    },
    {
      "id": "shell_command_prompt",
      "kind": "prompt_detected",
      "regex": "Run (?<cmd>.+?) \\? \\[y/N\\]",
      "scope": "tail",
      "choices": ["yes", "no"],
      "severity": "warning"
    },
    {
      "id": "ready",
      "kind": "ready",
      "regex": "^>\\s*$",
      "scope": "tail_line"
    },
    {
      "id": "task_done",
      "kind": "task_done",
      "regex": "(?:✓ Task complete|All done!)",
      "scope": "tail",
      "severity": "info"
    },
    {
      "id": "rate_limit",
      "kind": "error",
      "regex": "rate.limit|quota exceeded",
      "flags": "i",
      "scope": "tail",
      "severity": "error"
    }
  ],

  // How to detect "agent is sitting waiting for user, not working"
  "idleDetection": {
    "method": "cursor_at_prompt",   // 'cursor_at_prompt' | 'inactivity'
    "promptLineRegex": "^>\\s*$",
    "inactivityMs": 2000             // also fires if no output for this long after WORKING
  },

  // Optional defaults — overridable per role
  "defaults": {
    "autoAnswer": [
      // { "patternId": "tool_permission_prompt", "when": { "tool": "Read" }, "answer": "yes" }
    ]
  }
}
```

**The `ConfigDrivenAdapter` engine** (`src/lib/server/agents/adapters/ConfigDrivenAdapter.ts`) implements the `CliAdapter` interface by:

1. Holding compiled `RegExp` objects for each pattern (compiled once at load).
2. Maintaining a rolling tail buffer (default 8KB) of ANSI-stripped output via `strip-ansi`.
3. On every `ingest(chunk)` call: append, strip, slide, scan patterns top-to-bottom, emit `AdapterEvent`s with `patternId`, `choices`, and `detail` from named capture groups.
4. Tracking an explicit state-machine enum (`BOOTING → READY → WORKING → WAITING_PROMPT → …`) driven by which pattern kinds fire.
5. `buildSpawnSpec` interpolates `{{...}}` template variables with `Mustache`-style replacement (no logic, just substitution), then returns the resolved `SpawnSpec`.
6. `input.encode` and `input.answerPrompt` consult the config's `input` block.

**Loading and discovery** — at boot, `bootstrap.ts` scans `cli-adapters/*.jsonc`, validates each via Zod, and registers them in an `AdapterRegistry`. Each agent's `cli_kind` column matches a registered adapter `kind`. Unknown kinds in the DB → agent marked `crashed` with a clear error message in `events`.

**Hot reload** — `chokidar` watches `cli-adapters/` in dev and prod. On change, the file is re-validated. If valid, the registry is updated atomically; **already-running agents keep their old config snapshot** (so an in-flight prompt doesn't get reinterpreted), but new agents get the new config. This avoids re-attaching mid-prompt with new regexes.

**Auto-answer rules** — defined per config and overridable per role. Format: `{patternId, when?: {captureGroup: value}, answer: choice}`. Critical: auto-answer is opt-in per pattern, with a hard rule that destructive commands (`rm -rf`, `DROP TABLE`, etc.) cannot be auto-answered — the engine maintains a denylist of regex fragments that block auto-answer regardless of config.

**Custom TS adapter escape hatch** — if a CLI is too weird for the declarative format (e.g., needs binary protocol parsing, MCP control mode, unusual terminal redraws), drop a `*.adapter.ts` file in `src/lib/server/agents/adapters/custom/` exporting a `CliAdapter` instance. Boot scans this directory too. The interface stays the same so the rest of the system doesn't care which kind it's talking to.

**Why JSONC, not YAML/TOML/XML**:
- JSONC parses with `jsonc-parser` (~10KB) and emits position info for nice error messages.
- Regex strings in JSON need `\\` escaping but everything else is unambiguous; YAML's string-vs-number-vs-bool coercion has bitten too many configs.
- Editors highlight `.jsonc` natively and the `$schema` link gives free autocomplete in VS Code.
- Easy to generate or modify programmatically (e.g., a future "import from a recording" tool).

**Validating regexes at load time** — in addition to Zod schema validation, the loader compiles each regex with `new RegExp(...)` inside a try/catch, so malformed patterns fail at boot with a clear error (file + pattern id + the JS error message), not at runtime when the agent is mid-task.

**Pattern testing** — a tiny CLI script `pnpm test:adapter <config> <recording.txt>` replays a captured raw terminal session against an adapter config and prints which patterns fired in order. Used for tuning new CLIs against real recordings before they go live. Included as part of v0.1 because it's cheap and saves enormous debugging time.

## SvelteKit + WebSocket + singleton bootstrap

This is the trickiest piece of going fullstack-SvelteKit, so it's worth nailing down concretely.

**Production (`server.js`)** — adapter-node exports a Node `handler`. We wrap it with our own HTTP server so we can also attach `ws` to the same listener:

```js
// server.js
import { handler } from './build/handler.js';
import { WebSocketServer } from 'ws';
import http from 'http';
import { bootstrap, getWsHub } from './build/server/lib/server/bootstrap.js';

await bootstrap(); // starts AgentSupervisor, MCP server, alert pipeline, etc.

const server = http.createServer(handler);
const wss = new WebSocketServer({ noServer: true });

server.on('upgrade', (req, socket, head) => {
  if (new URL(req.url, 'http://x').pathname !== '/ws') return socket.destroy();
  // session cookie check
  wss.handleUpgrade(req, socket, head, (ws) => getWsHub().attach(ws, req));
});

server.listen(process.env.PORT ?? 3000, '127.0.0.1');
```

**Dev** — SvelteKit's dev server (Vite) doesn't surface its HTTP server for WS upgrades cleanly. The clean fix is a tiny Vite plugin in `vite.config.ts` that hooks `configureServer` and attaches the same `WebSocketServer` to `server.httpServer`:

```ts
// vite.config.ts
import { sveltekit } from '@sveltejs/kit/vite';
import { WebSocketServer } from 'ws';

export default {
  plugins: [
    sveltekit(),
    {
      name: 'maw-ws-dev',
      configureServer(server) {
        // bootstrap singletons in dev too
        import('./src/lib/server/bootstrap.ts').then(({ bootstrap, getWsHub }) => {
          bootstrap();
          const wss = new WebSocketServer({ noServer: true });
          server.httpServer.on('upgrade', (req, sock, head) => {
            if (new URL(req.url, 'http://x').pathname !== '/ws') return;
            wss.handleUpgrade(req, sock, head, (ws) => getWsHub().attach(ws, req));
          });
        });
      },
    },
  ],
};
```

This gives one consistent WS path (`/ws`) in dev and prod, with HMR still working for the Svelte side.

**Singleton bootstrap (`hooks.server.ts`)** — used by SvelteKit `+server.ts` routes to access the supervisor without re-initializing it:

```ts
// src/hooks.server.ts
import type { Handle } from '@sveltejs/kit';
import { bootstrap, getSupervisor } from '$lib/server/bootstrap';

await bootstrap(); // idempotent; safe in serverless adapters too

export const handle: Handle = async ({ event, resolve }) => {
  event.locals.supervisor = getSupervisor();
  // session resolution from cookie → event.locals.user
  return resolve(event);
};
```

`bootstrap()` uses a module-level `let started = false` guard so multiple imports/HMR reloads in dev don't create duplicate supervisors.

## Tmux integration

**Naming**: one tmux session per agent named `maw-agent-<ulid>`. One window, one pane. Pane runs a bootstrap script `cd $WORKTREE && exec env MAW_AGENT_ID=… MAW_URL=… $CLI_CMD $ARGS`. The env vars let the `maw` CLI tool inside the pane authenticate calls to the backend's MCP/HTTP bus.

**Commands used**:
- Spawn: `tmux new-session -d -s maw-agent-<id> -x 200 -y 50 -e MAW_AGENT_ID=<id> -- /path/to/bootstrap.sh`
- Live capture: `mkfifo /run/maw/fifo-<id>` then `tmux pipe-pane -t maw-agent-<id> -o 'cat >> /run/maw/fifo-<id>'`. Backend reads the FIFO end into the `AgentRuntime`.
- Send input: `tmux send-keys -t maw-agent-<id> -l -- <text>` then `tmux send-keys -t maw-agent-<id> Enter`. The literal `-l` flag prevents key-name interpretation on user content.
- Scrollback snapshot for new viewers: `tmux capture-pane -t maw-agent-<id> -p -e -S -10000`
- Kill: `tmux kill-session -t maw-agent-<id>`
- Boot reconciliation: `tmux list-sessions -F '#{session_name}'` filtered by `maw-agent-` prefix

**Reattach-on-boot**:
1. Backend starts, loads all `agents` rows where `status` ∈ `(running, waiting_input, idle, spawning)`.
2. For each, run `tmux has-session -t <session>`.
   - Present → recreate FIFO, re-run `pipe-pane -o`, `capture-pane` for scrollback, feed adapter (events tagged `replay:true` so they don't re-fire alerts), mark live.
   - Missing → mark `agents.status='crashed'`, close open runs, emit alert.
3. Reaper pass reconciles dangling worktrees against `git worktree list --porcelain`.

**Fan-out**: each `AgentRuntime` has one producer (FIFO reader) writing to (a) `terminal_log` append, (b) an `EventEmitter` that WS sessions subscribe to. Inputs from any client serialize through an **input queue** so simultaneous keystrokes don't interleave. Every input is broadcast back as a synthetic local-echo marker so multiple viewers can distinguish their own input from another client's.

## Realtime / WebSocket protocol

Single endpoint `/ws`, authenticated by the session cookie. JSON, discriminated on `type`. Terminal bytes travel as base64.

**Client → Server**: `subscribe_agent`, `unsubscribe_agent`, `send_input {agentId, text, submit}`, `answer_prompt {agentId, choice}`, `assign_task {agentId, task}`, `control {agentId, action: 'stop'|'sigint'|'restart'}`

**Server → Client**: `scrollback {agentId, chunks}`, `output {agentId, seq, b64}`, `event {agentId, kind, detail}`, `agent_state`, `alert`, `message` (inter-agent inbox), `ack` / `error`

Heartbeat ping every 20s; client reconnects with exponential backoff and re-subscribes using last-seen `seq` per agent to fetch gap-replay from `terminal_log`.

## Notification flow (end-to-end)

1. PWA installed on phone → user grants notification permission → frontend `pushManager.subscribe({applicationServerKey: VAPID_PUB})` → POST subscription JSON to `/api/push/subscribe` → stored in `push_subscriptions`.
2. Agent hits a permission prompt; adapter emits `prompt_detected`.
3. Alert pipeline dedupes by `(agent_id, reason)` within a 30s window; new alert → row inserted → `web-push` fan-out to all subs.
4. Push payload contains `{title, body, data: {agentId, alertId, deeplink}, actions:[yes,no,open]}`.
5. Service worker `push` event shows the notification. `notificationclick`:
   - `yes`/`no` → `fetch('/api/agents/:id/answer', {credentials:'include', body:{choice}})` → backend resolves via adapter `answerPrompt` (iOS pre-17 may ignore action buttons → fall back to plain tap-to-open)
   - `open` or bare tap → `clients.openWindow('/agents/' + agentId)` (focus existing client if matched)
6. PWA opens directly at the agent route; subscribes to WS; renders large touch-friendly Yes/No prompt UI when `waiting_input`.
7. User taps Yes → `answer_prompt` over WS → backend → `tmux send-keys` → agent continues. Backend broadcasts `alert_ack` to all clients.

**LLM overseer** runs as a per-agent interval task (default 60s, only when `working`/`idle`): fetches last N KB of `terminal_log`, strips ANSI, POSTs to chosen LLM with a fixed prompt that returns JSON `{verdict, rationale}`. Anything not `ok` synthesizes an alert. Hard-capped token budget per agent per hour.

## Inter-agent messaging

Two channels, both writing to the same `messages` table:

1. **MCP server** exposed by the backend on localhost: tools `maw_send_message(to_agent, body)`, `maw_read_inbox()`, `maw_tail(agent_id, lines)`. Authenticated by the `MAW_AGENT_ID` env var passed at tmux spawn.
2. **`maw` CLI binary** installed on PATH inside the agent's pane. Same operations: `maw send <agent> "msg"`, `maw inbox`, `maw tail <agent>`. Posts to a localhost-only HTTP endpoint with the same env-var auth.

This gives Claude Code the native MCP tool experience and gives codex/gemini (or any future CLI) a universal shell-out fallback. "Surveillance" combines `maw_tail` (raw output) with `maw_read_inbox` (structured messages).

> **Note on "all three CLIs from day 1"**: this is now genuinely cheap because the three adapters are JSONC files, not TypeScript classes. The risk shifts from "three state machines to write" to "three sets of regexes to tune against real CLI output." The `test-adapter.ts` replay tool exists specifically to make that tuning fast. Recommended workflow: capture a 10-minute session of each CLI doing realistic work (`script -q session.txt`), then iterate on the JSONC patterns until `pnpm test:adapter` reports the right events at the right times.

## Open risks to validate early

1. **Prompt detection fragility** — Claude/Codex/Gemini all redraw TUIs with ANSI cursor moves. Mitigation: ANSI-stripped 8KB rolling tail + "inactivity > 2s + cursor at prompt line" backup signal. Validate against real captured sessions on day 1.
2. **iOS Web Push quirks** — needs iOS 16.4+, must be "Add to Home Screen" installed, action buttons inconsistent. Always make plain tap-to-open the source of truth; treat action buttons as a bonus.
3. **`send-keys` with multiline user text** — use `-l` literal flag and a separate `Enter`; test multi-line pastes; bracketed-paste mode may confuse some TUIs.
4. **Output volume backpressure** — chatty agents emit MB/s bursts. Coalesce duplicate frames on the WS, never drop adapter events. Cap and vacuum `terminal_log`.
5. **Reattach correctness** — reopening FIFO + replaying `capture-pane` could double-log. Use `seq` dedupe and `replay:true` tag so replay doesn't re-fire alerts.
6. **Security** — service runs arbitrary code. Bind HTTP to loopback, expose only via Caddy + HTTPS (and optionally a second basic-auth gate at the proxy). CSRF double-submit token on REST mutations, WS origin check, rate-limited login with strong argon2id params.
7. **Worktree cleanup policy** — `git worktree remove` fails on dirty trees. Decide: prompt / force / archive — before auto-cleanup ships.
8. **LLM overseer cost loop** — overseer alerting on a chatty agent → infinite wakeups. Per-alert cooldown + per-hour token budget cap per agent.
9. **FIFO lifecycle** — clean up `/run/maw/fifo-*` on agent removal and on boot for any FIFO that doesn't map to a live agent.
10. **MCP tool auth** — MCP server trusts `MAW_AGENT_ID` env passed at tmux spawn time. Bind MCP/CLI HTTP endpoint to loopback only — never exposed off-host.

## Verification (how to know it works end-to-end)

1. **Backend boot smoke** — `pnpm --filter backend dev`, log shows "0 agents reattached"; `curl -k https://localhost/api/health` returns 200.
2. **Manual login flow** — log in via Caddy URL, get session cookie, hit `/api/me`.
3. **Spawn an agent** — create a project + repo + role + agent via UI; backend logs `tmux new-session ...`; `tmux ls` on host shows `maw-agent-<id>`.
4. **Live terminal** — open agent detail in browser, see xterm.js streaming; type a task, see the agent receive it.
5. **Reattach test** — kill backend (`systemctl stop maw`), confirm agent still alive in tmux; restart backend; UI reconnects; live stream resumes; no duplicate output (verify via `seq`).
6. **Prompt detection** — drive the agent into a permission prompt; backend emits `prompt_detected`; alert row inserted.
7. **PWA install + push** — install PWA on phone via Caddy HTTPS URL; enable notifications; trigger a permission prompt from a desktop browser; phone receives the push within ~5s; tap notification; PWA opens at agent detail; tap Yes button; agent unblocks.
8. **Reattach + push together** — restart backend with an agent in `waiting_input`; verify single push, not duplicates from replay.
9. **Inter-agent message** — from inside an agent's tmux pane, run `maw send <other-agent> "hello"`; the other agent's inbox UI shows the message.
10. **Vitest + Playwright** — `pnpm test` runs adapter unit tests (regex/state machine fixtures for each CLI) and a Playwright PWA smoke test against a stubbed agent.
