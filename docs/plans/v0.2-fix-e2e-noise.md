# Fix CI e2e noise: spawn-FAB strict-mode + agent-lifecycle race + chromium SEGV

## Context

The latest GitHub Actions e2e run reported 1 hard failure + 2 flakes:

1. **HARD FAIL** — `tests/e2e/smoke.spec.ts:35` *spawn FAB opens agent dialog* —
   Playwright strict-mode violation: `dialog[open] h2` resolved to **2** elements
   (`Spawn agent` + `Select directory`).
2. **FLAKE** — `tests/e2e/agent-lifecycle.spec.ts:205` *exit in one shell agent
   does not cascade-reap the others* — `agent-2 should be exited … Received: "running"`.
   Passed on retry.
3. **FLAKE** — `tests/e2e/smoke.spec.ts:26` *user menu opens on click* — chromium
   GPU process `Received signal 11 SEGV_MAPERR`. Passed on retry.

Two of these expose real bugs in the tests/components; the third is the
already-known headless-chromium GPU SEGV that the existing
`retries: process.env.CI ? 1 : 0` already absorbs. The plan addresses #1 and
#2 and explicitly leaves #3 alone (with a note for future).

---

## Diagnosis

### #1 — Spawn FAB strict-mode violation

[`SpawnAgentForm.svelte:477-486`](src/lib/client/components/SpawnAgentForm.svelte:477)
mounts `<DirectoryPickerDialog>` **unconditionally** as a sibling of the form
wrap. `DirectoryPickerDialog` wraps its body in a `<Modal>`, which always
renders a `<dialog class="maw-modal">` containing `<h2>{title ?? ''}</h2>`
(see [`Modal.svelte:62-80`](src/lib/client/components/Modal.svelte:62)) — the
`<dialog>` only gets the `open` attribute when `dialog.showModal()` is called
(`pickerOpen=true`), but **the entire dialog DOM exists either way**.

Because `SpawnAgentForm` is itself rendered inside the spawn modal's body
([`+page.svelte:114-125`](src/routes/%2Bpage.svelte:114)), the picker's
closed-but-mounted `<dialog>` ends up as a DOM **descendant** of the spawn
`<dialog open>`. Playwright's `locator('dialog[open]').locator('h2')` walks
the descendant tree and matches both `<h2>`s — strict-mode then fails
because it expected exactly 1.

This is **not** a UX regression — only the spawn dialog is visibly open. It
is a "DOM nesting outlives utility" issue compounded by a brittle test
selector.

### #2 — agent-lifecycle race

[`AgentSupervisor.finishAsExited`](src/lib/server/agents/AgentSupervisor.ts:383)
runs five steps after the tmux `session-closed` hook fires:

1. `this.runtimes.delete(agentId)` (sync)
2. `stopExitWatcher(agentId)` (sync)
3. `await runtime.stop()` (**async** — FIFO + pipe-pane teardown)
4. `updateAgentStatus(agentId, 'exited')` (sync DB flip)
5. `runtime.emit('state', 'exited')`

Meanwhile [`/api/agents/:id/snapshot`](src/routes/api/agents/%5Bid%5D/snapshot/%2Bserver.ts:35)
returns 410 as soon as `Tmux.hasSession()` reports the session is gone — that's
true the moment the hook fires, **before** step 4. The route also calls
`reapAgent`, which short-circuits if the runtime entry is already deleted by
the exit-watcher path (step 1 above).

So there is a window where:
- snapshot route returns 410 ✓
- DB row still says `running` ✗ (step 3's `await runtime.stop()` hasn't yielded yet)

The test's `waitForSnapshotStatus(page, agent2, 410)` followed by an immediate
`readAgentStatus(agent2)` lands in this window on slower runners. The fix is
to wait for the DB to actually flip — the test file already ships a
[`waitForDbStatus`](tests/e2e/agent-lifecycle.spec.ts:88) helper that does exactly that.

### #3 — Chromium SEGV

`Received signal 11 SEGV_MAPERR 0000000001b0` from `chrome-headless-shell` is a
known issue with chromium on `ubuntu-latest` runners; it has nothing to do
with our app. The plan from
[`v0.2-ci-node24-action-bump.md`](docs/plans/v0.2-ci-node24-action-bump.md)
already added `--disable-gpu --disable-software-rasterizer` to
`launchOptions.args` plus `retries: 1`, which absorbs it (the test went
green on retry in this run). No action needed; documenting only.

## Plan

Two targeted fixes. No new tests — the failing tests themselves are the
regression guard.

### Fix 1 — `SpawnAgentForm.svelte`: gate the picker dialog on `pickerOpen`

**File:** [`src/lib/client/components/SpawnAgentForm.svelte`](src/lib/client/components/SpawnAgentForm.svelte)

**Change:** wrap the `<DirectoryPickerDialog>` element at L477-486 in `{#if pickerOpen}`:

```svelte
{#if pickerOpen}
  <DirectoryPickerDialog
    open={pickerOpen}
    initialPath={newRepoPath || undefined}
    onClose={() => { pickerOpen = false; }}
    onSelect={({ path, cloneUrl }) => {
      newRepoPath = path;
      newRepoCloneUrl = cloneUrl;
      pickerOpen = false;
    }}
  />
{/if}
```

**Why this is the right shape, not the test selector:**
- Mirrors the existing pattern at
  [`+page.svelte:114-125`](src/routes/%2Bpage.svelte:114) where
  `{#if spawnOpen}<SpawnAgentForm .../>` keeps closed-modal DOM out of the
  tree. Tightening the test locator instead (`dialog[open] header h2:first-of-type`,
  `getByRole('heading').first()`, etc.) would mask the real DOM smell — a
  closed dialog whose body still costs render cycles, paints (briefly via
  `display:none`), and pollutes accessibility tree traversal.
- Picker has no useful state to preserve across closed→open cycles —
  [`DirectoryPickerDialog`](src/lib/client/components/DirectoryPickerDialog.svelte:67)
  resets and reloads via its `lastOpen` `$effect` on every reopen anyway.
- Single-line change, no schema/protocol/test churn.

### Fix 2 — `agent-lifecycle.spec.ts`: wait for DB flip, not just 410

**File:** [`tests/e2e/agent-lifecycle.spec.ts`](tests/e2e/agent-lifecycle.spec.ts)

**Change:** at L235, after `await waitForSnapshotStatus(page, agent2, 410)`, add
one line that waits for the DB to settle, then keep the existing strict
assertion:

```typescript
await waitForSnapshotStatus(page, agent2, 410);
await waitForDbStatus(agent2, 'exited');     // NEW — close the snapshot→DB race
expect(readAgentStatus(agent2), 'agent-2 should be exited after its bash closed').toBe(
  'exited'
);
```

**Why not change `finishAsExited` order to flip DB before `await runtime.stop()`:**
the await is there for a reason (FIFO+pipe-pane teardown must complete before
clients are told the runtime is gone, otherwise late bytes leak). The test is
the right place to bridge the gap — it has a matching helper for exactly
this purpose.

Update the comment block above the assertion (L231-234) to reflect that
`waitForSnapshotStatus` is necessary-but-not-sufficient; `waitForDbStatus`
is what guarantees the row is settled.

### Non-fix 3 — chromium SEGV stays as-is

No code change. If the SEGV becomes more frequent we can revisit by:
- Bumping `retries` from 1 to 2 (masks more, hides real flakes too — undesirable).
- Trying `chromium` instead of `chromium-headless-shell` (heavier; worth
  measuring before adopting).
- Adding `--single-process` (sometimes works around GPU sandbox SEGVs but
  changes the security model — explicitly NOT for production-style tests).

Track if frequency rises; otherwise leave it.
