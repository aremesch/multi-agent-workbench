# Remove Yellow Yes/No Question Bar from Agent Terminal

> Plan file will be renamed to `v0.2-remove-prompt-bar.md` immediately after
> approval (per repo convention).

## Context

When `claude-code` (or another CLI agent) emits a permission prompt, MAW
currently surfaces a yellow yes/no bar at the bottom of the agent terminal
modal. The bar shows two buttons (defaulting to `yes`/`no`) regardless of how
many choices the underlying prompt actually offers — so for Claude Code's
typical "1) Yes / 2) Yes, don't ask again / 3) No, tell Claude what to do
differently" prompt, the bar gives the wrong affordance and silently mismaps
clicks onto whatever the adapter's `promptAnswers` preset happens to be.

The user wants the in-modal bar gone entirely. They want to keep the PWA
**web-push** notification path (a separate WS channel: `SC_Alert` →
service-worker → OS push) so they still get told when an agent needs input —
they just don't want the bar anymore.

This is a surgical UI removal. Server-side detection of `prompt_detected`
events stays in place because it drives push.

## Approach

Delete the bar's render, state, click handler, CSS, and i18n key from
[`AgentTerminalPanel.svelte`](src/lib/client/components/AgentTerminalPanel.svelte)
+ the four locale files. Replace the panel's `onEvent` handler with a no-op
stub matching the existing pattern at
[StreamView.svelte:38](src/lib/client/components/StreamView.svelte:38).

**Intentionally NOT touched** (per "only remove this and nothing more"):

- **Push pipeline.** `processEvent → maybeAlert → AlertBus → SC_Alert +
  getPushService().notifyUser` — independent of the bar; stays.
- **Server-side `prompt_detected` event detection** in
  [`ConfigDrivenAdapter.ts`](src/lib/server/agents/adapters/ConfigDrivenAdapter.ts)
  — feeds the push pipeline; stays.
- **Orange `waiting_input` status dot** on dashboard tiles — driven by
  `SC_AgentState`, not the bar; stays.
- **Dashboard alert toast** — driven by `subscribe_user_alerts` + `AlertBus`;
  stays.
- **Settings push toggle** + the `settings.notify.prompt_detected` i18n key
  (`/settings`) — separate key, must stay so users can still opt in to push.
- **Orphaned WS plumbing** — `MawWsClient.answerPrompt`, `CS_AnswerPrompt`,
  `case 'answer_prompt'`, `handleAnswerPrompt`, `enqueueAnswer`,
  `input.answerPrompt`, `promptAnswers` (schema + jsonc). Becomes unreachable
  from the client after this change but is harmless inert code. Removing
  these would touch ~14 files (a separate, larger task) and contradicts the
  user's scoping. Leaving them also keeps the path available if a future
  richer prompt UI ever wants to reuse it.

## Steps

### 1. `src/lib/client/components/AgentTerminalPanel.svelte`

Delete the following spans (line numbers below match the current file; the
edit is one file, five contiguous regions):

1. **State** (lines 63-65) — delete the `let pendingPrompt = $state<...>(null);` declaration.
2. **`onEvent` handler body** (lines 89-95) — in the `handlers: AgentHandlers` object, replace the existing `onEvent: ({ kind, choices, detail }) => { ... }` block with a single-line no-op:
   ```ts
   onEvent: () => {},
   ```
   This drops the `prompt_detected` / `task_done` / `ready` branch. Matches the existing pattern at [StreamView.svelte:38](src/lib/client/components/StreamView.svelte:38).
3. **`answer()` function** (lines 166-169) — delete entirely.
4. **Markup** (lines 416-428) — delete the entire `{#if pendingPrompt} <section class="prompt"> ... </section> {/if}` block.
5. **CSS** (lines 551-566) — delete the three rules: `.prompt { ... }`, `.prompt .actions { ... }`, `.prompt button { ... }`.

After this single-file edit the bar cannot render: `pendingPrompt` no longer
exists, no setter remains, no consumer remains, no styles remain.

### 2. i18n — drop the `agent.promptDetected` key

Delete the `'agent.promptDetected'` entry from all four locale files. Order
matters for transient typecheck cleanliness (the partial locales reference
the key union defined by `en.ts`):

1. `src/lib/i18n/de.ts` (line 215)
2. `src/lib/i18n/es.ts` (line 218)
3. `src/lib/i18n/fr.ts` (line 218)
4. `src/lib/i18n/en.ts` (line 231) — last, since this is what defines the `TranslationKey` union.

**Do NOT touch** `settings.notify.prompt_detected` — that's a different key
used by the push-toggle UI in `/settings`.

## Verification

Run in this order:

1. **Typecheck** — `pnpm check`. Must stay clean (0/0).
2. **Unit tests** — `pnpm test:unit`. Must stay green; no existing test
   references the bar (verified via grep for `class="prompt"`,
   `promptDetected`, `agent.promptDetected`, `"Prompt detected"`).
3. **E2E push pipeline** — `pnpm test:e2e tests/e2e/claude-hook.spec.ts`.
   This spec POSTs a synthetic Claude Code Notification hook, then asserts
   that the `events` row + `alerts` row + alert content (tool name, command
   detail) all populate correctly. It is the authoritative check that PWA
   push still fires; if anything in the alert/push chain were unwired we'd
   see it here.
4. **Bar absence — code-level** — with `pendingPrompt` and `<section class="prompt">` deleted, no state path can mount the bar DOM. This is a static guarantee, not just a runtime one.
5. **Bar absence — visual (preview)** — `preview_start`, log in, open an
   agent terminal modal. Spawn a `claude-code` agent and have it ask a
   permission question (e.g. give it a task that runs `Bash`). Confirm:
   - No yellow bar at the bottom of the modal.
   - Orange `waiting_input` status dot still appears on the dashboard tile.
   - Push notification still fires (if subscribed in `/settings`).

The `shell` smoke adapter doesn't define `prompt_detected` patterns, so it
can't trigger the legacy path — use a real `claude-code` agent for the
manual smoke or rely on the e2e spec.
