# Per-agent git commit attribution — correctness + persistence

## Context

Today the archive page answers "which commits did this agent make?" by
running `git merge-base <agentBranch> <defaultBranch>` on every page load,
then `git log <mergeBase>..<agentBranch> --no-merges`. The current
implementation lives in `src/lib/server/git/agentCommits.ts:11` and is
called from `src/routes/repos/[id]/archive/+page.server.ts:66`. Nothing is
persisted — the list is recomputed from the current repo state every
time.

This heuristic is wrong whenever the repo diverges from "branch exists
untouched, base branch hasn't moved":

1. Agent's commits get merged/fast-forwarded into `main` → merge-base
   advances past them → they vanish from the agent's list.
2. Agent pulls or merges `main` into its branch → merge-base advances →
   foreign commits authored by other agents/users appear in this
   agent's list.
3. Agent's branch is deleted after archival → list is silently `[]`;
   everything the agent did is lost.
4. Agent force-pushes / rebases its branch → old SHAs become
   unreachable → history is lost.
5. Anyone commits on `maw/<agentId>` manually from outside the agent
   (direct git, IDE, another tool) → those commits are attributed to
   the agent.

The user reports "the assignment of git commits is wrong most of the
time," which is consistent with cases 1–2 in particular (the dominant
cases for any healthy multi-agent workflow).

The fix has two independent axes:

- **Attribution mechanism** — how we *decide* a commit belongs to an
  agent.
- **Persistence** — where we *keep* that decision so it survives branch
  mutation, deletion, and repo moves.

This document surveys the realistic options on each axis, documents the
tradeoffs, and specifies the combination chosen for implementation.

---

## Attribution mechanisms considered

### A. `git merge-base` range (today's code)

Range: `merge-base(agentBranch, defaultBranch)..agentBranch`.

- **Pro**: Zero setup; works with any git repo unmodified.
- **Con**: Breaks under merge-forward (commits vanish), branch-pull
  (foreign commits included), branch deletion (empty), force-push
  (lost), and manual branch edits (misattributed).
- **Verdict**: the broken baseline — retained only as a one-shot
  back-fill for legacy agents.

### B. `base_sha` anchored range

Range: `<base_sha>..agentBranch`, where `base_sha` is the SHA of the
default branch resolved at the moment `git worktree add` ran.

- **Pro**: Immutable anchor — doesn't move when `main` does. Cheap:
  one TEXT column on `agents`. Strict correctness improvement over A
  for the merge-forward case (#1) and the `main` advancing case.
- **Con**: Still attributes anything reachable from the agent's branch
  that's not reachable from `base_sha`. If the agent pulls `main` into
  its branch, those foreign commits are still attributed. Branch
  deletion / force-push still lose history at read time.
- **Verdict**: Not sufficient alone, but a valuable sanity-check /
  fallback range.

### C. Committer-date window

Filter by `started_at ≤ committed_at ≤ ended_at` on the agent's branch.

- **Pro**: No hooks, no env var changes, no schema.
- **Con**: Two agents running concurrently in the same repo cross-
  contaminate — every commit in the overlap window is attributed to
  both. Cherry-pick preserves committer timestamps (so old commits
  stay outside the window even though the agent just landed them).
  Clock skew on the host breaks boundary commits. Works against the
  user's actual model of "this agent did this work."
- **Verdict**: Ruled out.

### D. Commit-message trailer via `prepare-commit-msg` hook

Install a hook in each worktree that appends a trailer to every commit:

    <original message>

    MAW-Agent-Id: 01JABC...XYZ

Query via `git log --all --grep="MAW-Agent-Id: 01JABC...XYZ$"`.

- **Pro**: Most durable mechanism available. Trailer survives rebase,
  cherry-pick, merge, branch deletion (as long as the commit object
  still exists — `git log --all` finds it in reflog and unreferenced
  packs). `--no-verify` does **not** skip `prepare-commit-msg`.
- **Con**: Pollutes every commit message forever. When those commits
  are pushed upstream, reviewers see the `MAW-Agent-Id: ...` trailer
  on every commit. Requires per-worktree hook installation and must
  coexist with existing repo hooks (Husky, lefthook, pre-commit, …)
  by chaining rather than overwriting. Rebases of already-marked
  commits don't re-run the hook, which is fine (the trailer is
  already in the message).
- **Verdict**: Best durability, worst cosmetics. Good fallback if
  mechanism E turns out to be insufficient.

### E. Committer identity override via spawn env vars (chosen)

Inject when spawning the agent's CLI:

    GIT_COMMITTER_NAME=MAW-Agent-<agentId>
    GIT_COMMITTER_EMAIL=<agentId>@maw.local

Leave `GIT_AUTHOR_NAME` / `GIT_AUTHOR_EMAIL` untouched so the user's
real identity stays on the **author** field (visible in PR UIs).

Query via `git log --committer="<agentId>@maw.local"`.

- **Pro**: Invisible in most git UIs (PR pages show author, not
  committer). No hooks to install. Doesn't pollute commit messages.
  Correct semantics for cherry-pick (committer = whoever applied
  the commit = the agent that did the cherry-pick). Survives merge
  (merge commit has its own committer; merged commits retain theirs).
  Survives branch deletion: `git log --all --committer=<email>` finds
  reachable-from-anywhere commits; after GC, the DB snapshot is the
  fallback.
- **Con**: Rebase rewrites committer — if the **user** rebases the
  agent's branch outside the agent, those commits lose attribution.
  That's arguably correct (the user now owns them), but it's a
  behavior to document. A single distinctive committer per agent
  requires no new schema beyond persistence. If a CLI adapter strips
  `GIT_COMMITTER_*` env vars (none currently do), attribution falls
  back to the `base_sha` range.
- **Verdict**: Chosen. Strong correctness + no visible artifacts +
  minimal implementation.

### F. `post-commit` hook posts to MAW backend

Hook `curl`s MAW with `{ agentId, sha, message }` on every commit.

- **Pro**: Real-time; deterministic attribution.
- **Con**: Fragile. `--no-verify` *does* skip `post-commit`. `git
  rebase`, `git cherry-pick` and many merge paths also skip it.
  Requires MAW running at commit time, and a commit blocks on an HTTP
  call. Hook is the most-bypassed hook in everyday use.
- **Verdict**: Too fragile to be the authoritative mechanism. Out of
  scope for this change; could be added later as a *supplementary*
  fast-path.

### Chosen: E (committer identity) + B (base_sha anchor) as fallback

- Primary attribution signal: `git log --committer="<agentId>@maw.local"`
  scoped to the repo.
- Sanity check / fallback range: `<base_sha>..<agentBranch>`. Used
  when the committer filter returns zero rows and a range query
  returns non-zero — indicates an adapter stripped the env vars and
  we should warn rather than silently under-report.
- Neither touches the author field, and neither injects text into
  commit messages.

## Persistence

Independent of mechanism. The user explicitly asked for a database
table. Options on the time axis:

1. **Live every read** — not acceptable; branch deletion loses data.
2. **Snapshot at exit only** — one write per agent lifecycle; archive
   page reads DB.
3. **Snapshot at exit + on-demand refresh endpoint** (chosen) —
   covers post-exit edits (user commits after archival) and lets
   future UI add a "Refresh commits" button without schema changes.
4. **Snapshot at exit + periodic during run** — would support a
   live-commit panel on the dashboard; deferred (not needed for the
   archive correctness fix).

## Data model

One new migration. `base_sha` and committer identity columns go on
`agents` (cheap single-row read; `agents` stays the canonical row for
the archive join). Commits go in their own table.

`/home/maw/.local/share/maw/worktrees/wrong-git-assignments/migrations/006_agent_git_attribution.sql`:

```sql
-- Multi-Agent Workbench — durable git attribution for agents.
--
-- See docs/plans/v0.2-agent-git-attribution.md for the mechanism design.
-- Attribution is driven by a distinctive committer identity injected
-- into the agent's spawn env (GIT_COMMITTER_NAME/EMAIL). base_sha is the
-- immutable SHA of the default branch at worktree-creation time, kept
-- as a range fallback. Commits are snapshotted into agent_commits at
-- exit and on demand, so the archive survives branch deletion, repo
-- moves, and force-push.

ALTER TABLE agents ADD COLUMN base_sha TEXT;
ALTER TABLE agents ADD COLUMN committer_email TEXT;
ALTER TABLE agents ADD COLUMN head_sha_at_snapshot TEXT;
ALTER TABLE agents ADD COLUMN commits_snapshotted_at INTEGER;

CREATE TABLE agent_commits (
  id              TEXT PRIMARY KEY,
  user_id         TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  agent_id        TEXT NOT NULL REFERENCES agents(id) ON DELETE CASCADE,
  repo_id         TEXT NOT NULL REFERENCES repos(id) ON DELETE CASCADE,
  sha             TEXT NOT NULL,
  parent_shas     TEXT NOT NULL DEFAULT '[]',    -- JSON array
  author_name     TEXT NOT NULL DEFAULT '',
  author_email   TEXT NOT NULL DEFAULT '',
  committer_name  TEXT NOT NULL DEFAULT '',
  committer_email TEXT NOT NULL DEFAULT '',
  authored_at     INTEGER NOT NULL,              -- epoch seconds (%at)
  committed_at    INTEGER NOT NULL,              -- epoch seconds (%ct)
  subject         TEXT NOT NULL DEFAULT '',
  body            TEXT NOT NULL DEFAULT '',
  source          TEXT NOT NULL
                  CHECK (source IN ('committer','range','merge_base')),
  snapshotted_at  INTEGER NOT NULL,
  position        INTEGER NOT NULL,              -- stable order, 0=newest
  created_at      INTEGER NOT NULL,
  updated_at      INTEGER NOT NULL
);

CREATE UNIQUE INDEX idx_agent_commits_agent_sha ON agent_commits(agent_id, sha);
CREATE INDEX idx_agent_commits_agent_pos ON agent_commits(agent_id, position);
CREATE INDEX idx_agent_commits_repo ON agent_commits(repo_id);
```

Schema notes:

- `source` records which mechanism found each commit: `committer`
  (preferred), `range` (base_sha fallback), `merge_base` (legacy
  back-fill). Lets us diagnose when adapters strip env vars.
- `replaceAgentCommits` semantics: delete-then-insert in one
  transaction. Amended commits should *not* accumulate; the git
  state at snapshot time is authoritative.
- `position` is git's log output order, kept explicitly because
  `committed_at` ties happen (rebased sequences often share `%ct`).

## Implementation

### Layer 1 — git helper

Rewrite `src/lib/server/git/agentCommits.ts` to expose:

- `resolveSha(repoPath, rev): Promise<string | null>` — thin wrapper
  around `git -C <repoPath> rev-parse --verify <rev>^{commit}`; returns
  null on failure instead of throwing. Used at spawn.
- `listCommitsByCommitter(repoPath, committerEmail, baseSha | null, branch | null): Promise<AgentCommit[]>`
  — primary snapshot query. Runs
  `git -C <repoPath> log --all --committer=<email>^ --pretty=format:<FMT> --no-merges`
  (the `^` anchors the regex to start-of-field so committer emails
  sharing a prefix don't collide). If `branch` is supplied, scope
  the query to `<baseSha>..<branch>` as an additional filter —
  cheaper on large repos and excludes stale committer matches from
  earlier runs that re-used the worktree.
- `listCommitsInRange(repoPath, baseSha, branch): Promise<AgentCommit[]>`
  — the base_sha fallback. `git log <baseSha>..<branch> --no-merges`.
  Used when the committer query returns zero rows.
- `listAgentCommitsViaMergeBase(repoPath, branch, defaultBranch): Promise<AgentCommit[]>`
  — legacy merge-base path. Retained *only* for the one-shot
  back-fill of archived agents that predate this change.

Extended `AgentCommit` shape in `src/lib/shared/types.ts`:

```ts
interface AgentCommit {
  sha: string;
  shortSha: string;
  parentShas: string[];
  author: string;          // "<name> <email>", kept for existing UI
  authorName: string;
  authorEmail: string;
  committerName: string;
  committerEmail: string;
  authoredAt: number;      // epoch seconds
  committedAt: number;     // epoch seconds
  date: string;            // ISO, kept for existing UI
  subject: string;
  body: string;
}
```

`%H%x1f%P%x1f%an%x1f%ae%x1f%cn%x1f%ce%x1f%at%x1f%ct%x1f%aI%x1f%s%x1f%b%x1e`.

### Layer 4 — snapshot module

New file: `src/lib/server/git/commitSnapshot.ts`:

```ts
export async function snapshotAgentCommits(agentId: string):
  Promise<{ captured: number; source: 'committer' | 'range' | 'merge_base' }
         | { error: string }>
```

Steps:
1. Load agent row, worktree row (branch), repo row (path), project
   row (default_branch).
2. If `agent.committer_email` is set, try
   `listCommitsByCommitter(repo.path, committer_email, base_sha, branch)`
   → `source = 'committer'`.
3. If that returns empty **and** `agent.base_sha` is set, fall back
   to `listCommitsInRange(repo.path, base_sha, branch)` → `source = 'range'`.
   Log a warning: "committer filter returned zero, falling back to
   range — adapter may be stripping GIT_COMMITTER_* env."
4. If `agent.base_sha` is null (legacy agent, predates this change),
   call `listAgentCommitsViaMergeBase(repo.path, branch, project.default_branch)`
   → `source = 'merge_base'`.
5. Resolve branch HEAD via `rev-parse refs/heads/<branch>`.
6. `replaceAgentCommits(agentId, user_id, repo_id, commits, source)`.
7. `updateAgentCommitSnapshot(agentId, { head_sha_at_snapshot, commits_snapshotted_at })`.
8. Return `{ captured: commits.length, source }`. Never throws — on
   error, log and return `{ error: err.message }`.

### Layer 5 — exit hook wiring

`src/lib/server/agents/AgentSupervisor.ts`:

In `finishAsExited`, after `updateAgentStatus(agentId, 'exited')` and
after `runtime.emit('state', 'exited')`:

```ts
snapshotAgentCommits(agentId)
  .then((r) => {
    if ('error' in r) {
      console.warn(`[AgentSupervisor] commit snapshot failed for ${agentId}: ${r.error}`);
    } else {
      console.log(`[AgentSupervisor] captured ${r.captured} commits for ${agentId} (${r.source})`);
    }
  })
  .catch(() => {});
```

Ordering rationale: UI status transitions are instant regardless of
git latency; snapshot runs last, non-blocking; worktree is still on
disk at this point so git operations succeed.

Also call `snapshotAgentCommits(agentId)` in the same fire-and-forget
shape inside `kill(agentId)` (completes the contract for explicit
kills). Do **not** call it from `reap()` — that loops over
`finishAsExited`, which already does it.

### Layer 6 — on-demand API

New file: `src/routes/api/agents/[id]/commits/+server.ts`:

- `GET`: returns `{ commits: AgentCommit[], snapshottedAt: number | null, headShaAtSnapshot: string | null, source: string | null }` from the DB. No git calls.
- `POST`: re-runs `snapshotAgentCommits(params.id)` after ownership
  check; returns the fresh DB read.

Ownership check follows `src/routes/api/agents/[id]/+server.ts`.

`src/routes/repos/[id]/archive/+page.server.ts`:

Replace the `await listAgentCommits(...)` block with:

let rows = listPersistedAgentCommits(a.id);
if (rows.length === 0 && a.commits_snapshotted_at == null) {

### Layer 8 — scope fence (out-of-scope, explicitly)

Not in this change, but the schema supports them:

- UI "Refresh" button on the archive page (endpoint is exposed; UI

- `post-commit` hook real-time recording (mechanism F).
- Cherry-pick detection across branches (e.g. "agent A's commit was

- Surfacing `base_sha` / committer-email in the UI.
- Hook-chaining for repos with existing `prepare-commit-msg` hooks

## Verification

Against a throwaway repo:

1. **Happy path.** Spawn agent A. Inside its tmux session:
   `git commit --allow-empty -m c1`. Exit cleanly. Expect:
   - `SELECT base_sha FROM agents WHERE id='A'` non-null, matches
     `git rev-parse main` at spawn time.
   - `SELECT committer_email FROM agents WHERE id='A'` is
     `<A>@maw.local`.
   - `SELECT source, subject FROM agent_commits WHERE agent_id='A'`
     yields `committer | c1`.
   - `git log --committer=<A>@maw.local` shows c1 on `maw/A`.

2. **Archive reads DB, not git.** Visit `/repos/<id>/archive` — c1
   appears. `git -C <repo> branch -D maw/A`. Reload — c1 **still
   appears** (today: silently empty).

3. **Merge-back doesn't erase.** Spawn B, commit c2, exit. On `main`:
   `git merge --ff-only maw/B`. Archive still shows c2. (Today: the
   merge-base advance would have emptied the list.)

4. **Pull-forward doesn't pollute.** Spawn C at `main`. Third-party
   commit c3 lands on `main`. Inside C's branch: `git merge main`
   (brings c3 in). Commit c4 by the agent. Exit. Archive shows **c4
   only**. (Today: c3 would appear in C's list too.) The committer
   filter handles this; the range fallback would have failed it.

5. **Committer env propagated through adapter.** In a long-running
   agent, run `git log -1 --format="%cn <%ce>"` from inside the tmux
   session. Expect `MAW-Agent-<agentId> <<agentId>@maw.local>`. If an
   adapter is found to strip these vars, extend its config to
   preserve them and add a regression test.

6. **On-demand refresh.** Post-exit, manually commit on the agent's
   branch with the committer env set. `POST /api/agents/<id>/commits`
   → returns the new row. `GET` returns it too. Different-user
   access yields 403/404.

7. **Force-push survives.** Spawn D, commit c5, snapshot. Inside D:
   `git commit --amend -m c5'` (new SHA c5''). Exit. DB replaces c5
   with c5''. `git gc --prune=now` in the repo; archive still shows
   c5''.

8. **Genuinely empty agent.** Spawn E, commit nothing, exit.
   `agent_commits` empty, `commits_snapshotted_at` set. Two reloads
   of the archive: no git processes spawned for E (verify via
   `GIT_TRACE=1`).

9. **Legacy back-fill.** Pre-change, archive an agent. Apply the
   migration. First archive load triggers `snapshotAgentCommits` with
   `source='merge_base'`. Commits appear; `commits_snapshotted_at`
   set. Reload — DB-only.

10. **Unborn / odd default branch.** Attach a repo whose default
    branch has no commits (seeded by `ensureDefaultBranch`), spawn an
    agent. `base_sha` may be null; agent still spawns; snapshot at
    exit takes the merge-base fallback and succeeds.

11. **Concurrent agents, same repo.** Spawn A and B in the same repo
    at overlapping times. Both commit. Confirm each agent's
    `agent_commits` rows are disjoint (committer filter is per-agent
    email; timestamp contamination impossible).

12. **`pnpm check` clean** after all changes.

## Legacy agents

Two populations exist the day this deploys:

- **Archived, pre-change**: `base_sha` and `committer_email` both
  null; `agent_commits` empty. The archive page's one-shot back-fill
  (Layer 7) runs `snapshotAgentCommits`, which falls all the way
  through to `listAgentCommitsViaMergeBase` and persists the result
  with `source='merge_base'`. First visit: same cost as today; every
  subsequent visit: DB-only. Acceptable, strictly better than today
  from that point forward.

- **Running, pre-change** (spawned before the upgrade, still live at
  the moment the server restarts into the new build): `base_sha` and
  `committer_email` are null; the tmux session was started without
  our new env vars. We cannot inject env into a running tmux session
  retroactively. At exit, `snapshotAgentCommits` will fall through
  to the merge-base path — same as a legacy archived agent. The
  snapshot quality for these specific agents matches today's
  behavior; only new agents get the committer-filter improvement.
  Documented in the PR; no special migration needed.
