## Context

The dashboard at `src/routes/+page.svelte` is currently three static list cards (Projects, Roles, Agents). We want it to become a **spatial workspace**: one draggable/resizable card per live agent, each showing a ~5-second-old text snapshot of its tmux pane. Clicking a card opens a modal with the full live xterm (same experience as the existing `/agents/[id]` route). Positions persist per-user across sessions. Exited/crashed agents live in a side drawer where they can be deleted. A FAB in the corner starts the spawn flow.

This PR stops at the **minimal spawn path** (existing `/agents/new` page unchanged) and the **polled-snapshot thumbnail** strategy. Two follow-ups are noted inline: a unified one-step spawn wizard, and a hybrid live-xterm thumbnail mode.

Svelte 5 runes + SvelteKit 2 + Tailwind are already set up. No component library is installed; per the recommendation above, we will **not** add shadcn-svelte for this. Dialogs use a custom wrapper around the native `<dialog>` element.

## New dependency

- `gridstack` (added to `dependencies`). Dynamic-imported on the client only, mirroring how `src/lib/client/components/Terminal.svelte` handles `@xterm/xterm` for SSR safety.

## Backend

### 1. Migration вҖ” `migrations/002_user_settings.sql` (new)

Add a generic per-user key/value store. JSON blob keeps us flexible for future settings without new migrations.

```sql
CREATE TABLE user_settings (
  user_id    TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  key        TEXT NOT NULL,
  value_json TEXT NOT NULL,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  PRIMARY KEY (user_id, key)
);
```

Layout will be stored under key `dashboard.layout.v1` as `{ layout: [{ agentId, x, y, w, h }] }`.

### 2. `src/lib/server/db/queries.ts` вҖ” two new helpers

- `getUserSetting(userId: string, key: string): string | null` вҖ” parameterized SELECT.
- `setUserSetting(userId: string, key: string, valueJson: string): void` вҖ” `INSERT ... ON CONFLICT(user_id, key) DO UPDATE SET value_json = ?, updated_at = ?`.

Follow the prepared-statement pattern already in `queries.ts` (statements cached on first call).

### 3. `src/lib/server/tmux/TmuxSession.ts` вҖ” minor extension

`capturePane` at `TmuxSession.ts:106` currently hard-codes `-S -10000`. Add an optional `startLine` arg (default `-10000`) so callers can ask for shorter history (e.g. `-50` for thumbnails). Keep the existing call site in `src/lib/server/ws/hub.ts:122` unchanged by using the default.

### 4. New endpoint вҖ” `src/routes/api/agents/[id]/snapshot/+server.ts`

- `GET` вҖ” returns `{ text: string, ts: number }`.
- Auth: `locals.user` required; verify the agent row belongs to this user via existing `getAgentForUser(userId, id)` in `queries.ts`.
- Look up the agent's `tmux_session`, call `Tmux.capturePane(session, -50)`, run the result through `stripAnsi` (from the `strip-ansi` dep already in `package.json`), return as JSON.
- If the tmux session is gone (agent crashed since page load), return 410 with `{ text: '', ts }` so the card can show a placeholder without a console error.

### 5. New endpoint вҖ” `src/routes/api/user/dashboard-layout/+server.ts`

- `GET` вҖ” reads `getUserSetting(user.id, 'dashboard.layout.v1')`, returns parsed JSON or `{ layout: null }`.
- `PUT` вҖ” validates body with a small zod schema (`zod` already in deps): `{ layout: { agentId: string, x: number, y: number, w: number, h: number }[] }`. Persists via `setUserSetting`.

### 6. New endpoint вҖ” `src/routes/api/agents/[id]/+server.ts`

- `DELETE` вҖ” query param `removeWorktree=1|0` (default `0`).
  1. Verify ownership with `getAgentForUser`.
  2. If the agent is live, call `AgentSupervisor.kill(agentId)` (already exists at `src/lib/server/agents/AgentSupervisor.ts`).
  3. If `removeWorktree=1`, look up the associated worktree row and call `WorktreeManager.remove(path)` (add this wrapper to `src/lib/server/git/WorktreeManager.ts` вҖ” thin shell over `git worktree remove --force <path>`; the manager already wraps `execa('git worktree ...')`).
  4. Delete the agent row. The existing `CASCADE` rules in `migrations/001_init.sql` will take `agent_runs`, `tasks`, `events`, `messages`, `alerts`, `llm_oversight_verdicts`, and `terminal_log` with it.
  5. Return `204`.
